# RuStore-релиз — технический roadmap

> Требования RuStore-модерации × реальное состояние Void Dominion (сверено с кодом
> 2026-07-23, воркфлоу-фактчек 6 линз) → кирпичики до публикации в сторе.
>
> **Статус приоритета:** *на будущее.* Сейчас фокус — на игре; релиз приближаем
> понемногу. Эта дока — карта того, что понадобится карточке для прохождения
> модерации, а не план «делать прямо сейчас». Приоритеты (решение владельца):
> `RS-5.1` (TLS/wss) — ✅ **сделан** (продуктовая инфра, нужна и для живого онлайна);
> `RS-4.1` (офлайн-соло в player) — **отложен в самый конец** (по сути сторовое
> анти-WebView-требование, не игровая необходимость). Как следствие, сам сабмит в
> RuStore тоже уезжает в конец — RS-4.1 его жёсткий гейт (см. §3).
>
> Парные доки: `cross-platform-roadmap.md` (PWA/Capacitor; уведомления — Фаза 7),
> `secure-environment-roadmap.md` (TLS/деплой/прод-контур), `store-description.md`
> (черновики текстов витрины RU/EN), `../mobile/README.md` (сборка APK и подпись),
> `economy-roadmap.md` (монетизация/P2W → RuStore Pay).
>
> Формат — кирпичики: `id · зона · статус · deps · «Готово, когда»`. Зоны:
> `[legal]` (вне кода) · `[mobile]` (APK/Capacitor) · `[store]` (витрина/консоль) ·
> `[srv]` · `[proto]`. Статусы: ✅ готово · 🔶 частично · ⏳ можно брать ·
> 🔒(dep) · ❌ блокер · ⏸ отложено.

## 0. Требования RuStore (официальные, сверено 2026-07)

**Аккаунт разработчика.**
- **Физлицо** — допускается (паспорт РФ + реквизиты карты для выплат), достаточно
  для **бесплатного** приложения без встроенных покупок.
- **ИП / юрлицо** — ИНН, ОГРН(ИП), реквизиты; форма подписывается **УКЭП**.
  Обязательно для **монетизации** (платное приложение / IAP → 54-ФЗ, фискализация).

**Сборка и подпись.**
- Формат: **APK и AAB — оба принимаются** (AAB компактнее; «обязательно AAB» — миф).
- **Подпись — своя.** RuStore, в отличие от Google Play, **не подписывает загрузку
  за разработчика** — нужен действующий **секретный release-ключ** (потеря ключа =
  невозможность обновлять приложение той же личностью).
- **targetSdkVersion ≥ 28** (Android 9). **64-bit ABI** — если есть нативные `.so`
  (у нас чистый WebView-Capacitor → неактуально).

**Карточка (витрина).**
- Иконка **512×512 PNG, без прозрачности по краям**.
- Скриншоты: **минимум 4, до 8–10** на мобильную форму (1080×1920 портрет /
  1920×1080 ландшафт, PNG/JPEG).
- Краткое описание **≤ 80 знаков**, полное **≤ 4000 знаков**.
- **Возрастной рейтинг** (анкета в консоли) + **категория** (Игры → подкатегория).

**Приватность (жёстко модерируется).**
- **Отсутствие политики конфиденциальности или мёртвая ссылка = отказ.** Нужен
  реальный хостируемый текст + честное раскрытие сбора данных в карточке.
- Соответствие **152-ФЗ** «О персональных данных». Особый досмотр, если приложение
  собирает гео / камеру / микрофон / контакты (у нас — нет; но собираются логин и
  **email восстановления** → политика всё равно обязательна).

**Модерация** (~24 ч, часто быстрее). Частые причины отказа: сырая/недоделанная
версия, **WebView-«переходник» без самоценности**, избыток рекламы, разрешения не
по назначению, клоны с косметическими правками, немаркированный 18+, незаконный /
экстремистский контент (ФЗ-149), кликбейт.

**SDK (условно).**
- **RuStore Pay SDK** — только при платежах/IAP (+ 54-ФЗ). Для бесплатного старта
  не нужен.
- SDK обновлений — **необязателен**; стор обновляет приложение сам. Значит наш
  собственный GitHub-Releases-апдейтер (`updater.ts`) в стор-сборке надо **гасить**.

## 2. Кирпичики

### Блок RS-0 · Юр/аккаунт `[legal]` — вне кода, критический путь

- **RS-0.1** ⏳ **Регистрация аккаунта RuStore** (физлицо для старта; ИП — при
  будущей монетизации). **Готово, когда:** аккаунт подтверждён, консоль доступна.
- **RS-0.2** ⏳ **Текст Privacy Policy** — что собираем (логин/позывной, **email
  восстановления**, scrypt-хэш пароля, session/join-JWT, XP), где и сколько хранится,
  права субъекта ПД по 152-ФЗ; захостить публично (сайт / GitHub Pages). **Готово,
  когда:** публичный URL с политикой открывается без авторизации.

### Блок RS-1 · Сборка и подпись `[mobile]`

- **RS-1.1** 🔒(—) **Release keystore.** Секретный ключ **вне репозитория**, в CI —
  через `secrets` (не коммитить); `signingConfigs.release` в Gradle-патче; ассерт
  стабильности подписи (как текущий debug-ассерт). **Готово, когда:** `player`-APK
  подписан release-ключом, его cert-digest закреплён в CI.
- **RS-1.2** 🔒(RS-1.1) **Стор-профиль сборки.** Отдельный `store`-таргет:
  **выключить GitHub-self-updater** (стор обновляет сам), `cleartext:false`,
  `androidScheme:'https'`, срезать dev-хвосты. **Готово, когда:** стор-APK без
  внешнего апдейтера и без cleartext-трафика.
- **RS-1.3** ⏳ Верификация финального манифеста: `targetSdk ≥ 28`, только нужные
  `uses-permission` (ожидаем лишь `INTERNET`). **Готово, когда:** `aapt dump badging`
  подтверждает target и минимальный набор разрешений.

### Блок RS-2 · Персданные / 152-ФЗ `[proto]` `[legal]` 🔒(RS-0.2)

- **RS-2.1** **Экран согласия** при регистрации: чекбокс «согласен на обработку ПД»
  + живая ссылка на политику; `submit` заблокирован без галочки.
- **RS-2.2** Оживить футер-ссылку **«Политика конфиденциальности»** (сейчас
  «— скоро» → реальный URL из RS-0.2).
- **RS-2.3** Раскрыть сбор `email` в карточке; подтвердить, что email остаётся
  **опциональным** (уже так в `authApi.ts`). **Готово, когда:** регистрация требует
  согласия, ссылка ведёт на политику, карточка честно перечисляет данные.

### Блок RS-3 · Витрина `[store]`

- **RS-3.1** Иконка **512×512 без прозрачности** (даунскейл из 1024²).
- **RS-3.2** **5–8 скриншотов** 1080×1920 с реальной сборки (карта, активный бой,
  карточка флота, карточка планеты, дипломатия/чат, дерево техов).
- **RS-3.3** Подогнать тексты под **80 / 4000** из `store-description.md`.
- **RS-3.4** Возрастной рейтинг (вероятно **12+** — фэнтезийная война без
  натурализма) + категория **Игры → Стратегии**. **Готово, когда:** все поля
  карточки заполнены и приняты консолью.

### Блок RS-4 · Продуктовая самоценность `[proto]` — главный анти-WebView-рычаг

- **RS-4.1** ⏸ **ОТЛОЖЕНО В САМЫЙ КОНЕЦ** (решение владельца) — **вернуть
  офлайн-соло-vs-AI в `player`-сборку** (сейчас там лишь туториал; полный skirmish
  живёт только в `dev`). Это доказательство «завершённый самостоятельный продукт,
  работает офлайн». **Готово, когда:** стор-APK **без сети** даёт полноценный матч
  против ИИ. *Жёсткий гейт RuStore-модерации — пока отложен, сам сабмит в стор уезжает
  в конец вместе с ним (см. §3). Курс «сначала игра, стор последним».*
- **RS-4.2** ⏳ Стабилизация к сабмиту: заморозка фич, багфиксы, прогон на
  **реальных устройствах** (не эмулятор). **Готово, когда:** 30-мин сессия без
  крэшей на 2–3 реальных девайсах.

### Блок RS-5 · Транспорт / инфра `[srv]` — обязателен для онлайн-режима

- **RS-5.1** 🔶 **TLS / wss.** ✅ Нативный TLS в сервере (`TLS_KEY_FILE`/`TLS_CERT_FILE`
  → `wss://` in-process на обоих хостах, fail-secure на половинчатой настройке;
  интеграционный тест на живом self-signed-рукопожатии; compose + runbook). Клиент уже
  деривит `wss://` из `https`. Остаётся: выпуск Let's Encrypt-сертификата на прод-домен
  + `cleartext:false` в стор-профиле (RS-1.2). **Готово, когда:** прод-онлайн по `wss://`
  с реальным сертом, `cleartext` выключен. *(Продуктовый трек — не только для стора.)*
- **RS-5.2** Прод-хостинг: healthcheck, бэкапы Postgres, рестарт-политика (сейчас
  self-host compose без TLS). **Готово, когда:** сервер доступен по стабильному
  домену с мониторингом.

### Блок RS-6 · Подготовка к модерации `[store]`

- **RS-6.1** **Тестовый аккаунт** для модератора (логин+пароль-контур готов —
  оформить демо-креды в примечании к сабмиту).
- **RS-6.2** Пакет для модерации: обоснование каждого разрешения (ожидаем лишь
  `INTERNET`), краткое «как играть». **Готово, когда:** пакет собран, демо-креды
  работают.

### Блок RS-7 · Позже / опционально

- **RS-7.1** ⏳ Монетизация → **ИП + УКЭП + RuStore Pay SDK + 54-ФЗ** (это и есть
  P2W-ветка из `economy-roadmap.md`).
- **RS-7.2** 🔒 **Push-уведомления** (Фаза 7 в `cross-platform-roadmap.md`) —
  критичны для async-удержания (мир идёт 24/7), но **не блокер** релиза.

## 3. Критический путь и оценка

```
RS-0.1 (аккаунт) ─┐
RS-0.2 (policy)  ─┼─→ RS-2.* (согласие+ссылка) ─┐
RS-1.1 (ключ) → RS-1.2 (стор-профиль) ──────────┼─→ RS-3.* (витрина) → RS-6.* → САБМИТ
RS-4.1 (офлайн-соло в player) ───────────────────┘   (снимает анти-WebView-риск)
RS-5.* (TLS) ── параллельно; обязателен только для онлайн-режима
```

- **«Бумага + сборка» (RS-0, 1, 2, 3, 6)** — реально **~1 неделя**: release-ключ,
  текст политики, экран согласия, скриншоты, поля консоли. Инженерно почти всё
  тривиально; можно взять целиком перед самим сабмитом.
- **RS-5.1 (TLS) — ✅ сделан** (нативный wss в сервере). **RS-4.1 (офлайн-соло) —
  отложен в самый конец** (решение владельца). Т.к. RS-4.1 — жёсткий гейт модерации
  («несамоценно»), сам **сабмит в RuStore смещается в конец** вместе с ним — это
  сознательный размен «сначала игра, стор последним».
- **Итого:** «бумага + сборка» (RS-0,1,2,3,6) — ~1 неделя, но берётся **перед** самим
  сабмитом; а сабмит теперь привязан к завершению RS-4.1 в самом конце. (Оценка
  «4–8 недель» из внешнего аудита завышена — включала фантомные блокеры вроде
  `REQUEST_INSTALL_PACKAGES`, которого в сборке нет.)

## Инварианты (нарушение = провал модерации или релиза)

- **Секретность release-ключа.** Ключ и его пароли **никогда** не попадают в
  репозиторий (только CI-secrets). Потеря ключа необратима — бэкап офлайн.
- **Privacy-first.** Ни одной формы сбора ПД без живой ссылки на политику и
  согласия; список собираемых данных в карточке = реальности кода.
- **Самоценность.** Стор-сборка обязана давать ценность **без сети** (офлайн-соло),
  иначе WebView-обёртка читается модерацией как «переходник».
- **Без cleartext в сторе.** Финальная сборка — только `https`/`wss`.
