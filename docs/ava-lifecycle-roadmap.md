## AVA-1 · Командная дипломатия на серверном пути `[srv/core]` — ✅ реализовано

**Цель.** Сервер умеет создать матч, где стороны — реальные альянсы (та же команда
союзники, между — война), тем же посевом стоек, что и AVA-0 в прототипе.

**Опора.** `buildStateFromMap` уже сажает игроков в слоты и знает `slot.team`, но
**не пишет `state.diplomacy`**. В прототипе `newGame` (`game.ts:1888`) — готовая
эталонная логика посева пар по команде.

**Модель данных.** Без новых стор. Только заполнение `state.diplomacy` при сборке.

**Реализация.** Портировать посев в ядро: `buildStateFromMap` (или чистый хелпер
`seedTeamDiplomacy(slots, players) → Record<pairKey, DiplomaticStance>`) сеет по парам:
нет команд → `peace` (обратная совместимость обычных матчей); та же команда →
`alliance`; между командами → **`peace` если карта помечена «мирный старт» иначе
`war`** (для AvA нужен старт с мира — см. AVA-8, потому параметр `crossTeamStart`).
Сгруппированный спавн союзников — свойство раскладки карты (слоты одной стороны рядом),
уже выражается позициями в `ava-duel-1.json`.

**Интеграция.** Это шов, на который встанет AVA-7 (оркестратор зовёт билдер с
`crossTeamStart:'peace'`). Обычные матчи (`createDevMatch`) не трогаем — там команд нет.

**Тесты.** Ядро: 2v2 карта → союзники `alliance`, соперники по `crossTeamStart`; FFA
без команд → всё `peace`; детерминизм (тот же вход → тот же `diplomacy`). Anti-regression
на существующие карты.

**DoD.** `buildStateFromMap(avaMap, data, { slots, crossTeamStart })` возвращает state с
корректными стойками; `combat.isHostile` между сторонами следует `crossTeamStart`.

**Побочно (playtest):** **NET-2v2** — прото-хост `netserver.ts` сейчас сеет только 1v1
`DEFAULT_SETUP`; передать в `newGame` командные `seats` (2×team A, 2×team B) — маленький
прото-кирпич, даёт живой 2v2 до готовности мета-оркестратора.

> **Итог реализации.** Хелпер `seedTeamDiplomacy` в `buildFromMap.ts`: карта без
> команд → peace-FFA (конвенция прототипа); та же сторона → `alliance` (seeded —
> минует `E_BOT_ALLIANCE`, ИИ-союзник — настоящий); между сторонами — по
> `BuildFromMapOptions.crossTeamStart` (`war` дефолт / `peace` — мирный старт под
> AVA-8). Пары из отсортированных id → канонический JSON. NET-2v2: `TEAMS=2v2` на
> прото-хосте сеет A: p1+p2 (западные углы) vs B: p3+p4 (восточные); пустые кресла
> играет серверный ИИ. Тесты — `buildFromMap.test.ts`.

## AVA-2 · Очки влияния корпорации `[srv]` ✅

**Цель.** Корп-валюта: тратится на вызов (S1), начисляется за победу (S7), двигает
рейтинг корпораций (`main-menu.md` §4.6).

**Опора.** CORP-0 `CorpStore`/`CorpService`. Внутриматчевая казна игрока (`ResourceBag`,
`util/treasury.ts`) — **отдельная**, не переиспользуется (это межматчевый баланс корпы).

**Модель данных.** Расширить `CorpStore`: колонка `influence bigint NOT NULL DEFAULT 0`
в `corps` (Postgres) + поле в `MemoryCorpStore`. Мутации атомарны и fail-secure на
недостаток:

```
addInfluence(corpId, delta): Promise<void>                 // начисление (S7), delta>0
spendInfluence(corpId, cost): Promise<{ok:true}|{ok:false;code:'E_INSUFFICIENT'}>  // атомарно, cost>0
```

`CorpRecord`/`CorpSummary` получают `influence`. Аудит-действие `influence` (уже union —
добавить вариант) со `detail` = «+N: победа над X» / «−N: вызов Y».

**Сервис + API.** Списание/начисление — **не прямой интент игрока** (только эффект
вызова/итога), поэтому публичных write-роутов нет; баланс виден в `GET /corps/:id`
и `/corps/me`. Порог/цена вызова — константа (см. открытые вопросы).

**Тесты.** Контракт обоих адаптеров: начисление, атомарное списание, отказ
`E_INSUFFICIENT` при недостатке (без ухода в минус), аудит-запись.

**DoD.** Баланс переживает рестарт (Postgres), списание атомарно (нет гонки «двойной
траты»), отражается в read-моделях корпы.

## AVA-3 · Флаги готовности к AvA `[srv]` ✅

**Цель.** Два флага (`corporation-wars.md` §2): **корп-флаг** (глава → корпа в пуле
готовых) и **игровой флаг** (игрок → постоянное согласие на офлайн-развёртывание).

**Опора.** CORP-0 RBAC (`CorpService.mine`/`detail` дают роль). AVA-2 (готовность —
предусловие вызова).

**Модель данных.** Новый `AvaReadinessStore` (или расширение `CorpStore`):

- `corp_ready(corp_id PK, since)` — корп-флаг;
- `player_ready(account_id PK, corp_id, since)` — игровой флаг (аккаунт готов в составе
  СВОЕЙ корпы; при выходе из корпы — чистится).

**Сервис + API.** Интенты `corporation-wars.md` §9 через `CorpService`-подобный гейт:

```
setCorpReady / clearCorpReady          — только глава (headGate)
setPlayerReady / clearPlayerReady      — сам игрок, только член своей корпы
```

REST: `POST /ava/ready/corp`, `POST /ava/ready/player` (+ clear). `GET /ava/pool` —
список корп с корп-флагом (публичный в сессии). Всё fail-secure, аудит на корп-флаг.

**Тесты.** RBAC: не-глава не ставит корп-флаг (`E_FORBIDDEN`); не-член не ставит игровой;
выход из корпы чистит игровой флаг; пул отражает только флагнутые корпы.

**DoD.** Пул готовых читается; флаги durable; RBAC соблюдён.

## AVA-4 · Вызов / принятие (S0→S2) `[srv]` ✅

**Цель.** Готовая корпа вызывает другую готовую; вызов тратит влияние, истекает по
таймеру; цель принимает/отклоняет; принятие → «законтрактовано» (S2).

**Опора.** AVA-2 (влияние), AVA-3 (пул готовых), CORP-0 (кто глава). Таймер — тем же
`clockDriver`/`schedule`-паттерном, что и офлайн-события (инжектируемое время).

**Модель данных.** `AvaChallengeStore`:
`challenge(id PK, challenger_corp, target_corp, cost, status: pending|accepted|declined|expired, created_at, expires_at)`.
Инвариант: одна активная (`pending`) заявка на пару вызывающий→цель.

**Сервис + API (state-машина S0–S2).**

```
challengeCorp(targetCorpId)  — глава вызывающей; обе корпы ready; spendInfluence атомарно;
                               ПЕРЕД списанием проверки, при отказе создания — возврат.
                               Планирует expiry (timeScale-часы).
acceptChallenge(id)          — глава ЦЕЛИ; → status accepted → S2-матчап (запись + AVA-9 фид).
declineChallenge(id)         — глава цели; → declined; влияние вызывающей возвращается.
(expiry)                     — авто по таймеру; → expired; возврат влияния.
```

Коды: `E_FORBIDDEN`, `E_NOT_READY`, `E_SELF_CHALLENGE`, `E_ALREADY_CHALLENGED`,
`E_INSUFFICIENT`, `E_NO_CHALLENGE`, `E_CHALLENGE_CLOSED`. REST `POST /ava/challenge`,
`POST /ava/challenge/:id/(accept|decline)`, `GET /ava/challenges` (свои входящие/исходящие).

**Интеграция.** На `accept` создаётся **матчап** (запись контракта) — вход в AVA-6
(сбор ростера). Публикация матчапа в ленту — AVA-9.

**Тесты.** Полный state-машинный набор: списание при вызове, возврат при
decline/expiry, идемпотентность accept, гонка «двойной accept», RBAC глав, истечение по
инжектируемому таймеру.

**DoD.** Пара глав проходит вызов→принятие→S2; влияние корректно списано/возвращено;
истечение работает без подключённых клиентов.

## AVA-5 · Пул AvA-карт + eligibility `[core/data]` — ✅ реализовано

**Цель.** Отметить карты как AvA-пригодные и выбирать одну случайно на S4.

**Опора.** `MatchMapSchema` + `data/maps/ava-duel-1.json` (готовая 2-слотовая). Пробел:
нет тега пригодности и метаданных о числе слотов/сторон.

**Модель данных.** В `MatchMapSchema` добавить необязательные метаданные:
`ava?: { eligible: boolean; sides: number; slotsPerSide: number }` (или вывести
`sides/slotsPerSide` из `slots` и хранить только `avaEligible: boolean`). Пометить
`ava-duel-1.json` (`sides:2, slotsPerSide:1`); добавить хотя бы одну 2v2-карту
(`sides:2, slotsPerSide:2`) с сгруппированными слотами.

**Сервис.** Чистый хелпер-пул `pickAvaMap(maps, sides, slotsPerSide, rng) → MatchMap`
(seeded-выбор для детерминизма/реплеев). Живёт в server/meta (оркестратор), карты
валидируются существующим `parseMatchMap`.

**Тесты.** Схема принимает/отклоняет метаданные; пул фильтрует по размеру; `pickAvaMap`
детерминирован при том же seed; `ava-duel-1` проходит валидацию с тегом.

**DoD.** Оркестратор может запросить «AvA-карту на 2×2» и получить валидную `MatchMap`.

> **Итог реализации.** Выбран второй вариант модели: `avaEligible: boolean` в
> `MatchMapSchema`, а форма ВЫВОДИТСЯ из `slots` хелпером `avaShape(map) →
> {sides, slotsPerSide} | null` (null при <2 сторонах или неравных сторонах) —
> тегу и раскладке неоткуда разъехаться; `validateMatchMap` валит eligible-карту
> с кривой формой (`E_AVA_SHAPE`), без тега та же раскладка легальна. `ava-duel-1`
> помечена (2×1); добавлена `data/maps/ava-2v2-1.json` (2×2: союзные дома
> сгруппированы по флангам — A запад, B восток, нейтральные призы west/east,
> центр nexus). `pickAvaMap(maps, sides, slotsPerSide, rng)` —
> `packages/server/src/avaMapPool.ts`: фильтр тег+форма, канонический порядок по
> id (выбор не зависит от порядка массива), ровно один draw seeded-`Rng`; нет
> кандидата → null. Тесты: `buildFromMap.test.ts` (AVA-5 блок) + `avaMapPool.test.ts`.

**Цель.** После S2 — окно паузы (1–2 timeScale-дня): набор состава ОБЕИХ сторон; в
конце ростер **лочится** (GDD §2 консервация).

**Опора.** AVA-3 (флагнутые = кандидаты на авто-развёртывание), AVA-4 (матчап), таймер
паузы (`schedule`). CORP-0 (глава/офицер курируют — `officerGate`).

**Модель данных.** `AvaRosterStore`:
`roster_entry(matchup_id, account_id, side, source: flagged|self, PK(matchup_id, account_id))`

- `matchup(... , roster_locked bool, pause_ends_at)`. Кап на сторону = слоты карты (из
  AVA-5). Лок — необратимый флаг.

**Сервис + API.**

```
setRoster(matchupId, playerIds[])  — глава/офицер; только из ПУЛА флагнутых своей корпы;
                                     кап по слотам стороны.
joinAva(matchupId) / readyUp       — сам игрок; самозапись в окне паузы (в т.ч. нефлагнутый).
(pause end)                        — авто-лок: min-per-side проверка → недобор → отмена
                                     матча + возврат цены вызова (AVA-4); иначе roster_locked.
```

Коды: `E_FORBIDDEN`, `E_NOT_FLAGGED`, `E_ROSTER_FULL`, `E_ROSTER_LOCKED`, `E_WINDOW_CLOSED`.
REST `POST /ava/matchup/:id/roster`, `POST /ava/matchup/:id/(join|ready)`,
`GET /ava/matchup/:id` (свой ростер; чужой приватный).

**Тесты.** Кап по слотам; только флагнутые в `setRoster`; поздний вход только в окне;
лок замораживает изменения (`E_ROSTER_LOCKED`); недобор → отмена+возврат; RBAC.

**DoD.** Обе стороны собирают состав, в конце паузы ростер лочится или матч отменён с
возвратом.

## AVA-7 · Оркестратор: создание сессии (S4) `[srv]` ★

**Цель.** По локу ростера — собрать реальную сессию: выбрать AvA-карту, разложить
ростер по командным слотам (сгруппированно), создать матч, посадить аккаунты в **их**
слоты.

**Опора.** AVA-1 (посев командных стоек), AVA-5 (`pickAvaMap`), AVA-6 (залоченный
ростер), `buildStateFromMap({ slots })`, `MatchRegistry.register`, `LazyRoomRegistry`,
`clockDriver`. Ключевой пробел: `resolveSeat` даёт «любое свободное место» — для AvA
нужна привязка аккаунт→слот.

**Модель данных.** `matchup` получает `match_id` (созданный инстанс) + карту `slot→account`.

**Реализация (сердце фазовой машины).**

1. `pickAvaMap(pool, sides, slotsPerSide, rng)`.
2. Разложить ростер по слотам карты (сторона A → слоты team A, и т.д.), сгруппированно.
   Построить `slots: Record<slotId, SlotAssignment{ playerId: accountId, heroes, ... }>`
   (лоадаут MVP = стартовый набор слота карты, см. §отложено — снапшот арсенала позже).
3. `state = buildStateFromMap(map, data, { slots, crossTeamStart: 'peace' })` (мирный
   старт — S5).
4. Создать `MatchRoom({ id: matchId, initialState: state, kernel, data, config, persist,
gate, manualStart })`; отдать через `LazyRoomRegistry.load`; `MatchRegistry.register`.
5. **AvA-осведомлённая рассадка:** аккаунт заходит → его `playerId` фиксирован ростером
   (не первое свободное). Новый резолвер seat: `resolveAvaSeat(matchId, accountId) →
fixed playerId | E_NOT_ROSTERED` (mapping из `matchup.slot→account`), встраивается в
   WS-хендшейк рядом с `resolveSeat`.

**Интеграция.** Переиспускает существующий транспорт/персистентность/фог; AvA — «свой
режим матчмейкинга» рядом с обычным (`MM-2.1`).

**Тесты.** Оркестратор из залоченного ростера строит валидный state (нужные игроки в
нужных слотах, союзники сгруппированы); `resolveAvaSeat` возвращает фиксированное место
и `E_NOT_ROSTERED` чужому; кросс-командный старт = `peace`; матч виден в реестре.

**DoD.** Из принятого вызова + собранного ростера поднимается живая AvA-сессия; игроки
садятся строго в свои слоты; союзники — единый фронт; бой пока заперт (мир).

## AVA-8 · Мир → война → итог (S5–S7) `[srv]` ★

**Цель.** Мирный период (combat-lock, экономика идёт) → открытие войны → пост-матч
награды + архив.

**Опора.** AVA-7 (живая сессия, старт `peace`), дипломатия (`diplomacy.declare`),
`economyModule` (пассив на `time.advanced` — офлайн-экономика бесплатно),
`resolveBattle` (гарнизон обороняется детерминированно без игрока), `victoryModule`
(`match.ended`), `schedule` (таймеры фаз).

**Реализация.**

- **S5 мир:** кросс-командные стойки = `peace` ⇒ `combat.isHostile=false` ⇒ боя нет,
  а экономика/стройка/движение идут. Combat-lock **бесплатно** из посева (не нужен
  отдельный флаг в ядре). Длительность — timeScale-дни (константа, см. вопросы).
- **S6 война:** по таймеру оркестратор шлёт системные `diplomacy.declare(war)` по всем
  кросс-командным парам (через `submitAction`, мимо гейта — как серверные драйверы).
  Бой открывается. (Игроки внутри AvA войну сами не объявляют — стойками владеет
  оркестратор; их `diplomacy.declare` в AvA-режиме отклоняется.)
- **S7 итог:** на `match.ended` — начислить влияние победившей корпе (AVA-2
  `addInfluence`), XP аккаунтам (существующий `matchXp`-путь), опубликовать итог в ленту
  (AVA-9), заархивировать. **История матчей (MM-3.1)** — минимальный `AvaResultStore`
  (кто с кем, победитель, время) — фундамент под условия медалей (`corporations.md` §3,
  отложены) и рейтинг.

**Офлайн авто-развёртывание (MVP, §7 доки).** Пассивная экономика — движком; гарнизон
обороняется — `resolveBattle`; **никакого авто-наступления** (Хранителя к AvA-местам не
цепляем — атака только от вошедшего игрока). Лоадаут = стартовый набор слота (снапшот
личного арсенала — отложен до меты-инвентаря).

**Тесты.** В мир combat-lock (враждебное действие между сторонами — нет боя); по таймеру
война открывается и первый бой резолвится; `match.ended` → влияние победителю + запись
результата; офлайн-гарнизон обороняется без игрока; XP начислен.

**DoD.** Полный проход S5→S6→S7 на инжектируемых таймерах: мир копит экономику, война
идёт до победы, победившая корпа получает влияние, результат записан и опубликован.

## AVA-9 · Публичная лента корпораций `[srv]`

**Цель.** Оркестратор публикует **матчап на S2** (бой подтверждён) и **итог на S7**;
только публичные факты (без приватного ростера), read-model, server-authority.

**Опора.** AVA-4 (S2), AVA-8 (S7). Тонкий брик — можно частично влить в AVA-4/AVA-8.

**Модель данных.** `AvaFeedStore` (append-only): `entry(id, at, kind: matchup|result,
challenger, target, winner?)`. **Без** ростера/приватных данных.

**API.** `GET /ava/feed` (публичная лента, пагинация newest-first).

**Тесты.** Матчап появляется на accept; итог — на `match.ended`; приватные поля не
утекают.

**DoD.** Лента показывает подтверждённые бои и их исходы.
