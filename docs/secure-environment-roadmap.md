# Безопасное окружение приложения — технический roadmap

> **Направление:** как мы безопасно **эксплуатируем** — рантайм, инфраструктура, хостинг
> (build-time/код — в парном `secure-sdlc-roadmap.md`). Сквозной принцип — **минимальные
> привилегии** на каждом слое. Формат — кирпичики как в `backlog.md`: зона · статус ·
> зависимости · «Готово, когда». Расширяет Блок **F** (сервер) и Блок **SEC** (`backlog.md`),
> не дублирует. Статусы: ✅ · ⏳ · 🔒(dep).
>
> **Замечание о грунтовке:** сетевая/рантайм-модель уже спроектирована в
> `architecture.md` §A01–A10 и §«Сетевая защита» (слоистая оборона, БД/Redis без публичного
> доступа, JWT в WS-рукопожатии, TLS 1.3, rate-limit, идемпотентность). Этот roadmap
> превращает дизайн в задачи. Конкретные версии тулов/провайдеров (2026) перед внедрением
> сверять с первоисточником.

## Принципы

1. **Least privilege везде**: у каждого сервиса/роли/токена — минимум прав (роль БД без DDL,
   read-only FS контейнера, read-only CI-токен, scoped JWT).
2. **Defense-in-depth / zero-trust между ярусами**: клиент не доверенный; сервер ↔ БД/Redis —
   приватная сеть; каждый запрос реавторизуется.
3. **Хранилища данных невидимы из интернета** (нет публичного listener у Postgres/Redis).
4. **Секреты — вне кода и образов** (инъекция из секрет-стора в рантайме, ротация).
5. **Наблюдаемость = детект**: всплеск отклонённых действий — сигнал чита/атаки (A09).
6. **Деградация без краша**: рестарт/обрыв не теряет матч и не открывает доступ (fail-secure).

## Текущая база (факт)

- **Спроектировано** в `architecture.md`: слоистая защита (Cloudflare → reverse-proxy →
  изолированный сервер → приватные БД/Redis), TLS 1.3, JWT-handshake, per-account rate-limit,
  per-player очередь, фильтр видимости перед отправкой, idempotency с TTL.
- **Реализовано**: `maxPayload: 32_768` на WS (`wsServer.ts`); server-authority; **фильтр
  видимости на отправке** — `MatchRoom.broadcastState` диффит per-player дельту против
  `visibleState(playerId)` (`matchRoom.ts`), так что скрытое физически не уходит.
- **Не реализовано** (явные пробелы): JWT/авторизация (handshake принимает любой `?player=`),
  per-player очередь, секрет-менеджмент, логирование/мониторинг, персистентность/бэкапы.
  Всё это — Stage 3.

## Фаза 0 · Идентичность и авторизация на границе `[A01/A07]`

### SE-0.1 · JWT в WebSocket-рукопожатии `[srv][sec]` ✅ — M

**Цель:** заменить доверчивый `?player=` на проверяемую аутентификацию. **Это бирка F7.**
**Подзадачи:** валидация JWT **до** upgrade; короткий TTL + refresh; ротация ключей; **пиннинг алгоритма** (не `none`, не путать HS/RS); привязка к матчу/сессии; отказ → close без деталей.
**Готово, когда:** без валидного токена соединение отклоняется; токен нельзя переиспользовать как чужой.
**Сделано:** `auth.ts` (`verifyJoinToken`/`signJoinToken` на `jose`, в части коммитов помечено «SE-2.1» — это подкомпонент ЭТОЙ бирки, не секрет-стор SE-2.1) — пин алгоритма (нет `none`/HS-RS confusion), `typ`, iss/aud/exp, gate-side `maxTokenAgeSec`; проверка **до** upgrade (`wsServer.ts`), при `auth` — `?player=`/`?nick=` игнорируются; `matchId`/`playerId` токена сверяются с матчем и местом; отказ → 401/403 без деталей; Origin-allowlist. TTL 15 мин. Прошёл состязательное ревью (verified против исходников jose). **Открыто:** ротация ключей / refresh (сейчас один активный ключ, структура готова под JWKS); идентичность на create/join (OIDC) — отдельный трек.

### SE-0.2 · Авторизация на соединении и на сообщении `[srv]` 🔒(SE-0.1) — M

**Цель:** аутентификация ≠ авторизация — проверять права на каждое сообщение.
**Подзадачи:** auth-on-connect (Origin-проверка + токен) + per-message authz (отправитель = владелец действия, матч/сессия совпадают — слой действий это уже умеет, подключить к WS); fail-secure-коды без утечки.
**Готово, когда:** подделанное под другого игрока сообщение отклоняется `E_FORBIDDEN`.

### SE-0.3 · Сервисные идентичности и scoped-токены `[sec]` 🔒(F1) — S

**Подзадачи:** отдельная identity на каждый сервис (сервер/воркер/мигратор); никаких общих admin-кредов; токены с минимальной областью.
**Готово, когда:** компрометация одного сервиса не даёт прав другого.

## Фаза 1 · Сеть и периметр `[A05]`

### SE-1.1 · Cloudflare: DDoS / WAF / rate-limit на краю `[sec]` 🔒(F1) — M

**Подзадачи:** Cloudflare перед сервером — поглощение DDoS L3/4 и L7, WAF, bot-management, edge-rate-limit; правила под наши маршруты.
**Готово, когда:** трафик идёт через край; базовые WAF/rate-правила активны.

### SE-1.2 · TLS 1.3 везде + приватная сеть `[A02]` `[srv][sec]` 🔒(F1) — M

**Подзадачи:** WSS/HTTPS только TLS 1.3; БД/Redis — **только приватная сеть**, нет публичного listener; reverse-proxy (Caddy/Nginx) терминирует TLS/health.
**Готово, когда:** нет нешифрованного трафика; БД/Redis недостижимы из интернета (проверено сканом).
**Операционные шаги:** посурфейсный план перехода на HTTPS/WSS — `docs/https-roadmap.md`.

### SE-1.3 · Скрытие origin + egress-контроль `[sec]` 🔒(SE-1.1) — S

**Подзадачи:** скрыть IP origin (Cloudflare Tunnel / mTLS от края к origin); ограничить исходящие соединения сервера (egress allow-list — анти-SSRF/эксфильтрация, A10).
**Готово, когда:** origin недоступен в обход края; egress ограничен.

**Готово, когда:** есть процедура и лог ротации.

## Фаза 3 · PostgreSQL-хардненинг `[A01/A02/A05]`

### SE-3.1 · Least-privilege роли БД `[srv][sec]` 🔒(F2) — M

**Подзадачи:** app-роль без DDL (только DML по нужным таблицам); отдельная роль для миграций; запрет суперюзера приложению; ограничение по схемам.
**Готово, когда:** приложение не может менять схему/читать чужое; миграции — отдельной ролью.

### SE-3.2 · Шифрование at-rest + in-transit `[A02]` `[srv]` 🔒(F2) — S

**Подзадачи:** TLS `verify-full` от приложения к БД; шифрование тома at-rest; защита connection string.
**Готово, когда:** соединение к БД проверяет сертификат; том зашифрован.

### SE-3.3 · RLS как defense-in-depth `[srv]` 🔒(F2) — M

**Подзадачи:** Row-Level Security на ключевых таблицах (матч/игрок) — второй рубеж к application-authz; политики по владельцу.
**Готово, когда:** даже при баге в коде строки чужого матча/игрока не читаются.

### SE-3.4 · Бэкапы + PITR + проверенный restore `[srv][sec]` 🔒(F2) — M

**Подзадачи:** автоматические **зашифрованные** бэкапы; Point-In-Time-Recovery; **регулярно тестируемый** restore; pgaudit для аудита.
**Готово, когда:** restore из бэкапа проверен на учении; есть RTO/RPO.

## Фаза 5 · Хардненинг контейнера и хоста `[A05]`

### SE-5.1 · Минимальный non-root read-only образ `[srv][sec]` 🔒(F1) — M

**Подзадачи:** distroless/минимальная база (пин по `sha256`); non-root user; read-only root FS; drop Linux capabilities; seccomp/AppArmor; нет секретов в слоях; resource limits (анти-noisy-neighbor/DoS).

### SE-5.2 · Сканирование образа сервера `[sec]` ✅ — S

**Готово, когда:** образ сканируется в пайплайне; база запинена.

## Фаза 6 · Рантайм realtime/WebSocket `[A01/A05]`

### SE-6.1 · Лимиты соединений и сообщений `[srv]` 🔒(F4) — M

**Подзадачи:** Origin-проверка; лимит соединений per account/IP; idle-timeout + heartbeat; **payload cap** (✅ 32KB) проверить; backpressure для медленных клиентов.
**Готово, когда:** злоупотребление соединениями/размером отбивается, медленный клиент не топит сервер.

### SE-6.2 · Rate-limiting действий `[srv][sec]` 🔒(F4) — M

**Цель:** закрыть **отсутствующий** контроль (анти-спам/бот).
**Подзадачи:** per-account/per-connection rate-limit на действия и сообщения; пик ~3 действия/сек (async-жанр) — щедрый лимит; превышение → отказ + лог-аномалия.
**Готово, когда:** флуд действий ограничивается и логируется.

### SE-6.3 · Per-player очередь (анти-double-spend) `[srv]` 🔒(F2) — M

**Цель:** сериализация действий игрока (TOCTOU/двойная трата). **Это бирка F5.**
**Подзадачи:** очередь с concurrency=1 на ключ (player/match) — поверх атомарного редьюсера; согласовать с идемпотентностью.
**Готово, когда:** два одновременных действия одного игрока не вызывают двойную трату.

### SE-6.4 · Фильтр видимости перед отправкой `[srv]` ✅ — M

**Цель:** туман как граница безопасности — не слать невидимое (анти-maphack). **Это бирка F6.**
**Подзадачи:** дельты диффить против `visibleState(playerId)`, не полного состояния; per-peer проекция; тест anti-leak по байтам.
**Готово, когда:** игрок физически не получает данные вне своей видимости.
✅ уже в коде: `matchRoom.ts` — `broadcastState` держит per-player `lastVisible`-базлайн и шлёт `diffState(baseline, visibleState(playerId))`, а `eventVisibleTo` фильтрует и события по туману; welcome/state-снапшоты идут через ту же `viewFor` (тоже поверх `visibleState`).

### SE-6.5 · Масштаб WS без поломки auth/видимости `[srv]` 🔒(SE-6.4) — L

**Подзадачи:** шардинг по матчу (match-actor); фан-аут между инстансами через Redis pub/sub или sticky-сессии; сохранить per-player authz/видимость при масштабе.
**Готово, когда:** матчи разнесены по инстансам, авторизация/туман не ломаются.

## Последовательность

- **Без сервера (сейчас):** проектные кирпичи фиксируются; реализация почти вся 🔒 на **F1**
  (скелет сервера) и далее F2/F3/F4.
- **Первый рубеж к публичности:** SE-0.1 (JWT/F7) + SE-1.\* (сеть/TLS) + SE-6.2/6.4 (rate-limit,
  туман-на-отправке) + SE-2.1 (секреты) — это минимум «не выкатывать без».
- **С персистентностью (F2):** SE-3.\* (Postgres) + SE-6.3 (очередь/F5) + SE-9.1 (рестарт).
- **Соответствие backlog:** F5→SE-6.3, F6→SE-6.4, F7→SE-0.1, SEC-5→SE-5.2.

## Чего НЕ делаем (анти-цели)

Публичный доступ к БД/Redis «на время»; секреты в образе/репо «временно»; доверие клиенту;
доверие WAF вместо параметризации/authz; преждевременный мульти-регион/шардинг до нужды;
логирование PII/секретов; самописная крипта/TLS.
