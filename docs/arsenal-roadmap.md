# Арсенал — персистентный мета-инвентарь · технический роадмап

> Инженерная карта сборки **личного арсенала**: корпуса кораблей, модули, фитинги
> героев и чертежи, которыми аккаунт владеет МЕЖДУ сессиями. Канон уже очерчен
> пунктиром: `metagame.md` («личный арсенал (изученные корабли/модули/герои) +
> аренда корпоративного снаряжения»), `corporation-wars.md` §107 («лоадаут =
> снапшот личного арсенала на момент развёртывания», GDD §5.2), `gdd.md` §4.4
> (сила покупается через мета-экономику — с гайками: soulbound, pity, F2P-кэтчап).
> Слой — **server/meta**; детерминированное ядро арсенал НЕ знает: в матч он
> попадает только снапшотом при сборке сессии, как учёные/шаблоны/герои/мета-гранты
> (snapshot semantics — no live reads). Спутники: `economy-roadmap.md` (EC-* —
> валюты/аукцион/лутбоксы/крафт; арсенал — их СКЛАД), `ship-modules-roadmap.md`
> (SHIP-5 «модули как предметы мета-экономики» 🔒(EC-1.1) — этот роадмап и есть его
> развёртка), `account-level.md` (гейты по уровню), `main-menu.md` §5 (гардрейлы).
>
> Нарезано лесенкой кирпичей (один ≈ один PR ≈ одна сессия, правило `backlog.md`).
> Каждый: **опора** (что уже есть), **модель данных**, **реализация**, **тесты**, **DoD**.

## ARS-0 · Развилки за владельцем `[docs]`

Решить ДО кода (фиксируются здесь же, резолюцией):

1. **Чертежи vs экземпляры.** Чертёж = «разблокирован навсегда, строй за сессионные
   ресурсы» (горизонтальная прогрессия, дух GDD §5.2). Экземпляр = штучный предмет:
   можно продать/потерять (экономика, драма). _Рекомендация — гибрид:_ корпуса и
   базовые модули — чертежи; редкие/заточенные модули (EC-2.1) — экземпляры.
2. **Full-loot?** Терять ли экземпляры при гибели корабля В матче. _Рекомендация —
   НЕТ на старте:_ снапшот неприкосновенен (иначе отвал новичков); sink'и —
   износ (ARS-6) и комиссия аукциона (EC-3.2).
3. **Аренда: судьба предмета при поражении** — возвращается в корп-склад или сгорает
   (это и есть открытый вопрос `metagame.md` §107 «баланс аренды vs личного»).
4. **Валюта аренды** — существующее `influence` (AVA-2) или отдельные «очки
   снабжения» из `metagame.md`. _Рекомендация — influence на MVP_ (одна валюта,
   меньше инфляционных контуров; расщепить позже, если live-ops покажет).

**DoD.** Резолюции вписаны сюда; ARS-1 разблокирован.

## ARS-1 · Схема предмета/чертежа `[data][core]` 🔒(ARS-0, совместно с EC-1.1)

**Цель.** Data-driven модель того, ЧЕМ владеет аккаунт, с валидацией до использования.

**Опора.** `ModuleDefSchema`/`data/modules.json`, hero-каталоги, zod-дисциплина
(A05/A08 — валидировать всё внешнее).

**Модель данных.**
```
ArsenalItem {
  itemId,                 // uuid экземпляра / стабильный id чертежа
  kind: hull | module | hero_fitting | blueprint,
  defId,                  // → data.units / data.modules / data.heroFittings
  grade?,                 // заточка (EC-2.1), только у экземпляров
  soulbound: bool,        // донат/лутбокс — не торгуется (анти-RMT, GDD §4.4)
  durability?,            // износ (ARS-6); отсутствует = не изнашивается
  origin: starter|drop|craft|auction|lootbox|rent,
  acquiredAt
}
```
Схема — zod рядом с EC-1.1 (одна модель предмета на арсенал/аукцион/лутбоксы);
`defId` обязан существовать в каталоге (fail-secure на неизвестный id).

**Тесты.** Парс/отказ схемы; неизвестный `defId` валится; round-trip JSON.

**DoD.** Предмет любого вида описывается данными и валидируется до использования.

## ARS-2 · `ArsenalStore` + стартовый набор `[srv]` 🔒(ARS-1)

**Цель.** Durable-владение: предметы переживают рестарт; у нового аккаунта арсенал
никогда не пуст.

**Опора.** Паттерн `CorpStore` (Memory+Postgres, инварианты в сторе, контракт-тесты);
аккаунты SE-1.x (`accountId`).

**Модель данных.** Таблица `arsenal(item_id PK, account_id, kind, def_id, grade,
soulbound, durability, origin, acquired_at)` + индекс по `account_id`. Мутации:
`grant` (идемпотентно по itemId), `consume/transfer` (атомарно, владелец-guard —
двойная продажа одного экземпляра невозможна), `listOf(accountId)`.

**Реализация.** Стартовый набор — данными (`data/starterArsenal.json` или блок в
манифесте): выдаётся при регистрации (идемпотентно — повторная выдача невозможна).
«Пустой арсенал» не существует как состояние (урок `main.ts:7604`: первый выбор —
один тап, не стена пустых слотов).

**Тесты.** Контракт обоих адаптеров: grant-идемпотентность, атомарный transfer
(гонка двух покупателей), стартовый набор ровно один раз.

**DoD.** Владение переживает рестарт; экземпляр нельзя ни выдать, ни продать дважды.

## ARS-3 · Снапшот в матч `[srv]` 🔒(ARS-2)

**Цель.** Арсенал определяет, из чего собирается лоадаут, и замораживается на сессию.

**Опора.** `SlotAssignment` (учёные/технологии/герои уже сидят так);
`SetupConfig.ships` (лоадауты уже морозятся); AVA-6 лок ростера — готовая точка
снятия снапшота для AvA (GDD §2 «консервация деки»).

**Реализация.** `SlotAssignment.arsenal?: { hulls[], modules[], fittings[] }` —
список doступного, снятый с `ArsenalStore` при сборке матча (обычный) или при ЛОКЕ
ростера (AvA — AVA-7 оркестратор кладёт его в assignments). Валидация лоадаута
против снапшота — на сервере при `unit.build{modules}` поверх существующего
`validateLoadout` (модуль вне снапшота → `E_NOT_OWNED`, fail-secure). Изменения
арсенала ПОСЛЕ снапшота на идущий матч не влияют (детерминизм/реплеи).

**Тесты.** Лоадаут из снапшота строится; модуль вне снапшота → `E_NOT_OWNED`;
продажа предмета после лока не трогает идущую AvA-сессию.

**DoD.** «Что я взял на войну» фиксируется в момент развёртывания и только оттуда.

## ARS-4 · Источники: дроп по месту + сальваж `[srv]` 🔒(ARS-2)

**Цель.** F2P-путь пополнения: играешь → получаешь (GDD §4.4 «кэтчап — сила падает
с матчей, шанс растёт от места»).

**Опора.** SES-2 `match.ended.rewards[player].place` — место уже посчитано;
observe-поток сервера (симуляцию не трогаем — метрик-инвариант);
`battle.resolved` для сальважа.

**Реализация.** Server-side роллер на observe-`end`: таблица дропа — данными
(`data/dropTables.json`: веса по месту, pity-счётчик на аккаунт против чёрных
полос); сальваж — осколки (вход EC-2.2 крафта) победителю боя по таблице от
погибшего состава. ВСЁ на сервере, вне редьюсера: детерминизм ядра не трогается,
серверный RNG — политика EC-2.1.

**Тесты.** Место 1 > место N по ожиданию (стат-тест на фикс-seed'е); pity
гарантирует дроп за K матчей; осколки начисляются победителю и только ему.

**DoD.** Неделя игры без доната ощутимо пополняет арсенал; телеметрия дропа в
JSONL (EC-5.1 позже заберёт).

## ARS-5 · UI: витрина + фильтр Верфи `[proto]` 🔒(ARS-3)

**Цель.** Арсенал видно, им хочется хвастаться; Верфь предлагает только своё.

**Опора.** Хаб-вкладки прототипа (паттерн «★ Прокачка» META-1), кодекс
(`openCodex` — карточки предметов уже есть), Верфь SHIP-2.

**Реализация.** Вкладка «⚔ Арсенал»: сетка предметов (фильтры по kind/grade),
карточка → кодекс, у экземпляров — origin/износ/киллы; Верфь и ростер героев
фильтруются снапшотом. Персист-мост как у META-1 (localStorage по позывному до
серверных аккаунтов на прото-пути).

**Тесты.** uitest-паттерн прототипа: витрина рендерится, фильтр Верфи прячет
неимеющееся, «где добыть» ведёт в кодекс.

**DoD.** Игрок видит свою коллекцию и не может собрать лоадаут из чужого.

## ARS-6 · Корп-склад + аренда `[srv]` 🔒(ARS-3, AVA-8)

**Цель.** «Синергия личного и общего прогресса» (`metagame.md`): корпорация владеет
снаряжением и выдаёт его бойцам на войну.

**Опора.** CORP-0 RBAC (глава/офицер), AVA-2 `influence`, AVA-6/7 (лок ростера +
оркестратор — момент выдачи), аудит-лог корпы.

**Модель данных.** `corp_arsenal` (те же `ArsenalItem`, owner = корпорация) +
`rent(item_id, matchup_id, account_id)` — предмет на войне ровно один раз.

**Реализация.** Глава/офицер выдаёт предмет ростерному бойцу ДО лока (аудит);
предмет въезжает в снапшот ARS-3 как арендованный; после `match.ended` —
возврат/сгорание по резолюции ARS-0.3; износ (`durability--` за войну) — мягкий
sink против инфляции. Опциональная витрина склада в корп-UI (`corporation-ui.md`).

**Тесты.** Выдача только ростерному и только офицером+; предмет не выдаётся в две
войны сразу; возврат/сгорание exactly-once (паттерн refund AVA-4); аудит пишется.

**DoD.** Корп-флагман воюет в руках бойца и возвращается на склад (или гибнет) —
ровно один раз, с записью в аудите.

## Идеи за скоупом кирпичей (паркинг)

- **Именные корабли-ветераны**: killcount на экземпляре корпуса → имя + косметика
  (не статы — гардрейл §5). Дёшево, историй много.
- **Сеты модулей**: 3 предмета серии → косметика/минорный бонус через существующие
  хуки — данные, не код.
- **Страховка флота** (мета-услуга за Варранты) — только если ARS-0.2 решится в
  сторону full-loot.
- **Гейты по уровню аккаунта** (`account-level.md`): слоты арсенала/аукцион
  открываются уровнем — front-loaded кривая.

## Критический путь

`ARS-0 → ARS-1 → ARS-2 → ARS-3` — с этого момента арсенал РЕАЛЬНО ограничивает
лоадауты (играбельная ценность); дальше параллельно `ARS-4` (F2P-приток) и `ARS-5`
(витрина); `ARS-6` — после AVA-8 (нужен исход войны для возврата/сгорания).
EC-2/EC-3 подключаются к готовому складу без изменений в нём.
