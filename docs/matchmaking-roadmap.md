# Матчмейкинг и жизненный цикл матча — технический roadmap

> **Блок 5/7** из `tech-research.md` (🔴-пробел «матчмейкинг/лобби» + Этап 5 игровой цикл).
> Создание/подбор матчей (5–10 игроков), предматчевый экран, полный жизненный цикл
> создание→игра→завершение, условия победы. Формат — кирпичики. Закрывает **Этап 5** и
> бирки **H1/H3** (`backlog.md`). Статусы: ✅ · ⏳ · 🔒(dep).

## Текущее состояние (факт)

`victoryModule` существует в ядре (`GameState.match`, scoreboard, `match.ended` по доминированию/
уничтожению/лимиту/тайм-ауту) и **уже подключён в прототипе** (`prototype/src/game.ts` `MODULES` →
`createKernel`); `checkEnd()` в `prototype/src/main.ts` выводит баннер из авторитетного
`s.match.status==='ended'` (не фейк). На сервере есть **ручной старт-лобби** (хост + гард слотов,
`packages/server/src/matchRoom.ts`) и **пер-игроковый архив** через `MatchRegistry`
(`matchRegistry.ts`; хендлер HTTP `GET /matches` — в `matchApi.ts`/`registerBrowserApi`, смонтирован через `httpRoutes`-шов `wsServer.ts`). Нет формальной машины состояний
матча/ready-check, очереди матчмейкинга, предматчевого экрана. Старт по фракции — `core` (CR-1.3).

## Фаза 0 · Жизненный цикл матча (сервер)

### MM-0.1 · Состояния матча: lobby→active→ended→archived `[srv]` 🔒(F2) — L
**Цель:** управляемый полный цикл матча.
**Подзадачи:** машина состояний матча; переходы (старт по заполнению/таймеру, завершение по victory, архивация); хранение (persistence); события lifecycle для клиента.
**Готово, когда:** матч проходит весь цикл; переходы персистентны и наблюдаемы.
> Частично уже в коде: ручной старт-лобби (хост, гард слотов) в `packages/server/src/matchRoom.ts`; архивация (`ongoing`/`ended` + пер-игроковый архив) в `matchRegistry.ts`. Формальная машина состояний с переходами по заполнению/таймеру — ⏳.

### MM-0.2 · Подключить victoryModule + баннер `[core][proto][srv]` ✅ — S
**Цель:** реальное завершение вместо фейка. **Бирка H1.**
**Подзадачи:** включить `victoryModule` в кернел (прототип и сервер); баннер из `state.match`; завершение матча по `match.ended`.
**Готово, когда:** матч реально завершается по условиям победы; фейковый `checkEnd` удалён.
✅ уже в коде: `victoryModule` в `prototype/src/game.ts` (`MODULES` → `createKernel`); `checkEnd()` в `prototype/src/main.ts` строит баннер из авторитетного `s.match.status==='ended'` (фейка нет).

## Фаза 1 · Лобби и предматч

### MM-1.1 · Лобби (создание/присоединение) `[srv]` 🔒(MM-0.1,AC-0.1) — M
**Цель:** игроки собираются в матч.
**Подзадачи:** создать/найти/присоединиться к лобби; слоты 5–10; готовность; приватные/публичные; права хоста.
**Готово, когда:** игроки собираются в лобби и стартуют матч.
> Частично уже в коде: ручной старт-лобби с ростером, правами хоста и гардом слотов (`packages/server/src/matchRoom.ts`); присоединение через `MatchRegistry` + read-model `GET /matches` (хендлер `matchApi.ts`/`registerBrowserApi` поверх `matchRegistry.ts`, смонтирован через `wsServer.ts`, тест `matchRegistryHttp.test.ts`). Формальный ready-check остаётся ⏳.

### MM-1.2 · Предматчевый экран: фракция/технологии `[cli][srv]` 🔒(MM-1.1,CR-1.1) — M
**Цель:** выбор фракции/стартовых техов. **Бирка H3.**
**Подзадачи:** выбор фракции (CR-1.1) + стартовые бусты (CR-3.1); валидация выбора на сервере; сборка старта по фракции (CR-1.3).
**Готово, когда:** игроки выбирают фракцию/техи; матч стартует с корректным сетапом.

## Фаза 2 · Матчмейкинг

### MM-2.1 · Базовый подбор `[srv]` 🔒(MM-1.1) — M
**Цель:** автоподбор в матч без ручного лобби.
**Подзадачи:** очередь подбора; критерии (регион/режим/размер; рейтинг — позже); таймаут→бот/меньший матч; защита от абуза очереди (rate-limit).
**Готово, когда:** игрок встаёт в очередь и попадает в подходящий матч.

### MM-2.2 · Регион/латентность `[srv]` 🔒(MM-2.1) — S
**Подзадачи:** подбор по региону (для async латентность терпима, но честность окон/UX); согласовать с мульти-регионом (`operations-roadmap.md`).
**Готово, когда:** подбор учитывает регион.

## Фаза 3 · Завершение и пост-матч

### MM-3.1 · Подведение итогов и счёт `[srv]` 🔒(MM-0.2) — M
**Подзадачи:** финальный scoreboard из `state.match`; запись результата в аккаунт/историю; архивация матча (+ event-log для реплея).
**Готово, когда:** итог матча записан в историю аккаунтов; матч архивирован.

### MM-3.2 · История матчей и реплеи `[srv][cli]` 🔒(MM-3.1,PE-1.1) — M
**Подзадачи:** список прошлых матчей игрока; просмотр реплея (CR-0.2 + event-log PE-1.1).
**Готово, когда:** игрок видит историю и может пересмотреть матч.
