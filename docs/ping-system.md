# Система пингов (тактические метки для союзников)

> Ветка `claude/ping-system`. Игрок ставит метки на карте, чтобы предлагать союзникам
> тактику/планы (иконки: меч/движение/атака/защита) и предлагать постройку зданий.
> Открытые вопросы — в `docs/open-questions-visuals-content-pings.md` (Q1–Q15).

## Архитектура: эфемерный серверный сайд-канал (НЕ ядро)

Пинги живут **только в `MatchRoom`** (in-memory), рассылаются владельцу + союзникам,
скрыты от врагов. Они **никогда** не попадают в детерминированный `GameState` — поэтому
не трогают `hashState`, реплеи, манифест и расписание мира. Единственная точка касания
симуляции — **приём build-предложения**, который эмитит обычный `building.construct`
(вся валидация — в существующем хендлере). Подробный разбор «почему не в ядре» — в
артефакте вопросов.

**Зависимость:** настоящего отношения «союзник» в коде нет (combat: «другой владелец =
враг»). Стопгап — статические `teams` в опциях комнаты; единая точка подмены на будущую
дипломатию — `MatchRoom.areAllied`.

## Что реализовано (этот заход) ✅

Протокол (`packages/server/src/protocol.ts`):
- Типы `Ping` / `PingKind` (`mark|move|attack|defend|build`) / `PingAnchor` (`node` XOR `point`).
- Клиентские сообщения `ping.place` / `ping.clear` (+ структурная валидация в `parseClientMessage`).
- Серверные `ping.added` / `ping.removed` (`reason: cleared|expired`).
- Коды ошибок `E_PING_TARGET|UNSEEN|BUILD|RATE` (невалидный `kind` отбрасывается ещё
  структурной валидацией `parseClientMessage` → общий `E_BAD_MESSAGE`).

Хранилище (`packages/server/src/ephemeral.ts`):
- Пинги живут за интерфейсом **`EphemeralStore`** (KV + TTL + scan), in-memory-импл сейчас
  → **Redis одной заменой** на мульти-процессе (см. `docs/tech-stack.md`, триггер №5). Не в
  `GameState` — не трогают `hashState`/реплеи/расписание.

Комната (`packages/server/src/matchRoom.ts`):
- `teams` + `areAllied` + `canSeePing` (privacy: враг физически не получает пинг).
- `handlePingPlace` / `handlePingClear`, relay союзникам, выдача видимых пингов на join
  (читается из стора → поздний джойнер видит уже стоящие метки).
- **Туман у ставящего:** node-якорь требует, чтобы владелец **видел** этот узел
  (`identifiedNodes`), иначе `E_PING_UNSEEN`; point-якорь `{x,y}` — где угодно.
- **build-валидация:** реальное здание (`data.buildings`), иначе `E_PING_BUILD`.
- **Просрочка** (TTL, по умолчанию 5 мин) — ленивая, на стороне стора (как у Redis); клиент
  гасит метку по её `expiresAt`.
- **Анти-спам:** rate-limit (4 / 2с → `E_PING_RATE`, локальный) + лимит 8/игрок с вытеснением старейших.

Тесты (`packages/server/src/pings.test.ts`): союзник видит / враг нет; туман
(`E_PING_UNSEEN`); point-якорь свободен; build принимается; просрочка не реплеится
позднему джойнеру; rate-limit. Гейт зелёный (297 тестов).

## Что дальше (следующий инкремент на этой ветке)

Прототип-UI (`prototype/src/main.ts`):
- long-press / right-click → **радиальный пикер** иконок (первое контекст-меню в прототипе);
- направленные виды (move/attack) — второй тап для `to` (превью-стрелка);
- оверлей-рендер 5 видов (флаг / пунктирная стрелка / пульс-шеврон / щит / гост-иконка здания);
- приём `ping.added`/`ping.removed`, accept/dismiss у получателя;
- **build-accept** у владельца планеты → `act(ME,'building.construct',{planetId,building})`.

Клиент-транспорт (`packages/client/src/multiplayer.ts`): проброс `ping.*` сообщений
(сейчас адаптер их игнорирует — добавить обработчики/коллбэки).
