## Текущее состояние (факт)

Одиночный процесс (durable-Postgres — opt-in по `DATABASE_URL`); деплой-артефакты есть
(`Dockerfile`, `render.yaml`, `deploy/docker-compose.yml`) и базовая наблюдаемость тоже
(структурные pino-логи, минимальный `/metrics`). Горизонтального масштаба, игровой
телеметрии и модерации нет; OTel/Prometheus/Grafana/Sentry — нет.
SBOM/сканеры есть (SEC-0). Контент — `data/*.json` + manifest (валидируется), но без пайплайна
публикации/версий без даунтайма.

## Фаза 0 · Наблюдаемость (заложить с первого сервера)

### OPS-0.1 · Структурные логи + трейсинг `[srv][sec]` 🔒(F1) — M
**Цель:** видеть, что происходит, без PII/секретов.
**Подзадачи:** структурный лог (pino) + OpenTelemetry-трейсинг сквозь match-actor; корреляция по матчу/игроку; централизованный сбор. **(Связка SE-8.1.)**
**Готово, когда:** запрос/действие трассируется end-to-end; логи без утечек.

### OPS-0.2 · Метрики и трекинг ошибок `[srv][sec]` 🔒(OPS-0.1) — S
**Подзадачи:** Prometheus/Grafana (соединения/матч, исходящие байты/клиент, dropped/slow, due-события, лаг планировщика); Sentry (ошибки, без PII). **(= SE-8.3.)**
**Готово, когда:** есть дашборд состояния и поток ошибок.

### OPS-0.3 · Игровая телеметрия/аналитика `[srv]` 🔒(OPS-0.1) — M
**Цель:** данные для баланса async-экономики и удержания.
**Подзадачи:** события геймплея (без PII) → аналитическое хранилище; базовые воронки/удержание/баланс-метрики; разделение продуктовой телеметрии и тех-метрик.
**Готово, когда:** ключевые геймплей-события собираются и доступны для анализа.

## Фаза 1 · Деплой и устойчивость

### OPS-1.1 · Деплой при долгих WS-соединениях `[srv][sec]` 🔒(F1) — M
**Цель:** релиз не рвёт идущие матчи.
**Подзадачи:** rolling/blue-green; **drain + graceful shutdown** (дренить WS, клиент reconnect+resume — SV-2.2); миграции БД без даунтайма (expand/contract); пин образов по digest (SE-5).
**Готово, когда:** деплой проходит без потери матчей; клиенты переподключаются.

### OPS-1.2 · Нагрузочное и хаос-тестирование `[sec]` 🔒(OPS-0.2) — M
**Цель:** проверить до публичности.
**Подзадачи:** k6/Artillery на N параллельных матчей + WS-обрывы; хаос (kill инстанса/БД) → проверка восстановления (SE-9); пороги в CI/прогон по релизу.
**Готово, когда:** известна ёмкость; восстановление после сбоев проверено.

### OPS-1.3 · Статус-страница и инцидент-комм `[docs][sec]` 🔒(OPS-0.2) — S
**Подзадачи:** статус-страница; runbook'и инцидентов (= SE-9.2); канал уведомлений игроков.
**Готово, когда:** при сбое есть статус и регламент коммуникации.

## Фаза 2 · Масштаб

### OPS-2.1 · Шардинг по матчу + горизонт `[srv]` 🔒(SV-0.2,F2) — L
**Цель:** много матчей на многих инстансах.
**Подзадачи:** распределение match-actor по инстансам; fan-out (Redis pub/sub) — = SV-4.1; балансировка/перемещение матчей; ёмкость на инстанс.
**Готово, когда:** матчи распределяются и масштабируются горизонтально.

### OPS-2.2 · Мульти-регион `[srv][sec]` 🔒(OPS-2.1) — L
**Цель:** глобальная аудитория Bytro-жанра.
**Подзадачи:** регион хостинга по подбору (MM-2.2); глобальные аккаунты/лобби, региональные матчи; latency/данные по регионам; согласовать с приватностью (SE-10).
**Готово, когда:** игроки играют в близком регионе; аккаунты глобальны.

### OPS-2.3 · Стоимостная модель `[docs]` ⏳ — S
**Подзадачи:** посчитать always-on 24/7-матчи: Redis-горячее/Postgres/трафик/инстансы на N матчей; пороги для решений масштаба/региона.
**Готово, когда:** есть модель стоимости на единицу матча/игрока.

## Фаза 3 · Live-ops и контент-пайплайн

### OPS-3.1 · Пайплайн публикации контента + версии `[data][sec]` 🔒(CR-0.1) — M
**Цель:** данные-первый дизайн без даунтайма/поломок матчей.
**Подзадачи:** валидация/версионирование `data/*.json` (zod ✅) в пайплайне; публикация новой версии данных; новые матчи — на новой, идущие — на своей (CR-0.1); откат.
**Готово, когда:** контент публикуется версионно, не ломая идущие матчи.

### OPS-3.2 · Feature-flags и A/B `[srv]` 🔒(F1) — M
**Подзадачи:** флаги для постепенного выката/kill-switch (= SE-9.2); базовый A/B для балансных экспериментов; согласовать с детерминизмом (флаг как часть config матча).
**Готово, когда:** фичу можно включать постепенно и гасить мгновенно.

### OPS-3.3 · Тулинг баланса экономики `[docs][core]` 🔒(PE-1.1) — M
**Подзадачи:** симуляции/дашборды по телеметрии (OPS-0.3) для настройки констант; прогон сценариев на ядре; рекомендации правок данных.
**Готово, когда:** баланс настраивается данными по симуляциям/метрикам.

## Фаза 4 · Модерация и коммуникации

### OPS-4.1 · Модерация и анти-абьюз `[srv][sec]` 🔒(AC-0.1) — M
**Цель:** имена/чат/дипломатия — без абьюза.
**Подзадачи:** фильтр имён/чата; репорты; очередь модерации; баны (увязать с GI-3.1); хранение жалоб.
**Готово, когда:** игрок может пожаловаться; модератор разбирает; санкции применяются.

### OPS-4.2 · Транзакционная почта/коммуникации `[srv][sec]` 🔒(F1) — S
**Подзадачи:** провайдер транзакционной почты (верификация/восстановление — AC-1.2, уведомления безопасности); анти-спам/rate-limit; без утечки.
**Готово, когда:** письма доставляются надёжно и безопасно.
