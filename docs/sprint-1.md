# Спринт 1 — риски и пробелы → задачи (из инженерного ревью)

> Операционализация жёсткого ревью `tech-research.md`: «⚠️ риски» и «🔴 пробелы» превращены
> в **конкретные задачи с Definition of Done**, в порядке brutal-priority. Зафиксированы
> решения (чтобы не остались «опциями» в роадмапах). Один пункт уже сделан **кодом**, не текстом.
> Каждая задача указывает, в какой роадмап-блок она ложится биркой.

## Зафиксированные решения (из ревью — больше не «опции»)

1. **Auth = внешний OIDC-провайдер, своё для прода НЕ пишем.** Self-hosted (Keycloak/Zitadel)
   или managed (Clerk/Auth0). Сэкономленные месяцы → в анти-чит и тесты. → `accounts-roadmap.md`.
2. **Планировщик = pg-boss по умолчанию** (очередь на Postgres). Redis/BullMQ — только если
   Redis уже нужен для горячего состояния/кэша. Минус одна инфра-зависимость на старте. →
   `persistence-roadmap.md`.
3. **Миграция сейвов = backward-compatible-additions-only + версионирование схем (zod) +
   ленивая миграция при чтении.** Никакого массового ETL на деплое. → `core-roadmap.md` /
   `persistence-roadmap.md`.
4. **F6 (туман-на-отправке / AOI) — архитектурный фундамент сети, а не «фича».** Решается в
   дизайне протокола **до** первого боевого клиентского кода (иначе переделка ×3). →
   `server-roadmap.md`.
5. **JSONB-дисциплина:** запрашиваемые/индексируемые поля — в нормализованные колонки; JSONB —
   только для opaque-состояния, не «свалка». → `persistence-roadmap.md`.

## Спринт 1 — порядок (brutal priority)

### S1.1 · Дизайн F6/AOI в протоколе (необратимо) `[srv]` 🔴 — L → SV-3.1/F6
**Почему:** переход на per-player фильтрацию после боевого клиента стоит ×3.
**Задачи:** зафиксировать, что сервер шлёт per-player проекцию `visibleState`, а не полное
состояние; дельты диффятся против проекции игрока; спайк на текущем dev-сервере; тест **anti-leak
по байтам** (игрок не получает невидимое).
**DoD:** есть тест, падающий при утечке невидимых данных; протокол задокументирован как
per-player-by-default; клиентский код пишется уже под него.

### S1.2 · Выбрать и подключить OIDC-провайдер `[srv][sec]` 🔴 — L → AC-1.1/AC-1.3 (F7)
**Почему:** свой auth — мина; не наша ценность.
**Задачи:** выбрать self-hosted vs managed (критерии: стоимость, контроль, время); спайк OIDC-flow;
выпуск JWT из сессии; валидация JWT **до** WS-upgrade (issuer/aud/exp/alg-pin).
**DoD:** WS принимает только валидный JWT от провайдера; `?player=` отключён вне dev.

### S1.3 · Тест graceful restart с активными клиентами ✅ СДЕЛАНО (этот коммит) `[srv]` → SV
**Что сделано:** найден и **исправлен** изъян — `wsServer.close()` не дренил соединения
(`httpServer.close()` висел на живом WS). Добавлен дренаж (1001 + terminate-страгглеров) +
интеграционный тест `restart.test.ts`: drain + reconnect-resync проверены.
**Остаток (гейтится F2):** durable-resume — сейчас снапшот переносится в памяти; в проде он из
персистентности. → `persistence-roadmap.md` PE-0.1.
**DoD:** ✅ close() резолвится с активными клиентами; клиент получает 1001; новый сервер
восстанавливает игрока из сохранённого состояния (239 тестов).

### S1.4 · Стратегия миграции сейвов `[core][data]` 🔴 — M → CR-0.1 / PE-1.2
**Почему:** обновление не должно ломать матчи длиной в недели (продуктовый риск).
**Задачи:** реестр версий схемы (zod) per-match; правило **backward-compatible additions only**;
**ленивая** миграция при чтении (мигратор N→N+1), не массовая; round-trip тест.
**DoD:** матч версии N грузится после апдейта; есть тест миграции; правило закреплено в CONTRIBUTING.

### S1.5 · Матчмейкинг + контент-пайплайн — ТОЛЬКО ПОТОМ (Спринт 2) → MM / OPS-3.1
Не начинаем, пока S1.1–S1.4 не закрыты.

## Слепые зоны → задачи (ревью §3)

### S1.6 · Детекция десинхрона (определённость, не вера) `[core][srv]` 🚧 — M → CR-0.3
**Почему:** golden-RNG ≠ гарантия, что клиент/сервер не разошлись в рантайме.
**Задачи:** ✅ канонический **хеш состояния** (`hashState` в `shared-core`, `state/hash.ts`:
key-sort → серверное и дельта-восстановленное совпадают; golden + дельта-round-trip); ⏳ периодическая
сверка хеша сервер↔клиент (или shadow-mode) — нужен сервер; расхождение → форс-ресинк + алёрт.
**DoD:** ✅ искусственный десинк ловится сверкой хешей в тесте (`hash.test.ts`); ⏳ рантайм-алёрт (с сервером).

### S1.7 · DR для JSONB: измеренные RTO/RPO `[srv][sec]` 🔴 — M → PE-3.1
**Почему:** restore большой JSONB-таблицы может занять часы; потеря недель прогресса = смерть.
**Задачи:** **измерить** время restore на репрезентативном объёме; зафиксировать RTO/RPO; учение
restore; решить судьбу горячего слоя (Redis AOF vs восстановление из Postgres-снапшота).
**DoD:** известны измеренные RTO/RPO; учение restore пройдено.

### S1.8 · Бизнес-метрики с первого дня `[srv]` 🟠 — M → OPS-0.3
**Почему:** OTel/Prometheus покрывают CPU/RAM/WS, но не «активные матчи / средняя длина хода /
доля ошибок валидации действий».
**Задачи:** инструментировать активные матчи, среднюю длину хода, rate ошибок валидации
действий, длину очереди планировщика; дашборд.
**DoD:** дашборд показывает эти бизнес-метрики, отдельно от тех-метрик.
