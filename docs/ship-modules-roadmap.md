# Модули кораблей (лоадаут) — технический roadmap

> **Недостающий фундамент.** Корабельные «модули» уже разбросаны по докам как готовая
> вещь — модули на аукционе (`economy`), пусковой ракет (`missiles` MS-0.1), ремонтный
> и реген-щита (`shields` SH-1.1/2.3), лоадаут эскадрилий (`squadrons` SQ-0.2) — но
> **in-game системы лоадаута в коде НЕТ**: у `UnitStack` (gameState.ts:27-33) только
> `{unit, count, hp?}`, у флота нет поля модулей, нет схемы `ModuleDef`, нет действия
> «надеть». Этот роадмап создаёт **фундамент** (каталог модулей в данных + слоты на
> корабле + надеваемый лоадаут на стейте + чистый хелпер «эффективных статов» +
> действие equip) и два первых модуля на нём: **(1) грузовой расширитель** (плоский +N
> к трюму) и **(2) мобильная фабрика дройдов** (производство наземной дивизии в трюме на
> ходу). «Модуль» здесь = ИГРОВОЕ снаряжение, а механика реализуется КАК новый
> кернел-модуль `shipModules` + данные (CLAUDE.md «New game mechanic = new module + maybe
> JSON»). Парные доки: `economy-roadmap.md` (инвентарь/P2W-тормоза), `gdd.md` §6.1
> (модульность кораблей — канон), §5.2/§7.5/§4.4, `missiles`/`shields`/`squadrons`-роадмапы
> (плагины в тот же фрейм), `mechanics-roadmap.md`. Формат — кирпичики:
> `id · зона · 🔒deps · размер · подзадачи · «Готово, когда»`. Зоны: `[data]`·`[core]`·`[cli]`·`[srv]`.

## Статус реализации (обновляется по коду)

**Сделано** (ветка `claude/designer-role-discussion-zxler6`) — фундамент лоадаута + меню, с
двумя осознанными отходами от плана ниже **по прямому решению владельца**:

- **Слоты — ТИПИЗИРОВАННЫЕ** (`weapon` / `defense` / `utility`, `UnitDef.slots`), не скалярный
  `slotCost`-бюджет. Модуль встаёт только в свой тип; один экземпляр `ModuleId` на стек.
- **Лоадаут фиксируется ПРИ ПОСТРОЙКЕ и залочен**: едет в payload `unit.build` → штампуется на
  стек; **экшена переоснастки нет** (лок by omission). Заменяет порт-`loadout.equip/unequip` (SM-0.6).

Реализованные брики: `ModuleDef` + `data/modules.json` (SM-0.1, +рефайны анти-само-баф и
анти-P2W-soulbound) · `UnitStack.modules` + `slots` (SM-0.2) · лоадаут-aware идентичность стека
(SM-0.3) · `effectiveStats` (SM-0.4) и маршрутизация **всех** потребителей статов через него —
`sumUnitStat`, боевые пулы hp/щита/урона, `applyRetreatToll`, `artilleryPower/Range`,
`fleetBaseSpeed`, реген/ремонт (SM-0.5, расширено на боевые статы осознанно — владелец хочет
боевые модули) · проводка `unit.build{modules}` с оплатой корпус+модули и `validateLoadout`
(аналог SM-0.6, но build-time) · клиентский view-model меню `loadoutEditor` (аналог SM-0.7, под GUI).
**golden не тронут** — стеки без `modules` бит-в-бит.

**Не сделано:** экономика/добываемость модулей и P2W-гардрейлы (SM-1.2 / SM-2.6), фабрика дройдов
(SM-2.*), отдельный CLI. Каталог — сид из 6 модулей (трюм · радар · двигатель · наведение · плиты · щит).

## 0. Модель (read first)

- **Модуль = `ModuleDef` в данных** (`data/modules.json`, новый фрагмент + bump
  `manifest.json`, zod до ядра — A05/A08): `{ id, slotCost, allowed?, effects, tag,
  soulbound?, level? }`. Расширяемо данными, как `UnitDef`.
- **Два канала эффекта** (раздельно по оси баланса): (a) `effects.stats` — **плоские
  аддитивные дельты статов** (грузовой расширитель = `+N` к `cargoCapacity`); (b)
  `effects.enables` — **флаг, разблокирующий действие/способность** носителя (фабрика
  дройдов, пусковой ракет, ремонт) — проверяется владеющим модулем, хук статов не трогает.
- **`tag: 'horizontal' | 'vertical'`** (как `squadrons` SQ-0.2): чистый барьер §5.2 —
  боевые стат-баффы (vertical) гейтятся P2W-правилами; cargo/launcher/repair (horizontal) —
  нет. Донат/лутбокс-источник **не несёт `vertical`** (инвариант + тест, не текст).
- **Слоты — opportunity-cost** (§7.5): стат `moduleSlots` на корабле (default 0 → нет
  слотов = модули не носит). Слотов **мало, они конкурируют** (трюм ИЛИ фабрика ИЛИ
  пусковой) — это и есть выбор, а не бесконечный стек. **Один экземпляр `ModuleId` на
  стек**, тиры — через отдельные `ModuleDef` (T1/T2/T3), НЕ стакованием.
- **Лоадаут — per-stack** (`UnitStack.modules?: ModuleId[]`, решение пользователя 1a):
  массив строк → JSON-сериализуемо, переживает `deepClone`, эффект ×count (стек из N
  одинаково-снаряжённых кораблей). **Лоадаут — часть идентичности стека** (см. SM-0.3) —
  иначе `addUnits`/`findHealthyStack` молча сольют стеки с разным лоадаутом.
- **Эффективные статы — чистый хелпер**, НЕ обёртка-агрегат: `sumUnitStat`
  (stacks.ts:22-31) **уже итерирует per-stack** (`s` в руках), поэтому
  `effectiveStats(def, s, data)` читает `s.modules` из стейта и не требует
  `HandlerContext`. База = `def.stats`; нет модулей → ровно `def.stats` (base-default, не
  падать).
- **Фундамент возит только горизонталь** (cargoCapacity + `enables`). Боевые
  (vertical) модули — броня/урон/скорость — **вне этого фундамента**; для них запинены
  правила-предусловия (hp↔пул, speed↔ETA, P2W) — см. инварианты.

## Фаза 0 · Лоадаут-фундамент `[data][core]`

### SM-0.1 · Схема `ModuleDef` + каталог `data/modules.json` `[data]` 🔒(—) — M
**Подзадачи:** `ModuleDefSchema` (zod, по образцу `UnitDefSchema` schemas.ts:39): `id`,
`slotCost: z.number().int().positive()`, `allowed?: { domain?, traits?: string[], units?: string[] }`
(модуль ставится, если корабль удовлетворяет ВСЕМ заданным полям), `effects: { stats?:
Record<string,number>, enables?: string[] }`, `tag: z.enum(['horizontal','vertical'])`,
`soulbound?`, `level?`. `GameDataSchema.modules: z.record(z.string(), ModuleDefSchema)`.
**zod-refine (закрывает эксплойты):** `effects.stats` **НЕ содержит `moduleSlots`** (и
слото-влияющих ключей — иначе модуль расширяет свою же ёмкость, вертикаль); **`vertical`
запрещён на `soulbound`-экземплярах** (донат/лутбокс не покупает боевую вертикаль).
Фрагмент `data/modules.json` + bump `manifest.json`; включить в композицию загрузчика.
**Готово, когда:** модуль добавляется данными; `parseGameData` валидирует; кривой (slotCost≤0,
`moduleSlots` в `effects.stats`, `vertical`+`soulbound`) — отвергается; тест композиции.

### SM-0.2 · Стат `moduleSlots` + поле `UnitStack.modules` `[data][core]` 🔒(SM-0.1) — S
**Подзадачи:** `moduleSlots: z.number().nonnegative().default(0)` в `UnitStatsSchema`
(schemas.ts:17-37). `UnitStack.modules?: ModuleId[]` (gameState.ts:27-33), опционально.
**Свободные слоты считаются от БАЗОВОГО `def.stats.moduleSlots`** (не эффективного — анти
само-усиление). Пример корабля со слотами в `data/units.json`. Без применения эффекта.
**Готово, когда:** `moduleSlots` парсится (default 0 не ломает существующее); `UnitStack` с
`modules:[...]` сериализуется/`deepClone`-ится без потерь; round-trip JSON-тест.

### SM-0.3 · Лоадаут-aware идентичность стека `[core]` 🔒(SM-0.2) — M
**Подзадачи:** `findHealthyStack`/`addUnits` (stacks.ts:5-18) сейчас мёржат healthy-стеки
по `unit`+`hp===undefined`, **игнорируя `modules`** → два стека одного корпуса с разным
лоадаутом молча сольются (потеря/подмена модулей при `army.load`/выходе в `landing`).
Включить **лоадаут-сигнатуру в ключ слияния**: стеки с разным `modules` НЕ мёржатся.
**Готово, когда:** load/unload двух одинаковых корпусов с РАЗНЫМ лоадаутом их не сливает;
переживает JSON round-trip; регресс-тест на анти-мердж.

### SM-0.4 · Хелпер `effectiveStats(def, stack, data)` `[core]` 🔒(SM-0.2) — M
**Подзадачи:** кернел-модуль `shipModules`. Чистая функция: база `def.stats`, для каждого
`m ∈ stack.modules` прибавляет `data.modules[m]?.effects.stats` (неизвестный модуль —
graceful-пропуск, как `sumUnitStat` пропускает unknown unit). Нет модулей → ровно
`def.stats`. Детерминизм, порядок — фикс. порядок `modules[]`. **Опц. seam:** выставить
как value-хук `unit.effectiveStats` (base-default `def.stats`), чтобы будущие кернел-модули
(террейн/герой) пайплайнили — но фундаменту достаточно чистого хелпера.
**Готово, когда:** без модулей `=== def.stats`; модуль `+2 cargoCapacity` даёт `+2`;
неизвестный — игнор; чисто/детерминированно; unit-тест.

### SM-0.5 · Маршрутизация `cargoCapacity` через эффективный лукап `[core]` 🔒(SM-0.4) — M
**Подзадачи:** `sumUnitStat` (stacks.ts:22-31) читает `effectiveStats(def, s, data)[stat] ?? 0`
вместо `def.stats[stat] ?? 0` — он уже per-stack. Это даёт эффект `fleetCapacity`
(army.ts:14-16) и контр-проверке `free` (army.ts:76). **Скоуп фундамента —
ТОЛЬКО `cargoCapacity` носителя.** `army.ts:77 cargoSize` **НЕ маршрутизируем**: это стат
ГРУЗИМОГО наземного юнита (у него нет ship-лоадаута) — спекулятивная проводка/категориальная
ошибка, пока нет наземных модулей. **golden:** стеки без `modules[]` не меняют существующие
combat/route/army-значения (формулировка «бит-в-бит» сужена до «без модулей»).
**Готово, когда:** надетый расширитель поднимает эфф-`cargoCapacity` и `fleetCapacity` его
видит; пустой лоадаут не меняет golden; combat/route не затронуты; тест.

### SM-0.6 · Действие `loadout.equip` / `loadout.unequip` `[core]` 🔒(SM-0.5) — M
**Подзадачи:** имя `loadout.*` (не `module.*` — не путать с кернел-модулями). Зеркало
`army.load` (army.ts:67-95): флот **idle + пристыкован к СВОЕЙ планете-порту**; цель —
конкретный `UnitStack`; гейты: `slotCost ≤ свободные базовые слоты`, `allowed`-предикат,
**`soulbound`-владелец == `action.playerId`** (привязка к игроку, анти-RMT), наличие в
инвентаре (экономика, вне фундамента). **equip/unequip запрещён, когда `stack.hp!==undefined`**
(стек повреждён/в бою — зеркало idle-правила; защита пула, см. инвариант hp). Снятие
расширителя при перегрузе (несомое > базовой ёмкости) → **`E_OVER_CAPACITY`** (отказ, без
авто-выгрузки — не мутировать несомую армию молча). fail-secure стабильные коды.
**Готово, когда:** equip в порту на корабль с местом проходит; нет слота/не-в-порту/в-бою/чужой
soulbound/перегруз — стабильный отказ; снаряжённый корабль несёт эффект; тест.

### SM-0.7 · CLI лоадаута `[cli]` 🔒(SM-0.6) — S
**Подзадачи:** equip/unequip в порту; просмотр слотов (занято/свободно) и **эффективных vs
базовых статов** стека; пометка `tag`/`soulbound`.
**Готово, когда:** видно лоадаут, свободные слоты и как модуль меняет статы.

## Фаза 2 · Мобильная фабрика дройдов `[data][core]`

> **Отход от planet-only.** `unit.build` (construction.ts:261-298) строит **только на
> планете**, выход в `planet.garrison`. Фабрика — **fleet-scoped производство**: за ресурсы
> собирает **дивизию дройдов** в трюм флота (`fleet.landing`), **корабль может двигаться**
> во время сборки (полётная фаза), **вступление в бой замораживает** сборку, модуль — **только
> в транспорт**. Зафиксированные решения пользователя.

### SM-2.1 · Семейство «дройды» (новый контент) `[data]` 🔒(—) — S
**Подзадачи:** 2-3 роли наземных юнитов `domain:ground`, дёшево/расходно (`cost`/`hp`/
`cargoSize`/`upkeep` относительно `drop_infantry` — баланс). Это **готовый блок** (§7.5):
фабрика собирает **только дройдов**, не «любой юнит» (иначе мобильная верфь под всё).
**Готово, когда:** дройды парсятся; грузятся/десантируются как обычная наземка; кривые отвергаются.

### SM-2.2 · Трейт `transport` + модуль-фабрика `[data]` 🔒(SM-0.1, SM-2.1) — S
**Подзадачи:** в `data/units.json` **нет трейта `transport`** (есть artillery/immobile/…;
транспортность сейчас = `cargoCapacity>0`). Пометить транспортные `UnitDef` трейтом
`transport` (новый контент) — якорь для `allowed`. `ModuleDef` фабрики: `allowed:{ traits:
['transport'] }` (только транспорт, решение №3), `effects.enables:['fleet.assembleDroids']`,
`tag: horizontal`.
**Готово, когда:** фабрика ставится на транспорт и отвергается на боевом корабле; `enables`
разблокирует действие сборки только при надетом модуле; тест.

### SM-2.3 · `fleet.assembleDroids` — fleet-scoped производство `[core]` 🔒(SM-2.2) — L
**Подзадачи:** действие сборки (гейт: у флота надета фабрика, `effects.enables`). **`payCost`
из пула игрока** (как construction). **Capacity-gate:** заказ ≤ свободная **эффективная**
`fleetCapacity` (переиспользовать проверку army.ts:76 — НЕ переполнять трюм, закрыть
in-flight-overflow). **НЕ требует idle/дока** (двигается во время сборки). **Order-time gate:**
нельзя заказывать, пока `fleet.battleId != null` (в бою — зеркало `E_BOMBARDED` на входе
`construction.ts:271`). **In-progress build — явная запись на стейте флота** (напр.
`fleet.assembling`), дедуп против НЕЁ (не против `state.scheduled`, иначе re-defer создаёт
ложные дубли). `scheduleCompletion(buildTimeHours)` → `droidFactory.complete`.
**Готово, когда:** заказ тратит ресурсы и планирует выход; заказ сверх ёмкости / в бою / без
модуля — стабильный отказ; повтор не дублирует; тест.

### SM-2.4 · Завершение + бой-фриз + ОГРАНИЧЕННЫЙ re-defer `[core]` 🔒(SM-2.3) — L
**Подзадачи:** на `droidFactory.complete`: **forfeit**, если `state.fleets[fleetId]` отсутствует
ИЛИ сменил владельца (флоты **уничтожаются, не захватываются** — не импортировать planet-
семантику). Если `fleet.battleId != null` (в бою) **ИЛИ** трюм полон → **re-defer** (зеркало
`construction.ts:309-313`, шаг `MS_PER_HOUR/timeScaleOf`), **НО с границей**: `deferUntil =
ctx.now + maxFreezeHours` (детерминированно через `Context.now`/timeScale) / счётчик попыток →
по превышении **дед-леттер в `failures` + возврат ресурсов** (анти-зомби-событие, liveness).
Иначе `addUnits(fleet.landing, droid, count)` (лоадаут-aware, SM-0.3). **«Двигается во время
сборки» = только полётная фаза:** `startBattle` обнуляет `fleet.movement` (combat.ts:207),
т.е. бой и движение не сосуществуют — фриз реализуется именно как re-defer завершения; флот,
прижатый боем на ребре (`battleId` задан), достраивает после релиза (combat.ts:336).
**Готово, когда:** не-в-бою с местом → дройды в трюм; в бою/полный трюм → пауза; вечная
заморозка → дед-леттер с возвратом (не висит в `scheduled`); погиб носитель → инвестиция
сгорает; тест.

### SM-2.5 · Рендер фабрики `[cli]` 🔒(SM-2.3) — S
**Подзадачи:** прогресс сборки, заморозка в бою/при полном трюме, выход дивизии в трюм.
**Готово, когда:** видно сборку, паузу и пополнение десанта.

### SM-2.6 · Фейрнес/P2W-гард фабрики `[data][srv]` 🔒(SM-2.4, economy) — M
**Подзадачи:** модуль earnable F2P (не донат-эксклюзив; soulbound-если-донат). **Контр-игра
механически доступна:** транспорт-фабрика хрупкий и убиваем (потеря флота = потеря несобранного
+ ресурсов), бой замораживает сборку; **cap параллельных фабрик на игрока** + **`upkeep`
дройдов кусается** в трюме, чтобы кочующая офлайн-фабрика имела стоячую цену, которую защитник
переигрывает измором. Не «кнопка мгновенной армии»: гейт = время сборки + перехват + ёмкость.
**Готово, когда:** фабрика earnable F2P; есть стоячая цена и дешёвый перехват; собранная дивизия
не обходит ёмкость трюма; раскрытие шанса дропа фабрики по правилам economy.

## Инварианты (нарушение = баг)

- **Хук эфф-статов — единственный путь** для carrier-модифицируемых статов: нет потребителя,
  читающего сырой `def.stats[x]` в обход (для статов, которые модули меняют). Пока-немодифицируемые
  top-level поля (`upkeep`/`signature`/`superUnit`/`line`) — явно вне хука.
- **`moduleSlots` не модифицируется модулями** (zod-refine); свободные слоты — от **базового**
  стата. Один экземпляр `ModuleId` на стек; тиры — отдельными `ModuleDef`, не стакованием.
- **Лоадаут — часть идентичности стека** (no silent merge, SM-0.3); `modules[]` — массив строк
  (JSON-сериализуемо, переживает `deepClone`).
- **`tag` horizontal|vertical:** донат/лутбокс-источник не несёт `vertical`; боевой (vertical)
  модуль обязан быть F2P-earnable + дешёвая общедоступная контра + раскрытие шансов (§4.4/§5.2).
- **hp-граница (предусловие armor-модулей):** эффективный `hp` — только для свежих стеков
  (`hp===undefined`); equip запрещён при `hp!==undefined`; **никогда не ретро-масштабировать
  существующий `stack.hp`-пул** (combat.ts:128-140: `ceil(pool/perShip)` иначе молча убьёт корабли).
- **speed-граница (предусловие двигатель-модулей):** через `fleetBaseSpeed` (route.ts:20) —
  единый источник авторитетного плеча И клиентской ETA (route.ts:4-10,108-110), чтобы офлайн-будилка
  прибытия совпадала с сервером; `fleet.speed`-хук — только terrain/transient.
- **`soulbound` привязан к ИГРОКУ** (equip проверяет владельца); передача несущей сущности снимает
  или блокирует soulbound-модуль (анти-RMT).
- **Fleet-scoped производство:** выход **capacity-gated** (не переполнять трюм); **ограниченный
  re-defer** (дедлайн/дед-леттер, не зомби-событие в `scheduled`); дедуп против записи на флоте, не
  против очереди; forfeit по `fleets[id]` отсутствует/сменил владельца (флоты уничтожаются, не
  захватываются).
- **golden:** стеки без `modules[]` не меняют существующие combat/route/army-golden; сценарии **с**
  модулями получают свои зафиксированные значения (CLAUDE.md инвариант №1).
- **Детерминизм / fail-secure / server-authority / модули-через-шину** — как везде; кернел и
  `createKernel` не трогаем, механика — новый модуль `shipModules`/`droidFactory` + данные.

## Касание ядра

Кернел НЕ трогаем (новый модуль + хелпер + данные). **Осознанно** правим общих потребителей статов
в `shared-core`: `util/stacks.ts` (`sumUnitStat` → эффективный лукап; идентичность слияния стеков) и
читателей `cargoCapacity` в `army.ts` — иначе надетый модуль физически не влияет на ёмкость. golden
для боевой математики обновляется осознанно **только** для сценариев с модулями.

## Последовательность

`SM-0.1`(схема) → `SM-0.2`(слоты+поле) → `SM-0.3`(лоадаут-aware мердж) → `SM-0.4`(хелпер) →
`SM-0.5`(маршрутизация cargoCapacity, осознанный golden) → `SM-0.6`(equip) → `SM-0.7`(CLI) →
`SM-1.*`(расширитель — простейший потребитель, валидирует хук) → `SM-2.1`(дройды) → `SM-2.2`(трейт+
модуль) → `SM-2.3`(assemble) → `SM-2.4`(complete/фриз/дед-леттер) → `SM-2.5`(CLI) → `SM-2.6`(фейрнес).
Развилки хранения лоадаута (SM-0.3) и гранулярности слотов фиксируются рано — от них зависит вся
маршрутизация и баланс.
