# Map roadmap — секторная карта (плоская модель)

Технический роадмап по **карте**: модель данных, настраиваемые пути, точки
захвата, карта-как-контент и инструменты. Брики в формате репозитория
(`фаза · id · зона · статус · deps · «Готово, когда»`). Парные доки: `gdd.md`
(дизайн), `architecture.md` (ядро), `core-roadmap.md` (механики), `metagame.md`
(мета-слой), `cross-platform-roadmap.md` (рендер CP4/Голопульт).

Зоны: `[core]` shared-core · `[data]` контент/схемы · `[cli]` клиент/рендер ·
`[tools]` редактор/генерация · `[docs]` дизайн.

---

## 0 · Зафиксированная модель (плоские сектора)

Выбор владельца: **плоская модель, без слоя регионов.**

- **Сектор** — единственная атомарная единица карты. У него:
  - **точка захвата** — `owner` (`null` = нейтральный/незахваченный); меняется
    через бой (`combat`) или заход на необоронённый сектор;
  - **пути** — `links`, **настраиваемый элемент**, **только к соседним**
    секторам (симметричные рёбра графа; флоты ходят по ним);
  - **вид** (`kind`): `planet` / `asteroid` / `nebula` / `empty` — задаёт
    `capturable` / `buildable` / `orbit` и масштаб;
  - **террейн** (модификаторы скорости/живучести) и, если это планета, **тип
    планеты** (`planetType`, бонусы добычи/обороны);
  - гарнизон, застройка, ресурсы, трейты.
- **Планета — это сектор, просто меньше**: подтип сектора (`kind: planet`),
  захватываемый и застраиваемый, с `planetType`. `empty` — необоронённый
  путевой узел (не захватывается), `asteroid`/`nebula` — промежуточные виды.
- **Регионов (групп секторов) нет.** Это сознательно отложено — см. §5.

### Коллизия термина «сектор» (разводим в M0)

Сейчас слово занято трижды и это противоречиво:

| Где | Значение сейчас | Куда по модели |
|---|---|---|
| `gdd.md` | **регион** из планет + бонус «контроль целиком» | → отложено (нет регионов), см. §5 |
| `shared-core` `Planet.sectorType` | **террейн-тег** узла (`empty_space`/…) | → переименовать в `terrain` |
| `Planet` (тип) | узел карты | → **это и есть «сектор»** |
| прототип `SECTOR_TYPES` | вид узла (planet/asteroid/…) | → `kind` сектора в данных (M2.1) |

Итог термина: **Сектор = `Planet`-узел**; «вид» = `kind`; «террейн» = бывший
`sectorType`. Слово «сектор» больше не значит «регион».

«сектор» как узел/провинцию (правок не потребовал). Ссылка на этот роадмап в `state.md`.
**Готово, когда:** ✅ в доках нет трактовки «сектор = регион».

### M0.2 · Развести `sectorType` → `terrain` `[core]` 🔒(M0.1) — ✅
**Цель:** убрать перегрузку слова в коде. **Сделано:** поле `Planet.sectorType`
(и `PlanetSnapshot.sectorType`) → `terrain` во всех зонах (ядро, модули, fog,

применяет террейн-бонусы сектора; полное переименование реестра `data.sectors →
data.terrains` — опциональный косметический follow-up, не блокирует M1.

## Фаза M1 · Карта как контент (data-driven)

> Сейчас карта собирается двумя несовместимыми путями: прототип — процедурно
> (`withNeighborLinks`, RNG-граф), сервер — хардкодом в `scenario.ts`. Цель —
> **одна карта = данные**, как остальной контент (`data/*.json` + zod).

### M1.1 · Схема карты `data/maps/*.json` `[data]` 🔒(M0.2) — ✅
**Цель:** карта — валидируемый контент. **Сделано:** `shared-core/data/mapSchema.ts`
(`MatchMapSchema` + `parseMatchMap`/`safeParseMatchMap`): сектор `{ position, kind,
terrain?, planetType?, owner?, buildings?, garrison? }` + рёбра `paths` (пары
соседей) + `players` + `fleets`. Пример `data/maps/skirmish-1.json`. Тесты
`mapSchema.test.ts` (парс + дефолты + отказ кривых). `kind` пока несётся данными,
проецируется в ядро в M2.1.
**Готово, когда:** ✅ карта парсится zod-схемой, кривая отвергается.

### M1.2 · Загрузчик карты → `GameState` `[core]` 🔒(M1.1) — 🚧
**Цель:** из карты-данных строить начальное состояние. **Сделано:**
`buildStateFromMap(map, data, opts)` (`shared-core/state/buildFromMap.ts`):
строит planets из секторов, **выводит `links` из undirected `paths`** (sorted,
симметрично), players, fleets; чистый/детерминированный; fail-secure (бросает
`E_INVALID_MAP`). Тесты `buildFromMap.test.ts` (стейт, links, hp, детерминизм).
**Осталось:** подключить dev-сервер и прототип к загрузке карты-файла (прототип
зависит от M4.1 — рендер из стейта, а не из `MAP`-константы).
**Готово, когда:** прототип и сервер стартуют из одной карты-файла.

### M1.3 · Валидация путей: только к соседям `[core]` 🔒(M1.1) — ✅
**Цель:** инвариант «пути только к соседним секторам». **Сделано:**
`validateMatchMap` — **neighbour-only** по relative-neighbourhood (ребро `a–b`
валидно, только если нет третьего сектора «между» — ближе к обоим); плюс
симметрия/без self-loop/без дублей, связность графа (BFS), существование
владельцев/целей флотов и (с данными) террейна/типа/юнитов/построек. Тесты:
дальнее ребро → `E_PATH_NOT_NEIGHBOR`, изолят → `E_MAP_DISCONNECTED`, и т.д.
**Готово, когда:** ✅ карта с «дальним»/односторонним путём не проходит.

## Фаза M2 · Сектор как сущность (механика)

### M2.1 · Виды секторов (`kind`) в данные `[core]` `[data]` 🔒(M0.2) — M
**Цель:** перенести прототипный `SECTOR_TYPES` (planet/asteroid/nebula/empty +
`capturable`/`buildable`/`orbit`) в контент+ядро, чтобы виды были data-driven.
**Подзадачи:** `data/sectorKinds.json`; ядро читает флаги через хук/возможность
(нет вида → мягкий дефолт); рендер берёт `kind` из сектора.
**Зависит от:** M0.2. **Готово, когда:** новый вид сектора добавляется данными,
без правок логики.

### M2.2 · «Захват заходом» как правило ядра `[core]` 🔒(M2.1) — M
**Цель:** необоронённый захватываемый сектор берётся приходом флота — сейчас это
**клиентский хак** прототипа (`seizeSector`), в MP его нет. Перенести в
kernel-модуль (реагирует на `fleet.arrived`/`transit`), чтобы работал и в сети.
**Подзадачи:** модуль `captureOnArrival` (guard: capturable, без гарнизона, не
оспаривается) → `planet.captured`; убрать клиентский `seizeSector`; тесты.
**Зависит от:** M2.1. **Готово, когда:** в MP заход на нейтральный
необоронённый сектор меняет владельца на сервере; одиночка и сеть едины.

### M2.3 · Масштаб сектора (планета = меньше) `[core]` `[cli]` 🔒(M2.1) — M
**Цель:** размер как свойство сектора (планета меньше «больших» секторов).
**Подзадачи:** `size`/`scale` в данных сектора; рендер масштабирует ноду;
(опц.) влияет на вместимость застройки/гарнизона через хук.
**Зависит от:** M2.1. **Готово, когда:** планеты визуально и (опц.) ёмкостно
меньше крупных секторов, всё из данных.

## Фаза M3 · Настраиваемость (редактор путей/карты)

### M3.1 · Редактор карты `[tools]` 🔒(M1.2) — L
**Цель:** «пути — настраиваемый элемент» по-настоящему: ставить сектора, тянуть
пути (только к соседям), задавать владельца/вид/тип; экспорт в `data/maps/*.json`.
**Подзадачи:** режим редактора в прототипе (или отдельная страница); snap к
соседям; запрет дальних рёбер; импорт/экспорт JSON.
**Зависит от:** M1.2. **Готово, когда:** карту можно собрать мышью и сохранить
в валидный файл, который грузит M1.2.

**Цель:** нынешний RNG-граф остаётся как генератор-пресет, но **пишет в тот же
формат** (а не в рантайм). Детерминированно, вне ядра.
**Зависит от:** M1.1. **Готово, когда:** генератор выдаёт `data/maps/*.json`,
который проходит M1.3.

## Критический путь

`M0.1 → M0.2 → M1.1 → M1.2` (одна карта-данные грузит и прототип, и сервер) —
это MVP «карта как контент». Дальше параллельно: `M1.3` (инвариант соседей),
`M2.*` (виды/захват/масштаб), затем `M3.*` (редактор). `M4.1` подхватывает по
мере готовности данных.
