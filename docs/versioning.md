> Как игра обновляется, **не ломая живые матчи**. Сводит воедино уже заложенный
> per-match version-pinning (код/`architecture.md`) и **стратегию миграции** (🔴-пробел

## 1. Принцип (уже архитектура, не новизна)

`architecture.md:210`: *«Версия правил привязана к матчу… идущие матчи доигрываются на
старых правилах, новые подхватывают новые. Часть гибкости и целостности (A08)»*. Код:
`GameVersion { data, manifest }` — **«frozen per match»** (`gameState.ts`). Этот док
закрывает не базовую модель (она есть), а **стратегию**: миграции, мульти-версия, деплой.

## 2. Оси версий (что пиннится в матч)

- **`data`** — контент `data/*.json` (версия из `manifest.json`).
- **`manifest`** — набор и **порядок** модулей кернела (версионируется per-match).
- **core-algo** — семантика ядра и RNG (`sfc32`, golden-локнут). Поля нет, но это
  **тоже ось версии**: смена алгоритма меняет результат при том же `(data, manifest)`.

Всё это фиксируется в `GameState.version` при старте матча и далее **не меняется**.

## 3. Граница core ↔ meta (где живёт детерминизм)

- **Бегущий матч заморожен.** Ничто извне не дотягивается внутрь — иначе рушатся
  детерминизм и реплеи (golden-тест).
- **Мета** (аккаунт/корпа/экономика/уровень/медали) — на сервере, **вне ядра**;
  обновляется **глобально**.
- **Мост — снапшот при старте** (GDD §5.2): матч берёт всё нужное от меты **один раз**,
  иммутабельно. Поэтому мета-апдейт меняет **окружение игрока**, а **не правила идущего
  матча** — даже если игрок прямо сейчас в старом матче.
- **Пост-матч (решение, = твой #3):** результат старого матча конвертируется мета-слоем
  по **текущей** мета-версии на момент финализации (XP, очки влияния, награды, условия
  медалей живут в мете, observation-only) — **не** по пиннутой версии матча.
- **Совместимость ingest'а:** мета обязана уметь читать **старый формат результата**
  матча версии N, завершившегося уже после апдейта.

## 4. Таксономия изменений (главный практический фильтр)

| Класс | Пример | Старые матчи | Как делать |
|---|---|---|---|
| **Аддитив** | новый юнит/здание/модуль **данными** | не трогает | новая `data`-версия; безопасно |
| **Изменение значений** | баланс существующего юнита | **пиннятся** на старом | новая `data`-версия (уже работает) |
| **Ядро / RNG / порядок модулей** | алгоритм боя, `sfc32` | **ЛОМАЕТ реплеи** существующих | редко; через **новый модуль/манифест**, не правку; golden обновлять осознанно |
| **Мета-схема** | новое поле аккаунта/корпы | применяется **глобально** | additions-only + zod-версия + ленивая N→N+1 |

**Правило по умолчанию:** предпочитать **аддитив** (новый модуль/данные) правке
существующего — совпадает с расширяемостью модульной системы (`modulesystem.md`).

## 5. Миграция персиста (стратегия S1.4)

- **Только-добавление полей** + **версионирование zod-схем** + **ленивая** миграция
  при чтении (мигратор N→N+1), **без** массового ETL на деплое; round-trip-тест.
- Применяется к **сейв-формату матча** (JSONB `GameState`) **и** к **мета-схемам**
  (аккаунты/корпы/экономика).
- JSONB: добавление поля **не требует** миграции БД (`architecture.md:193`).

## 6. Окно мульти-версионности и выбытие

- Матчи живут **недели-месяцы** (timeScale ×1 → до 100 дней, GDD §3.1). Значит сервер
  держит **старые кернелы + `data`-версии загружаемыми**, пока на них есть живые матчи
  → **несколько версий live одновременно**. Версию нельзя ретайрить, пока на ней идёт
  хоть один матч.
- **Выбытие (решение):** «срок поддержки-загрузки версии» = **максимальная длина матча**
  (потолок возраста, привязанный к timeScale) + буфер на реплеи. Очень старые матчи не
  зависают: триггер **кризиса по времени** (GDD §3.2) форс-финалит их на отметке
  `timeScale → дни`. После выбытия всех матчей версии её код/данные можно ретайрить.
- **Реплеи** завершённых матчей: либо самодостаточный **event-log** (PE-1.1), либо
  event-log + старая версия кода — см. §9.

## 7. Деплой без потери матчей (OPS-1.1)

- **rolling / blue-green**; **drain + graceful shutdown** (дренить WS); клиент
  **reconnect + resume** (SV-2.2); **expand/contract** миграции БД без даунтайма; пин
  образов по digest (SE-5).
- Клиент (`cross-platform-roadmap.md`): UX «доступна новая версия» с управляемым
  обновлением; умеет говорить с сервером, где **живёт несколько версий матчей**;
  инвалидация кэша по `matchId`/`data`-версии.

## 8. Версия клиента ↔ сервера (force-update handshake)

- **На коннекте** (после JWT-аутентификации, `accounts-roadmap.md` `AC-1.3`) сервер
  отдаёт `minSupportedClientVersion` и `currentClientVersion`. Клиент **ниже min** →
  жёсткий экран «обнови, чтобы играть»; **ниже current, но ≥ min** → мягкое «доступно
  обновление» (матч не прерывает — на следующем запуске / безопасной границе).
- **Это ось клиент↔сервер ПРОТОКОЛА, не правил матча.** Новый клиент **доигрывает
  старые матчи**: клиент — тонкий рендер серверного состояния, а не ре-симуляция, так
  что версия данных матча ему не нужна. Форс-апдейт **не теряет матч** — клиент
  перезагружает новый билд → reconnect + resume (SV-2.2) → продолжает тот же матч
  (состояние авторитетно на сервере).
- **Безопасность (server-authority, OWASP A06/A08):** самозаявленная версия — лишь
  **UX-сигнал**. Реальный backstop — **серверная валидация каждого сообщения** по
  текущему протоколу: несовместимый старый клиент получает fail-secure отказ. Соврать
  про версию ради обхода гейта бесполезно. Бамп `minSupportedClientVersion` — только на
  **ломающих** изменениях протокола (редко; совместимые → мягкое обновление).
- **Анти-soft-lock (SW):** при форс-гейте клиент **активно гонит обновление Service
  Worker** (`skipWaiting` + reload), а не просто показывает «иди обнови»; манифест версии
  и app-shell — **network-first / не долгий кэш**, иначе застрянет на старом билде,
  который сервер уже не принимает. Сокет-данные SW не кэширует (CP2.2).

## 10. Открытые вопросы

```
□ Реплей завершённых матчей: самодостаточный event-log (не нужна старая версия кода)
  vs event-log + пиннутая версия — определяет, как долго хранить старые кернелы.
□ Нужен ли явный мигратор N→N+1 для САМОГО ядра, или ядро меняем только аддитивно
  (новый модуль/манифест), а правки существующих модулей запрещены без новой версии?
□ [РЕШЕНО, §8] Хендшейк клиент↔сервер: force по `minSupportedClientVersion` (жёстко) /
  soft «доступно обновление»; ось протокола ≠ версия матча; backstop — валидация сообщений.
  Осталось: частота бампа min и дисплей-данные для контента, удалённого в новой версии.
□ Мета-схема: единый сквозной версионный номер vs per-entity версии.
□ Числа: буфер хранения версии после max-match-age; верхняя граница числа live-версий.
```
