# Корпорации — членство, роли, медали · дизайн

> Корпорация как **постоянная организация** — социальный/орг-слой альянсов из
> `metagame.md`. Слой — **server/meta**, вне детерминированного ядра. Парные доки:
> `metagame.md` (корпорации/очки влияния), `corporation-wars.md` (AvA-битва, офицеры
> курируют ростер), `main-menu.md` (карточка игрока §4.2, рейтинг корпораций §4.6),
> `accounts-roadmap.md` (аккаунт/identity), `metrics-roadmap.md` (история матчей →
> условия медалей).

---

## 1. Где это живёт и инвариант

- Членство, роли, медали — **состояние между матчами на сервере**; детерминированное
  ядро (`shared-core`) о них **не знает**.
- **Условия медалей считаются на сервере** из истории матчей (`MM-3.1`)/агрегатов
  метрик — **никогда в ядре** (детерминизм: никаких медальных счётчиков в `applyAction`).
- **Server-authority + авторизация (OWASP A01)** — главный инвариант: каждое
  чувствительное действие (роль/кик/приём/вручение) проверяется по роли на сервере;
  любой интент **fail-secure** (отказ со стабильным кодом). Чувствительные действия
  пишутся в **аудит-лог**; на соц-действия — rate-limit (анти-абуз).

## 2. Членство и роли (фиксированный RBAC)

**Роли** (жёсткий набор, без конструктора прав):

- **Глава** — владелец корпорации (ровно один).
- **Офицер** — помощник: приём/кик рядовых, курирование AvA-ростера, вручение медалей.
- **Член** — рядовой участник.
- **Заявка/рекрут** — кандидат, ещё не принят.

**Жизненный цикл:** поиск/инвайт → **заявка** → **приём** (глава/офицер) → Член →
(повышение/понижение главой) → кик / выход / передача главенства.

**Матрица прав:**

| Действие | Глава | Офицер | Член |
|---|:--:|:--:|:--:|
| Принять/отклонить заявку | ✓ | ✓ | ✗ |
| Кикнуть | любого, кроме себя | только рядовых/рекрутов | ✗ |
| Назначать роли (повышать/понижать) | ✓ | ✗ | ✗ |
| Вручать корп-медали | ✓ | ✓ | ✗ |
| Корп-флаг готовности к AvA | ✓ | ✗ | ✗ |
| Вызов на AvA (тратит очки влияния) | ✓ | ✗ | ✗ |
| Курировать AvA-ростер (`setRoster`) | ✓ | ✓ | ✗ |
| Передать главенство / распустить | ✓ | ✗ | ✗ |
| Покинуть корпорацию | ✓ (передав главенство) | ✓ | ✓ |

**Правила авторизации (явно):** один Глава; Главу нельзя кикнуть; офицер **не может**
кикнуть/понизить Главу или другого офицера и **не может** менять роли (эскалация
закрыта). Передача главенства — отдельное явное действие Главы.

## 3. Медали и достижения (фиксированный каталог + гибрид-выдача)

**Каталог — это данные** (напр. `data/medals.json`), не код и не произвольные критерии:

```jsonc
"ava_veteran_10": {
  "name": "Ветеран AvA",
  "scope": "corp",            // corp = привязана к корпорации; account = глобальная
  "condition": { "type": "corp_matches", "count": 10 },
  "grant": "manual"           // manual = вручает глава/офицер; auto = система сама
}
```

- **Условия объективны и проверяются сервером** из истории матчей/метрик. Пример из
  ТЗ: «сыграть 10 матчей AvA за **эту** корпорацию» → `corp_matches ≥ 10` (сервер знает `corp_wins`/`corp_matches`).
  Клиент условие не самозаявляет (нельзя подделать).
- **Гибрид-выдача:**
  - **Достижения** (`scope:account`, `grant:auto`) — система **авто-выдаёт** при
    выполнении (без участия главы).
  - **Корп-медали** (`scope:corp`, `grant:manual`) — сервер помечает игрока
    **eligible** при выполнении условия; **глава/офицер вручает** (акт признания);
    сервер **перепроверяет право в момент вручения** → не eligible ⇒ `E_NOT_ELIGIBLE`.
- **Перманентность:** заработанное — **постоянная история аккаунта** (даже после
  выхода из корпы; помечается «заработано с корпорацией X»). В MVP медали **не
  отзываются** (исключает абуз/драму). Вручение **идемпотентно** (нет дублей).

## 4. Витрина в карточке игрока

- Заработанные медали/достижения живут в **карточке игрока** (`main-menu.md` §4.2).
- Игрок **закрепляет несколько на витрину** (showcase); остальные — в полном списке.
- Видимость — по настройкам приватности (`main-menu.md` §5.2): витрина публичная,
  полный список — по выбору игрока.

## 5. Интенты (клиент → сервер)

```
createCorp(name, …)                                (игрок: уровень ≥ порога → создатель = Глава)
applyToCorp(corpId) / cancelApplication            (игрок)
acceptApplicant(playerId) / declineApplicant(...)  (глава/офицер)
kickMember(playerId)                               (глава: любой; офицер: рядовой/рекрут)
setMemberRole(playerId, role)                      (глава)
transferHeadship(playerId)                         (глава)
leaveCorp()                                         (любой; глава — после передачи)
grantMedal(playerId, medalId)                      (глава/офицер; сервер гейтит по eligible)
pinShowcase(medalIds[])                            (игрок, своя карточка)
```

Авто-достижения интента не требуют (выдаются сервером по факту).

## 6. Авторизация и аудит (A01)

- Проверка роли **на каждое** чувствительное действие, server-side, fail-secure.
- Инвариант одного Главы; «Главу не кикнуть»; офицер не эскалирует права.
- **Создание корпы** — server-side, личность только из валидной сессии/JWT
  (`CorpService.create`); **гейт по уровню аккаунта запланирован** (`account-level.md`
  `AC-0.3`; `E_LEVEL_TOO_LOW` в коде пока нет), rate-limit + аудит.
- **Аудит-лог:** смены ролей, кики, вручения медалей, передача главенства.
- **Анти-абуз:** rate-limit на заявки/инвайты/вручения; вручение идемпотентно;
  `grantMedal` отвергает не-eligible (нельзя «подарить» незаслуженную медаль).

## 7. Открытые вопросы

```
□ Глава ушёл без передачи главенства → авто-наследование (старший офицер) vs заморозка корпы.
□ Лимиты: число офицеров, размер корпорации, число заявок.
□ Отзыв медали — в MVP нет; нужен ли вообще (модерация фрода)?
□ Стартовое наполнение каталога достижений/медалей (какие условия) — задача данными.
□ Витрина: сколько слотов на showcase; дефолт видимости полного списка.
□ Связь медалей с экономикой (медаль/достижение даёт немного очков влияния корпе?) — кандидат.
□ Тип условий: какие `condition.type` поддерживаем на старте (матчи AvA/обычные,
  победы, очки, длительность членства) — растёт данными, не кодом.
```
