# Технологический стек — решение и обоснование (ADR)

> Какой стек выбран, **почему именно он**, и по каким **измеримым триггерам** его стоит
> пересматривать. Спутник `architecture.md` (§3) и `tech-research.md` (анализ опций).
> Цель — чтобы выбор не пере-обсуждался: здесь зафиксированы причины и условия смены.

## Принцип, который решает 80% выбора

**Одна и та же детерминированная симуляция обязана крутиться и на сервере (авторитет), и
в браузере (превью «если атакую — что будет», сверяемое `hashState`).** Это
«same-code-both-sides». Он один отсекает почти всё: язык ядра должен исполняться в браузере
**без границы перевода** — значит JS/TS или что-то→WASM. Всё остальное в стеке — следствие
этого плюс жанра (real-time асинхронное MMO, 24/7, server-authority, детерминизм).

## Стек по слоям

| Слой | Выбор | Почему | Триггер пересмотра |
|------|-------|--------|--------------------|
| **Sim-ядро** | **TypeScript** (`shared-core`) | Нулевая поверхность перевода сервер↔браузер; data-driven контент на JSON+zod — родная эргономика; нагрузка логическая, не вычислительная (мир идёт по формуле, не по кадрам). Детерминизм держится в бит-точном подмножестве IEEE-754 (`Math.sqrt` бит-точен **по спеке**, трансцендентные забанены ESLint-ом) | Rust→WASM только при **И** заморозке схемы **И** *измеренном* сбое (воспроизведённый desync на подмножестве — крайне маловероятно — **или** упёртый профилем CPU редьюсера) |
| **Слой действий** | **TS** (`action-layer`, вне ядра) | Валидация/авторизация/идемпотентность в одном узле; ядро остаётся чистым | — |
| **Серверный рантайм** | **Node 20+ / TS** | Тот же язык и тот же `shared-core`, что у клиента; нагрузка I/O-bound и событийная (`advanceTo` спит между событиями); скорость команды | Go/Elixir-**шлюз** только когда кластер Node — *измеренное* узкое место по числу stateful-соединений |
| **Транспорт** | **сырой `ws`** | Минимум, ты владеешь протоколом (`protocol.ts`), `noServer`-режим даёт контроль над upgrade/auth. socket.io/Fastify-ws — лишний вес и магия | Тонкий **Fastify** добавить только если понадобится REST-API рядом; `ws` оставить |
| **Персистентность** | **Postgres + JSONB** | `GameState` уже JSON → в JSONB **без маршалинга**; ACID + транзакции + оптимистичные блокировки (anti-double-spend); один датастор. Mongo не даёт выигрыша над JSONB и теряет транзакции | — |
| **Планировщик событий** | **pg-boss на Postgres** | Истина расписания и так в состоянии (`scheduled[]`); pg-boss = «разбудить сервер к событию» в **той же** БД и транзакции, что и запись состояния. Никакого второго инфра-зверя | Redis+BullMQ только если очередь станет горячее, чем тянет Postgres (десятки тысяч джоб/с) |
| **Кэш / pub-sub** | **пока ничего** → Postgres `LISTEN/NOTIFY` | До нескольких процессов кэш не нужен (состояние горячее в памяти комнаты) | Redis — при >1 серверного процесса (presence/locks/cross-node fan-out) |
| **Клиент** | **PWA-first** (TS + сборщик), карта **Canvas2D→WebGL** | Одна TS-кодовая база, **переиспользует `shared-core`** для мгновенного превью; асинхронному жанру не нужна нативная input-латентность. RN/Flutter = второй рантайм и потеря общего ядра | WebGL/PixiJS — когда на экране *сотни* движущихся сущностей (сейчас десятки блипов) |
| **Мобильная обёртка** | **Capacitor** | Тонкий нативный шелл вокруг PWA → push + офлайн + APK, без переписывания | Нативный Kotlin/Swift — практически никогда, кроме тонкого шелла при жёсткой блокировке push/фон |
| **Аналитика / античит-ML** | **Python** (офлайн) | Воронки, когорты, anomaly-detection по отклонённым действиям — вне горячего пути, ядра не касается. *Первый* слой античита — **replay на TS-ядре** (Python не нужен) | Подключать пост-лонч, когда накопятся матч-данные |
| **Инфра / хостинг** | **Docker + Fly.io/Railway** (managed PG) + **Cloudflare** | Дёшево на старте, managed Postgres + приватная сеть; Cloudflare = DDoS + скрытие IP + TLS | Своя оркестрация/k8s — только на мульти-региональном масштабе |
| **Auth** | **внешний OIDC** (Keycloak/Zitadel self-host или Clerk/Auth0) | Не писать свою крипту/хранение паролей; JWT в ws-handshake (`accounts-roadmap`) | — |
| **CI** | **GitLab** (`.gitlab-ci.yml`) | Полный DevSecOps уже там (Semgrep/Bearer/Trivy/Gitleaks/SBOM). На GitHub — только сборка APK | Решить: зеркальный GitHub Actions для гейта, или явно принять «истина CI — GitLab» |

## Уже выбрано в коде (на момент написания)

Стек **сошёлся** на лёгком варианте, и это правильно:

- Ядро: `@void/shared-core` (TS), детерминированное, ESLint-гард детерминизма.
- Action-layer: `@void/action-layer` (TS).
- Сервер: Node + TS, сырой **`ws` 8.21**, **`pg` 8.22** (стор `store/postgres.ts`).
  **Нет** Redis, BullMQ, Fastify — от исходного плана отказались осознанно.
- Клиент: PWA-first (TS), прототип на Canvas2D, Capacitor-обёртка (APK).
- CI: GitLab.
- **Redis-шов уже заложен:** эфемерные данные матча (пинги) живут за интерфейсом
  `EphemeralStore` (`packages/server/src/ephemeral.ts`) с in-memory-имплом. Это и есть
  «заложить Redis заранее» без преждевременного зверя: на мульти-процессе Redis-импл
  вставляется **одной заменой файла**, логика комнаты не меняется (триггер №5 ниже).

> Историческая сноска: `architecture.md §3` и старый `roadmap.md` упоминают Fastify +
> Redis + BullMQ. Это **исходный** набросок; персистентность пере-зафиксирована на
> **pg-boss/Postgres** (`persistence-roadmap.md`), а транспорт — на сырой `ws`. Этот
> документ отражает актуальное решение.

## Чего НЕ добавлять (и почему)

- **Redis/BullMQ сейчас** — для одного процесса дублируют Postgres + in-memory комнату; меньше зверей = меньше точек отказа.
- **Переписывать ядро на Rust/Go/C#** — сменишь удержанный, измеримый, чинимый-в-TS риск на неудержанный на golden-границе нового языка и потеряешь общий код с клиентом. Rust→WASM — только «план Б» под триггер.
- **ORM поверх JSONB** — воюет с JSONB, тащит неявную не-детерминированность; хватит тонкого слоя на `pg`.
- **GraphQL / микросервисы / k8s** — решения для проблем масштаба, которых ещё нет.
- **React Native / Flutter** — второй рантайм и потеря переиспользования `shared-core` на клиенте.

## Триггеры смены (консолидировано)

Полиглот по краям — **только** по измеримому условию, не «на всякий случай»:

1. **Rust→WASM-ядро** — заморожена схема **И** воспроизведён desync на бит-точном подмножестве, либо профиль упёрся в CPU редьюсера.
2. **Elixir/Go-шлюз** — измеренный потолок stateful-соединений у кластера Node (только край транспорта, **не** sim).
3. **WebGL-рендер** — сотни движущихся сущностей в кадре.
4. **Python-аналитика/ML** — накоплены матч-данные пост-лонч.
5. **Redis** — появился второй серверный процесс (cross-node pub-sub / presence / locks).

## Идеал одной фразой

**TypeScript-моностек на общем `shared-core`, тонкий `ws`-сервер на Node, Postgres+JSONB как
единственный датастор (он же — через pg-boss — планировщик), PWA-клиент с Capacitor-обёрткой,
Python только для офлайн-аналитики.** Минимум движущихся частей, один язык от карты до
авторитета, одна БД. Языки по краям добавляются только по названным триггерам — до тех пор
новый язык = налог без выгоды.
