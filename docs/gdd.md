# Игровой дизайн-документ (GDD)

> Браузерная и мобильная сессионная 4X grand strategy в сеттинге тёмного космоса.
> Концептуально вдохновлена Bytro Labs (Iron Order, Call of War, Conflict of Nations) — жанр и темп, не код/арт/тексты.

## 1. Концепция

4X-стратегия (eXplore, eXpand, eXploit, eXterminate) с сессионной структурой на 10 игроков. Сеттинг — тёмный космос: чужие цивилизации, выживание, атмосфера ужаса. Расширяет формулу Bytro модульностью кораблей, героями с уникальными способностями (например, червоточины), расовыми фишками и двойной петлёй прогрессии (внутрисессионная + межсессионная).

### Ключевые параметры

| Параметр         | Значение                                                            |
| ---------------- | ------------------------------------------------------------------- |
| Игроков в сессии | до 10                                                               |
| Длительность     | определяется скоростью времени (раздел 5) — до 30–100 реальных дней |
| Платформа        | браузер + мобайл (один TS-движок, см. `architecture.md`)            |
| Карта            | граф **секторов** (узел = точка захвата + пути к соседям); планета — сектор поменьше. Модель: `map-roadmap.md` |
| Старт            | край галактики, позиция — случайно или из списка                    |
| Темп             | асинхронный, check-in раз в несколько часов                         |

## 2. Сессия: консервация и изоляция

**Жёсткая консервация при старте.** Выбор расы, стартовой точки и деки модулей фиксируется при создании сессии и не может быть изменён. Попытка смены в процессе — действие, которое сервер обязан отклонять (fail-secure, см. `architecture.md` раздел 5/10).

Назначение: предотвращает накопление необратимого преимущества (нельзя перестроиться под мету на лету), упрощает баланс, убирает класс читерства "пересборка под обстоятельства".

**Две изолированные экономики:**

```
Сессионный контур
  Ресурсы внутри цикла (см. раздел 4)
  Полностью сбрасывается вместе с сессией

Глобальный контур (мета-рынок)
  Донат и мета-предметы между сессиями
  НЕ влияет на производство юнитов или опыт героев напрямую
  (см. раздел 4.3 про объединённый ресурс — уточнение ниже)
```

## 3. Скорость времени и завершение сессии

### 3.1 Скорость времени — единая ось масштабирования

Вместо нескольких независимых "режимов длительности" — один глобальный множитель `timeScale`, применяемый ко всем таймерам матча разом (движение флотов, стройка, восстановление, частота событий). Хранится в манифесте матча (`modulesystem.md`).

| timeScale | Реальных дней до кризиса (максимум) |
| --------- | ----------------------------------- |
| ×1        | 100                                 |
| ×2        | 60                                  |
| ×4        | 30                                  |

С высокой вероятностью реальные сессии завершаются раньше максимума — по очкам или по неактивности (см. ниже).

### 3.2 Три независимых триггера завершения (гонка)

```
1. ПОБЕДА ПО ОЧКАМ
   Игрок: 500 очков → немедленная победа
   Коалиция: порог ниже одиночного на игрока (стимул к союзам, формула ниже)

2. КРИЗИС ПО ВРЕМЕНИ (верхняя граница, не гарантированная длина)
   Наступает на отметке timeScale → реальные дни (таблица выше)
   Принудительный финал, ранжирование по очкам на этот момент

3. ЗАВЕРШЕНИЕ ПО НЕАКТИВНОСТИ
   Все игроки неактивны → короткий таймер → авто-финал
   Защита от мёртвых сессий
```

Любой триггер запускает финальный снапшот очков и распределение мест.

### 3.3 Порог победы для коалиций

Коалиция должна быть явно выгоднее одиночной игры. Порог на коалицию растёт **медленнее линейного** от числа участников:

```
Черновая формула (тюнится как балансные данные, не код):
  порог_коалиции = 500 × N_участников × 0.7

  2 игрока: 700 (350/игрока — выгоднее чем 500 соло)
  3 игрока: 1050 (350/игрока)
  4 игрока: 1400 (350/игрока)
```

Коалиция = формальный игровой союз с явным статусом (часть модуля дипломатии), не неформальное "не воюем друг с другом" — нужен недвусмысленный критерий "чьи очки суммируются".

### 3.4 Награды по итогам сессии

При финализации (любым триггером) после вычисления таблицы по очкам — распределение наград, масштабированное по месту:

- Опыт аккаунта
- Ресурсы для мета-прокачки
- Небольшое количество донатной валюты

Открыт вопрос баланса (не блокирует архитектуру): плавная функция от места или пороговые "лиги" (топ-3/середина/низ).

## 4. Ресурсы и экономика

### 4.1 Базовый список (динамический, см. `architecture.md` 2)

```
"resources": ["credits", "metal", "biomass", "dark_matter", "artifacts", "premium_shard"]
```

Новый ресурс — новая запись в списке, экономика подхватывает автоматически.

### 4.2 Сессионный рынок

Ресурсы внутри цикла, торговля и производство только в пределах сессии. Сбрасывается вместе с ней.

### 4.3 Премиум-ресурс — объединённая модель

Единый ресурс (`premium_shard`), а не два разных:

```
Источник 1 — добыча: редкие планеты на спорных секторах карты,
  малые количества, медленнее чем покупка, провоцирует военный конфликт
Источник 2 — покупка: глобальный мета-рынок, донат

Зона применения (ОБЯЗАТЕЛЬНОЕ правило, чтобы не нарушать честность):
  ✓ Ускорение исследований (с убывающей эффективностью при доната)
  ✓ Удобство/декоративное
  ✗ НЕ производство юнитов напрямую
  ✗ НЕ боевая мощь
  ✗ НЕ опыт героев

Платящий игрок ускоряется, но не получает эксклюзивную силу,
недоступную тому, кто отвоевал ресурс в бою.
```

Планеты с этим ресурсом — горячие точки конфликта; разумно давать им бонус обороны по умолчанию, чтобы захват требовал военной операции, а не разового рейда в первый день.

### 4.4 Монетизация (пересмотрено — деньги покупают силу, с гайками)

> **Решение развёрнуто осознанно.** Изначально (см. _superseded_ ниже) донат **не мог** купить силу — ради честности камерной сессии. Теперь философия иная: сила (модули кораблей, скиллы героев) **покупается** через мета-экономику — аукцион за Варранты, лутбоксы за Суверены, плюс RNG-дроп в конце матча и крафт. Модель, валюты, комплаенс и анти-RMT — `economy-roadmap.md`.

Гайки, удерживающие модель от «кошелёк всегда побеждает»: **раскрытие шансов** лутбоксов + **pity**-гарант; **кэтч-ап для F2P** — сила также падает с матчей (шанс растёт от места), так что бесплатный догоняет игрой; **soulbound** на донат/лутбокс-предметы (анти-RMT; заработанное/скрафченное — торгуется); лутбокс-комплаенс (раскрытие/возрастной гейт/юрисдикции, `economy-roadmap.md` EC-0.4). Доступ к аукциону и создание корпорации гейтятся уровнем аккаунта (`account-level.md`).

_Superseded:_ ~~донат не может купить боевую мощь, опыт героев или производство; критично для честности камерной сессии на 10~~ — заменено осознанным P2W-разворотом выше.

## 5. Прогрессия — двойная петля

### 5.1 Сессионная прогрессия

Внутри одного матча: экономика, застройка, технологии сессии — сбрасывается при завершении.

### 5.2 Межсессионная прогрессия

Реализована через меню между сессиями. Применяется при создании следующей сессии. Превращает неудачи в долгосрочные инвестиции, сохраняет мотивацию.

**Упор на горизонтальную прогрессию** (новые опции и способности), а не вертикальную (голые числовые бонусы к статам). Сознательный уход от ловушки Bytro ("юниты с новыми цифрами под новым именем").

**Уровень аккаунта.** «Опыт аккаунта» (§3.4) копится в **уровень** — монотонную функцию суммарного XP. Уровень **гейтит доступ** к мета-функциям (канонически — создание корпорации, `corporations.md`) и пейсит доступность **только горизонтального** подмножества этой прогрессии (тактики/модули-опции/трейты, стат-нейтральные слоты профилей). Числовую/вертикальную ветвь (модификаторы `computeDamage`/`computeHP`, параметры героев, асимметричные стартовые профили) уровень **не пейсит никогда** — она добывается по премиум-неускоряемой оси, иначе ускорение исследований за донат (§4.3/§4.4) дало бы раннюю силу. Уровень не даёт боевой силы и не является балансным входом; живёт на сервере вне ядра. Детали — `account-level.md`.

```
Прокачивается между сессиями:
  Параметры героев (адмиралы, губернаторы, генералы десанта)
  Открытие новых модулей кораблей
  Открытие новых технологий
  Открытие новых тактик боя (раздел 7.4)

Смесь горизонтали и вертикали:
  Числовые модификаторы → через хук-конвейер (computeDamage, computeHP и т.д.)
  Трейты, открывающие новое поведение → через систему событий
  (та же шина из modulesystem.md, источник теперь — профиль аккаунта)
```

**Архитектурное решение, которое предстоит явно зафиксировать в `core-engine.md`:** матч читает профиль аккаунта **снапшотом при старте** (не динамически) — упрощает детерминизм, матч не меняется, если игрок прокачался в середине сессии.

## 6. Корабли, модули, герои

### 6.1 Модульность кораблей

Игровая фича поверх data-driven юнитов (`architecture.md` раздел 2). Корабль собирается из модулей — формализация существующей гибкости данных в механику для игрока, не только для разработчика.

### 6.2 Герои — три типа, разные зоны влияния

```
Адмирал       → бонусы флоту в космическом бою (HP, урон, и т.д.)
Губернатор    → бонусы планете (экономика, застройка, оборона)
Генерал       → бонусы десантным войскам в наземном бою
```

Способности героев — горизонтальная прогрессия: разблокируют тактики/трейды, не только добавляют проценты к статам. Уникальные способности (пример: червоточины) — потенциально меняют сами правила перемещения/боя для конкретного героя.

## 7. Боевая система

Основная механика **сохраняется от Bytro** (консольное вдохновение, не копия кода/арта) — почасовые раунды, совокупное HP флота, приоритет линий получения урона. Расширяется модульностью и тактиками.

### 7.1 Базовая модель боя

```
Раунд боя = 1 реальный час (дискретные раунды, не непрерывный поток)

HP стороны = Σ HP всех кораблей во флоте + бонусы адмирала (хук)

Каждый раунд:
  1. Распределение входящего урона по линиям (см. 7.2)
  2. Применение урона
  3. Проверка погибших юнитов → publish "unit.died" (шина)
  4. Пересчёт линий для следующего раунда
  5. Повтор, пока одна из сторон не уничтожена или не отступила
```

Бой как и весь движок — детерминированная функция: `resolveBattle(fleetA, fleetB, seed) → BattleResult` с логом раундов (для воспроизведения боя на клиенте, см. `architecture.md` 4.2).

### 7.2 Линии и приоритет получения урона

```
Передняя линия   — принимает урон первой
Средняя линия    — принимает урон после передней
Дальняя линия    — принимает урон последней

Артиллерия — отдельный класс:
  наносит урон БЕЗ вступления в прямой бой (трейт, исключающий
  юнит из обычного распределения входящего урона по линиям)
  бьёт на расстоянии, сама может быть уязвима только если
  передняя/средняя линии полностью уничтожены (открытый вопрос
  баланса — конкретное правило уязвимости артиллерии уточняется
  при наполнении data-schemas.md)
```

Реализуется как трейты/хуки поверх базовой системы (`modulesystem.md` механизм 2) — линия и тип урона юнита определяют его место в конвейере распределения урона, не отдельная подсистема.

### 7.3 Бой за планету — та же механика, два связанных объекта

```
Десантный стек:
  Каждый флот несёт N десантных войск (отдельный стек, как мини-флот)
  Планета имеет свой гарнизонный стек десанта (оборона)

Бой за планету проходит ПО ТОЙ ЖЕ МОДЕЛИ что космический бой:
  почасовые раунды, совокупное HP, линии, приоритет урона
```

### 7.4 Двухфазный захват: орбита → десант

```
Фаза 1 — Орбитальный бой (космос)
  Атакующий флот vs оборона на орбите (если есть)
  Разрешается ПОЛНОСТЬЮ как обычный космический бой (7.1–7.2)

Фаза 2 — Наземный бой (десант) — НАЧИНАЕТСЯ ТОЛЬКО ПОСЛЕ ПОБЕДЫ В ФАЗЕ 1
  Десантный стек атакующего vs гарнизон планеты
  Та же модель боя (раунды, HP, линии)

  Бонусы наземной фазы:
    Генерал десанта      → хук к HP/урону десантного стека
    Орбитальная бомбардировка → хук от ПОБЕДИВШЕГО космического флота
      (читает состояние объекта, разрешённого в Фазе 1, как вход
      для Фазы 2 — порядок фаз строгий, не параллельный)
```

Архитектурно: наземный бой как процесс не существует, пока не разрешена орбитальная фаза — простая последовательная зависимость, не требует синхронизации двух параллельных систем.

**Одна орбита (без ближней/дальней).** Орбита у объекта одна. Флот либо стоит на
ней (`orbit: 'near'` — значение оставлено для обратной совместимости), либо в
перелёте/ещё не на орбите (`undefined`). Стоя на орбите, флот может бомбардировать
и высаживать десант — и в ответ его достаёт орбитальная ПВО гарнизона; отдельной
«дальней» безопасной орбиты-стоянки нет. Это сознательное упрощение интерфейса:
один слой вместо переключателя ближняя↔дальняя — меньше перегруза для игрока.
Прибытие на объект автоматически ставит флот на орбиту; действие `fleet.orbit`
(только `'near'`) — явный «выйти на орбиту».

### 7.5 Тактики авто-боя (применимо к обеим фазам)

```
Игрок ЗАРАНЕЕ настраивает для боя:
  1. Пресет — расположение юнитов по линиям/позициям (данные)
  2. Тактика — выбор трейта-тактики НА юнит из готового набора
     (например: "держит строй", "фокус по слабым", "залповый огонь")

Способности героев (раздел 6.2) могут РАЗБЛОКИРОВАТЬ новые тактики
  в этом же наборе — одна система с межсессионной прогрессией, не две

Разрешение — полностью на сервере, детерминированно, без участия
  живого игрока в моменте боя (см. resolveBattle выше)
```

Выбрана модель "готовые трейты-тактики", не пользовательский конструктор условий — дешевле архитектурно (расширение существующей системы трейтов), безопаснее (не нужна валидация произвольных пользовательских "программ").

## 8. Очки и победа

### 8.1 Источники очков

```
Сектор: scoreValue по виду/террейну + бонусы за постройки/улучшения (хук-конвейер).
        Планета — это сектор покрупнее (выше scoreValue); пустой узел — 1 очко.

computeScore(sector) = scoreValue + Σ хуков от построек/улучшений
totalScore(player)   = Σ computeScore по всем подконтрольным секторам

# Отложено (плоская модель карты, map-roadmap.md §5): региональный бонус
# «контроль группы секторов целиком» — будущая надстройка над атомарными
# секторами, не входит в текущую модель.
```

### 8.2 Момент подсчёта

Снапшот на момент завершения сессии (по серверному времени, не по присутствию игроков онлайн) — не накопительный учёт со временем. Простая, предсказуемая модель; trade-off — нет встроенного поощрения долгого удержания планеты сверх момента финала.

Технически: единичный вызов в core-engine при срабатывании любого из трёх триггеров завершения (раздел 3.2), не отдельная тикающая система.

## 9. Битвы и координация альянсов (UX-принцип)

Браузерная оптимизация через дискретные серверные тики (раздел 7.1), автоматизацию координации и защиту оффлайн-игроков. Удобство — через контекстный UI и пакетное вычисление результатов, а не реалтайм-микроменеджмент. Это прямое следствие event-based модели времени (`architecture.md` 4.1) — никто не обязан быть онлайн в момент разрешения боя.

## 10. Безопасность и честность как продуктовая черта

Архитектура проекта (server-authority, изоляция сессий, защищённые рынки, консервация состояния) синхронизирована с целями кибербезопасности (`architecture.md` раздел 5–8, OWASP-разбор раздел 6). Эта же архитектура — естественный полигон для практики secure coding, аудита API и защиты от эксплойтов в рамках разработки.

Монетизация (раздел 4.4) **пересмотрена**: P2W допустим, а камерная чувствительность сессии на 10 удерживается не запретом, а **гайками** — кэтч-ап для F2P, раскрытие шансов, soulbound-анти-RMT (`economy-roadmap.md`). «Защищённые рынки» выше теперь означают целостность транзакций и анти-фрод (леджер, идемпотентность, RMT-энфорсмент), а не запрет покупки силы.

## 11. Открытые вопросы дизайна

Не блокируют архитектуру/код, но требуют решения при детализации:

```
□ Финальный состав фракций (черновик: Люди-выжившие, Рой, Древние,
  Симбионты, Корпорация) — не подтверждён окончательно
□ [ОТЛОЖЕНО] Региональный слой и бонус «контроль группы секторов целиком»
  — вне плоской модели карты (map-roadmap.md §5); вернуться как надстройка
□ Конкретная формула убывающей эффективности доната на ускорение
□ Правило уязвимости артиллерии (когда становится доступна для урона)
□ Награды по месту — плавная функция или пороговые лиги
□ Дипломатия — формальные статусы союза, буферы по времени (была
  намечена для фэнтези-версии, не пересмотрена под текущую модель)
□ Тёмные события (космические аномалии) — не детализированы под
  итоговый сеттинг и текущую боевую/очковую систему
```

## Сводка решений сессии

1. Сессия консервируется при старте (раса/позиция/дека) — изменения недопустимы технически (fail-secure).
2. Две экономики: сессионная (сбрасывается) + глобальный мета-рынок.
3. Премиум-ресурс объединён: добывается на спорных секторах ИЛИ покупается; тратится только на ускорение/удобство, не на боевую силу.
4. Скорость времени (×1/×2/×4) — единая ось, определяющая максимальную длительность сессии.
5. Победа — гонка трёх триггеров: очки, кризис по времени, неактивность.
6. Коалиционный порог ниже одиночного на игрока — явный стимул к союзам.
7. Двойная петля прогрессии: сессионная (сброс) + межсессионная горизонтальная (снапшот при старте матча).
8. Бой — почасовые раунды, совокупное HP флота, линии (передняя/средняя/дальняя), артиллерия дальнего боя.
9. Захват планеты — двухфазный: орбита → десант, строго последовательно.
10. Тактика боя — выбор готовых трейтов на юнит, не пользовательский конструктор.
11. Очки — снапшот в конце сессии, база + бонусы от застройки.
