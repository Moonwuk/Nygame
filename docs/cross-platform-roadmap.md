# Кросс-платформенный клиент — технический roadmap

> **Направление:** доставить геймплей на десктоп-браузер, мобильный браузер и
> магазинные приложения (Android/iOS) из **одной TypeScript-кодовой базы**, с
> наилучшим качеством на каждом экране. Спутник `roadmap.md` (Этап 4) и
> `deep-technical-roadmap.md` (там — гонки, sync, server-perf, render-pipeline §7).
> Разбит на задачи-кирпичики в формате `backlog.md`: **1 кирпич ≈ 1 PR ≈ 1 сессия**.
> Уведомления — последней фазой, **отложены** по решению.

## Решение (decision record): PWA-first веб-клиент, не React Native

`roadmap.md` Этап 4 и `backlog.md` Блок G описывали клиент как **React Native +
Skia**. Здесь мы разворачиваемся на **PWA-first веб-клиент** и понижаем RN до
опционального будущего. Почему:

- **Ядро `@void/shared-core` — на TypeScript и обязано крутиться на клиенте**
  (детерминированный предпросмотр). Веб-клиент потребляет его напрямую; Flutter/Dart
  это не может, а RN тащит за собой переписывание рендера на Skia.
- **Один артефакт закрывает три из четырёх целей сразу.** PWA — это и десктоп-браузер,
  и мобильный браузер, и «установленное» приложение с иконкой; APK добавляется тонкой
  обёрткой (TWA/Capacitor). Обновления мгновенные (это веб).
- **Прецедент жанра.** Bytro (Supremacy/Call of War) с 2009 — один HTML5-движок,
  обёрнутый для мобилок, кросс-платформа на одних серверах. Качество там даёт не
  «нативность», а практики ниже.

RN остаётся как поздний опциональный путь, **не предпосылка**. Эта переориентация
заменяет бирки **G1–G4** (ниже — карта соответствия).

## Принципы

1. Одна кодовая база (web/TS); сервер — единственный источник истины.
2. Клиент переиспользует `@void/shared-core` напрямую: и рендер из состояния, и
   предпросмотр тем же `applyAction`. Никакой форкнутой копии ядра/данных.
3. Прогрессивное улучшение с откатом: Canvas2D → WebGL (Pixi) → WebGPU; PWA → store-wrap.
4. Жанровый фильтр: async, не twitch. Никакого lockstep/rollback — «предсказание» = это
   ограниченный предпросмотр, который сверяется с авторитетной дельтой.
5. Деградация без краша: нет сети → последний известный мир; нет GPU → Canvas; нет
   установки → обычная вкладка.

## Текущее состояние (факт, сверено с кодом)

- **`prototype/`** `[proto]` — один self-contained HTML (esbuild `build.mjs`),
  **Canvas 2D**, гоняет ядро **локально** (одиночка + локальный ИИ), мобильный UI,
  туман/радар/FPS, секторный граф (RNG-связи). **Не сетевой.** Вне типчека; гейт —
  `build.mjs` + `uitest.mjs`.
- **`@void/client`** `[cli]` — тонкий `MultiplayerClient` (применяет дельты через
  `applyDelta`). **Без UI и без создания WebSocket.**
- **`@void/server`** `[srv]` — `MatchRoom` + WS + **per-player дельты тумана на отправке**
  (`visibleState` на каждый broadcast — `matchRoom.ts`) + запускаемый `pnpm dev:server` +
  headless e2e-тест двух игроков. **✅ уже в коде:** персистентность `GameState`
  (`store/` — `MatchStore`, Postgres + in-memory адаптеры) и оффлайн-будилка
  (`msUntilNextEvent`/`tick` в `matchRoom.ts` — запуск наступивших `scheduled`-событий, пока все офлайн).
  Авторизация: JWT join-токен в WS-handshake + Origin-allowlist (`auth.ts`,
  включается `AUTH_JWT_SECRET` — см. `[→F7]` ниже).
- **`@void/shared-core`** — ядро + `diffState`/`applyDelta` + `visibleState` (проекция
  тумана как граница безопасности).

## Фаза 1 · Сетевой клиент (сервер = источник истины)

Цель фазы: превратить локальный клиент в серверно-авторитетный. Тестовый субстрат уже
есть — `pnpm dev:server` + headless e2e (`server/scenario.test.ts`).

### CP1.1 · Реальный WS-транспорт `[cli]` 🔒(CP0.1) — M
**Цель:** создать настоящий `WebSocket` и связать с `MultiplayerClient` (статус/снапшот/реджекшн).
**Подзадачи:** фабрика сокета; конфиг `url`/`matchId`/`playerId`; жизненный цикл open/close; проброс `onStatus/onSnapshot/onRejection/onError` в UI.
**Готово, когда:** клиент подключается к `dev:server`, получает `welcome` и рисует серверный снапшот.
**Зависит от:** CP0.1

### CP1.2 · Снять локальную авторитетность `[cli]` `[proto]` 🔒(CP1.1) — M
**Цель:** клиент перестаёт быть симулятором — рендерит из снапшотов сервера.
**Подзадачи:** убрать локальные часы и локальный ИИ из клиентского пути; время — из `state.time`/`serverTime`; (в прототипе оставить локальный режим как «песочницу» за флагом).
**Готово, когда:** при подключении к серверу клиент не мутирует состояние сам — только сервер.
**Зависит от:** CP1.1

### CP1.3 · Интенты из UI `[cli]` 🔒(CP1.1) — M
**Цель:** действия игрока → `Action` → `sendAction`; показ кодов отказа.
**Подзадачи:** строитель `Action` (`id` формата `session:player:seq`, `issuedAt`); маппинг UI-жестов на типы действий; UX для `rejection`-кодов.
**Готово, когда:** приказ из UI применяется сервером и возвращается дельтой; отказ показывается человеку.
**Зависит от:** CP1.1

### CP1.4 · Реконнект и резюме `[cli]` 🔒(CP1.1) — M
**Цель:** пережить обрыв мобильной сети без перезагрузки.
**Подзадачи:** экспоненциальный бэкофф; ресинк полным `welcome`/`state` по `seq`-гэпу (детали протокола — `deep-technical-roadmap.md` §3.4/§5); локальная очередь неотправленных интентов, флаш при восстановлении.
**Готово, когда:** разрыв и восстановление связи возвращают клиента в актуальное состояние без дублей действий.
**Зависит от:** CP1.1. (Честный туман `[→F6]` уже в коде — `visibleState` на отправке.)

> **Веха MVP-сети:** CP0 + CP1 ⇒ двое людей реально играют через `pnpm dev:server` в двух браузерах.

> **Статус (эта ветка): CP1.1–CP1.3 реализованы на прототипе.** `pnpm dev:proto-server`
> поднимает мир самого прототипа (его `kernel`+`data`+`newGame()`), а прототип получил
> оверлей подключения: войти в живую сессию как **Azure (p1)** или **Crimson (p2)**, либо
> остаться в одиночном режиме (локальная «песочница» за флагом `NET`, как и просили в CP1.2).
> В сетевом режиме приказы уходят `sendAction`, локальная симуляция/ИИ выключены, экран
> красит «тебя» зелёным независимо от p1/p2. Проверено end-to-end: приказ одного клиента
> реконструируется у другого через `applyDelta`. Осталось **CP1.4** (авто-реконнект/бэкофф/
> очередь неотправленных интентов — сейчас переподключение ручное через оверлей). Эта же
> сборка едет в Capacitor-обёртке (**CP6.2**) → тест мультиплеера через APK
> (`docs/multiplayer.md` → «Two phones, one session»).

## Фаза 2 · PWA (устанавливаемость, мгновенная загрузка, оффлайн-шелл)

Можно вести параллельно с Фазой 1 после CP0.1.

### CP2.1 · Web App Manifest `[cli]` 🔒(CP0.1) — S
**Цель:** установка на десктоп и мобильный домашний экран.
**Подзадачи:** `manifest.webmanifest` (имя, `display: standalone`, тема/фон), иконки incl. **maskable**, splash.
**Готово, когда:** браузер предлагает «Установить»/«На домашний экран»; приложение открывается в standalone.
**Зависит от:** CP0.1

### CP2.2 · Service Worker + app-shell `[cli]` 🔒(CP2.1) — M
**Цель:** мгновенный старт и работа оффлайн-оболочки.
**Подзадачи:** Workbox (прекэш app-shell, рантайм-стратегии); UX «доступна новая версия» с управляемым обновлением; не кэшировать игровые сокет-данные.
**Готово, когда:** второй запуск открывается мгновенно офлайн (оболочка); обновление накатывается контролируемо.
**Зависит от:** CP2.1

### CP2.3 · Кэш снапшота (IndexedDB) `[cli]` 🔒(CP2.2, CP1.1) — M
**Цель:** открытие сразу показывает «последний известный мир».
**Подзадачи:** хранить последний серверный снапшот в IndexedDB; на старте рисовать его, затем досинхрон дельтами; инвалидизация по `matchId`/версии данных.
**Готово, когда:** запуск без сети рисует последнее состояние; с сетью — догоняет до актуального.
**Зависит от:** CP2.2, CP1.1

### CP2.4 · Install UX и Lighthouse-бюджет `[cli]` 🔒(CP2.2) — S
**Цель:** пройти порог установимости (нужно для TWA в Фазе 6).
**Подзадачи:** `beforeinstallprompt` (desktop/Android), инструкция Add-to-Home для iOS, standalone/safe-area-стайлинг; Lighthouse PWA ≥ 80 в CI.
**Готово, когда:** Lighthouse PWA-проверка проходит порог; установка работает на desktop и Android.
**Зависит от:** CP2.2

### CP2.5 · Авто-обновление + force-update handshake `[cli][srv]` 🔒(CP2.2) — M
**Цель:** клиент «сам ловит и применяет» обновление; несовместимо-старый — не пускается в онлайн.
**Подзадачи:**
- **OTA — SW-first:** Service Worker (CP2.2) — основной механизм авто-апдейта; TWA наследует тот же веб, Capacitor (iOS) грузит ту же PWA. **Отдельный сервер не нужен** — SW тянет ассеты с веб-хоста, манифест версии статический. Capgo — только если позже понадобится офлайн-бандл с OTA. Нативная оболочка (новые плагины/права) — через стор/сайдлоад.
- **Force-update handshake:** на WS-коннекте (после JWT, AC-1.3) сервер отдаёт `minSupportedClientVersion`/`currentClientVersion`; ниже min → жёсткий экран обновления, ниже current → мягкое «доступно обновление». Server-authoritative; реальный backstop — валидация сообщений по протоколу (соврать про версию бесполезно). Инварианты — `versioning.md` §8.
- **Анти-soft-lock:** форс-гейт активно триггерит обновление SW (`skipWaiting`+reload); app-shell/манифест — network-first; форс-апдейт не теряет матч (тонкий клиент + reconnect/resume SV-2.2).
- **Комплаенс:** форс-апдейт-UX без тупика (стор-ревью, iOS 4.2); SRI/целостность ассетов (A08).
**Готово, когда:** новый веб-билд подхватывается клиентом без переустановки; несовместимый клиент получает экран обновления и не ломает сессию; форс-апдейт продолжает текущий матч после reconnect.
**Зависит от:** CP2.2 (SW), AC-1.3 (JWT-хендшейк).

## Фаза 4 · GPU-рендер карты (PixiJS v8 / WebGL)

Качество карты — главный визуальный рычаг. **Детали render-pipeline/culling/LOD/слоёв —
`deep-technical-roadmap.md` §7**; здесь — переход на GPU и кросс-DPI. Можно параллельно с 1–3.

### CP4.1 · Интеграция Pixi v8 (слой карты) `[cli]` 🔒(CP0.2) — L
**Цель:** перенести отрисовку карты на Pixi (WebGL; WebGPU — прогрессив с авто-откатом), HUD оставить на DOM.
**Подзадачи:** Pixi-сцена для секторов/связей/флотов/провинций/тумана; нормализованный render-list из состояния (не рендерить из полного `GameState` каждый кадр — §7.2); откат на Canvas2D.
**Готово, когда:** карта рисуется через Pixi с визуальным паритетом прототипа; на устройстве без WebGL — Canvas-откат.
**Зависит от:** CP0.2

### CP4.2 · Камера, culling, DPI `[cli]` 🔒(CP4.1) — M
**Цель:** плавный зум/пан на телефоне, чёткость на retina.
**Подзадачи:** инерционные pan/zoom; culling вне вьюпорта (§7.3); корректный `devicePixelRatio`; статический/динамический слои (§7.4).
**Готово, когда:** плавный зум 60fps на среднем телефоне; текст/иконки чёткие на retina.
**Зависит от:** CP4.1

### CP4.3 · Off-thread рендер/симуляция (опц.) `[cli]` 🔒(CP4.1) — L
**Цель:** не фризить UI на тяжёлых кадрах/предпросмотре.
**Подзадачи:** `OffscreenCanvas` + Web Worker для рендера и/или preview-прогона; деградация на основной поток, где не поддерживается.
**Готово, когда:** прокрутка/предпросмотр не блокируют ввод; есть безопасный фолбэк.
**Зависит от:** CP4.1

### CP4.4 · Перф-бюджет в CI `[cli]` 🔒(CP4.1) — S
**Цель:** не дать FPS деградировать незаметно.
**Подзадачи:** метрика кадра/числа draw call на сцене из N секторов; порог в `uitest`/headless.
**Готово, когда:** превышение бюджета роняет гейт.
**Зависит от:** CP4.1

## Фаза 6 · Упаковка в магазины (TWA Android + Capacitor iOS)

### CP6.1 · Android TWA `[cli]` `[sec]` 🔒(CP2.4) — M
**Цель:** установимый APK/AAB на базе PWA.

**Готово, когда:** Capacitor-сборка открывает игру на iOS-устройстве/симуляторе.
**Зависит от:** CP2.4

> над сборкой прототипа (`mobile/`: `capacitor.config.json`, `sync-www.mjs`, `patch-android.mjs`,
> `debug.keystore`) + CI `.github/workflows/android.yml`. iOS-сборка и store-ready bundle (CP6.1)
> — всё ещё впереди.

### CP6.3 · CI-артефакты сборок `[sec]` 🔒(CP6.1) — M
**Цель:** воспроизводимая выдача установимых сборок.
**Подзадачи:** пайплайн-джобы для APK/AAB (и IPA, когда будет подпись Apple); версияирование; гейт Lighthouse перед упаковкой.
**Готово, когда:** пайплайн отдаёт подписанный Android-артефакт по тегу.
**Зависит от:** CP6.1

## Последовательность и параллельность

- **Критический путь к «двое играют на телефонах»:** CP0 → CP1 → CP2 → CP6.
- **Параллельно после CP0.1:** Фаза 4 (рендер) и Фаза 2 (PWA) независимы; Фаза 3 — после CP1.3; Фаза 5 — после CP1.3, лучше с CP4.2.
- **MVP кросс-платформы:** Фазы 0–2 (+ CP6.1 для APK). **Полировка:** 3, 4, 5. **Уведомления:** 7 (потом).

| Фаза | Что даёт | Можно параллелить | Гейтится |
| ---- | -------- | ----------------- | -------- |
| 0 Фундамент | веб-клиент-пакет на ядре | — | — |
| 1 Сеть | двое играют через сервер | с Ф2, Ф4 | честный туман `[→F6]` ✅ (в коде) |
| 2 PWA | установка + оффлайн-шелл | с Ф1, Ф4 | — |
| 3 Предпросмотр | мгновенная отзывчивость | после CP1.3 | — |
| 4 GPU-рендер | плавная чёткая карта | с Ф1–Ф3 | — |
| 5 Ввод/UX | один UI на все экраны | после CP1.3 | — |
| 6 Упаковка | APK/iOS-сборки | — | Lighthouse (Ф2) |
| 7 Уведомления | удержание | — | `[→F3]` (отложено) |

## Открытые вопросы (решить при старте фазы)

- **Сборщик клиента:** Vite (HMR + зрелые PWA-плагины) против сохранения esbuild как в прототипе. Рекомендую Vite.
- **Эволюция vs новый пакет:** растить прототип в `@void/client` (CP0.2) или собрать клиент заново, переиспользуя извлечённые модули. Рекомендую извлечь модули и сделать прототип их потребителем — одна реализация рендера.
- **Хостинг клиента:** статика (Cloudflare Pages/Netlify) — TWA требует HTTPS + Digital Asset Links; согласовать с хостингом сервера (`roadmap.md` Этап 7).
