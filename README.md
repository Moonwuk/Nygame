# Void Dominion

**Real-time** космическая grand strategy с массовым мультиплеером для телефона и браузера —
в духе игр Bytro Labs (Iron Order, Call of War, Supremacy 1914), но в сеттинге тёмного
космоса. Мир идёт в непрерывном реальном времени и **круглосуточно, даже пока игрок
оффлайн**: это не пошаговая игра — «асинхронность» про манеру игры (зашёл, отдал приказы
на реальные часы вперёд, вышел). Жанр и темп — вдохновение; весь код, контент и механики
оригинальные. Главная ставка — **гибкое расширяемое ядро**: механики, юниты и фракции
добавляются данными, а не переписыванием логики.

**Играбельная альфа (Android APK, rolling-релиз):**
https://github.com/Moonwuk/Nygame/releases/download/alpha/void-dominion-alpha.apk

Браузерная версия — тот же single-file HTML: собери `pnpm run prototype` и открой
`prototype/dist/void-dominion.html`. Поднять сессию для друзей — `pnpm host` (раздаёт
игру и WebSocket на одном порту; полный путь и туннель — [`docs/launch-runbook.md`](./docs/launch-runbook.md)),
или боевой контур в контейнере — `pnpm stack` (docker-compose: сервер + Postgres). GitHub
Pages настроен (`pages.yml`), но на приватном репозитории Pages недоступен.

## Что играбельно сейчас

Играбельный клиент — это `prototype/`: single-file HTML на **реальном** ядре симуляции
(29 модулей поверх того же кернела, что и сервер). Режимы: скирмиш против ИИ, **командный
бой** (стороны-союзники) и **онлайн-матч до 10 живых игроков** — FFA / 5v5 / 2v2, вход по
позывным или по аккаунту, seat-lock-билеты, durable Postgres (матч и сессия переживают
рестарт сервера), ИИ-подмена отключившихся после грейса. Внутри: экономика и постройки,
флоты ⊕ **наземные дивизии**, бой с двухфазным захватом (орбита → десант) + **дальнобойная
артиллерия**, **герои**, **дерево технологий**, **дипломатия** по согласию
(мир/война/пакт/союз), **сессионная биржа ресурсов**, **шпионаж с контрразведкой**,
**«Хранитель»** (делегируй место ИИ на время сна), **мета-прогрессия командира** (XP между
матчами), онбординг и справочник, экран итогов, сессионный чат/пинги, **локализация RU/EN**,
мобильный UI (портрет, safe-area, 44px-таргеты). Серверный слой меты — **корпорации** и
**войны корпораций (AvA)** — работает по REST; клиентский UI к ним ещё впереди.

Полный, всегда-актуальный каталог механик — [`docs/game-description.md`](./docs/game-description.md);
живой снимок «что готово и как» — [`docs/state.md`](./docs/state.md).

## Принципы

- **Data-driven ядро** — игровые объекты описаны в JSON (`data/`), ядро лишь выполняет
  правила. Новый юнит/здание/технология — это правка данных, не кода.
- **Микроядро + модули** — маленькое неизменяемое ядро; всё остальное — подключаемые
  модули, общающиеся **только через шину** (события / хуки / реестр возможностей). Ни один
  модуль не импортирует другой; каждая точка расширения деградирует к базовому дефолту.
- **Детерминизм** — ядро это чистая функция: одинаковый `(state, action, context)` →
  одинаковый результат. Никаких `Math.random()`/`Date.now()` в ядре: seeded RNG (golden-тест)
  и время как параметр. Отсюда replay боёв, предпросмотр на клиенте и античит.
- **Server-authority** — клиент шлёт намерение, сервер решает. Валидация/авторизация/
  идемпотентность живут в слое действий перед редьюсером.
- **Туман войны как граница безопасности** — сервер физически не отправляет невидимое
  (`visibleState`-проекция на рассылке), а не «шлёт всё и прячет на клиенте».
- **Fail-secure** — любая ошибка → безопасный отказ со стабильным кодом `E_*`, не тихий
  пропуск; внутренние детали не утекают в ответ (только в логи сервера).

Дизайн-документы: [`docs/gdd.md`](./docs/gdd.md) (геймдизайн),
[`architecture.md`](./docs/architecture.md), [`modulesystem.md`](./docs/modulesystem.md),
[`tech-stack.md`](./docs/tech-stack.md) (ADR по стеку). Роадмап и статус —

## Структура монорепы

pnpm-workspaces; порядок пакетов = порядок сборки (ядро первым, в изоляции).

```
.
├── packages/
│   ├── shared-core/   # детерминированное ядро-симуляция (готово, покрыто тестами)
│   ├── action-layer/  # Stage 2: envelope-валидация, авторизация, идемпотентность, порядок
│   ├── server/        # авторитетный сервер — Stage 3 (WS + туман + Postgres + реестр матчей)
│   └── client/        # клиент — Stage 4 (PWA-first: transport-adapter + view-models + Vite-shell)
├── data/              # игровой контент (data-driven): units, factions, buildings, technologies, sectors, maps, …
├── prototype/         # играбельная альфа: single-file HTML на реальном ядре (+ netserver для онлайна)
├── mobile/            # Capacitor-обёртка прототипа → APK (бренд-иконка/сплэш, портрет)
├── deploy/            # docker-compose (сервер + Postgres), скрипты запуска
└── docs/              # проектные документы и роадмапы
```

### `@void/shared-core`

Детерминированное ядро — фундамент, собираемый первым и в изоляции (без сервера, БД и сети).
`createKernel(modules)` компилирует неизменяемый кернел из упорядоченного списка модулей и
даёт две чистые точки входа: `applyAction(state, action, ctx)` (применить намерение игрока)
и `advanceTo(state, ctx)` (сдвинуть мировые часы, отработав созревшие события в порядке
`(at, seq)` и эмитя непрерывные спаны `time.advanced` — модули копят ресурсы по формуле, а
не потикам). Порядок модулей = порядок массива, зафиксирован в манифесте и версионируется
на матч (инвариант детерминизма).

| Зона       | Что внутри                                                                                                            |
| ---------- | ------------------------------------------------------------------------------------------------------------------- |
| `kernel/`  | Микроядро: `createKernel`, `applyAction`, `advanceTo` (real-time), шина, хуки, реестр возможностей, манифест         |
| `state/`   | `GameState`; `visibleState` (туман — граница безопасности); `diffState`/`applyDelta` (дельта-sync); `hashState` (детект десинка); `diplomacy` (стойки war/peace/pact/alliance) |
| `action/`  | Контракт действия (`Action`/`Context`/`ApplyResult`), `Rejection`, `timeScale`                                       |
| `data/`    | zod-схемы игровых данных + `parseGameData` (валидация всего входа до ядра, A05/A08)                                   |
| `modules/` | Базовые модули-плагины (см. ниже) — каждая механика подключается через шину                                          |
| `rng/`     | Seeded PRNG (sfc32) — детерминизм; состояние сериализуется в `GameState`, поток заперт golden-тестом                 |
| `util/`    | `deepClone`/`deepFreeze` (immutable-контракт), общие хелперы казны/стеков/времени                                    |

**Базовых модулей — 25:** `sector`, `planetType`, `technology`, `scientist`, `economy`,
`market`, `movement`, `combat` (мелэ-бой, двухфазный захват орбита→десант), `orbital`
(ПВО + бомбардировка), `artillery` (дальнобойный standoff-огонь + barrage), `intercept`
(перехват на лейнах), `captureOnArrival` (walk-in захват необоронённого нейтрала),
`construction`, `arsenalSync` (live-рефреш владения каталогом построек), `army`
(флот ⊕ наземная армия + транспорт), `station`, `faction`, `hero`, `heroEffects`,
`diplomacy`, `espionage`, `steward` («Хранитель»), `victory` (data-driven очки/счёт),
`visibility` (память тумана), `effects` (интерпретатор `data.events`, trigger→effect).
Бой считает враждебность из `state.diplomacy` — атака только при объявленной войне.
Новая механика = новый модуль (+ данные), кернел не трогается: сервер собирает свой кернел
из 21 базового модуля, прототип — из 29 (базовые + свои: налоги, дивизии, командные цепочки,
стоячие приказы, форс-марш, …).

### `@void/action-layer`

Security-gate перед тем, как авторитет сервера применит действие: валидация конверта
`action.v1` → per-type zod-схемы payload'ов → авторизация по сессии → идемпотентность
(receipts) → строгий per-session порядок `clientSeq`; стабильные коды `E_*`. Слой намеренно
вне `shared-core`: ядро остаётся чистым и считает, что действие уже прошло эти ворота.

### `@void/server`

Авторитетный сервер на **реальном** ядре. `MatchRoom` (advance → авторизация → `applyAction`
→ **пофракционная дельта с туманом**: каждый игрок получает только свою `visibleState`),
WebSocket-слой (`createMultiplayerServer`) с JWT-джойн-токенами и Origin-allowlist. Плюс:
**персистентность** (`MatchStore`/`ReceiptStore`, in-memory + Postgres JSONB — матч и
квитанции переживают рестарт, commit-before-broadcast), **офлайн-шедулер** (`startClockDriver`
— мир идёт 24/7 без подключённых; живой хартбит держит часы у подключённых, не морозя экран
между действиями), **мульти-матч реестр с гибернацией** (`LazyRoomRegistry`) + **MatchKeeper**
(держит пул открытых матчей), **аккаунты логин/пароль** (scrypt, session-JWT → короткий
join-JWT), **action-gate** (`GATE=1`), эфемерные ally-пинги, сессионный чат. DoS-границы:
cap+FIFO на receipts, per-player rate-limit действий, per-IP лимиты HTTP. Мета-слой:
**корпорации** (роли/заявки/аудит) и **войны корпораций AvA** (влияние → вызов → ростер →
авто-сборка сессии → итог в историю) поверх durable-сторов.

Хостится через `packages/server/src/main.ts`; тот же срез несёт прототипный хост
`prototype/netserver.ts` для плейтестов. Безопасность включается окружением
(`AUTH_JWT_SECRET`, `GATE=1`, `SEAT_LOCK=1`, `ALLOWED_ORIGINS`) и уже прогоняется на боевом
контуре (`deploy/docker-compose.yml`); дефолтный dev-запуск — открытый ник-вход без пароля.
Впереди: OIDC-идентичность и мульти-процессный масштаб (pg-boss).

### `@void/client`

Клиент (Stage 4, направление — **PWA-first** web: TWA Android + Capacitor iOS). Держит
`MultiplayerClient` transport-adapter (`welcome`/`state`/`delta` через `applyDelta`,
`action`/`rejection` — включая `action.v1`-конверты со строгим `clientSeq`, когда сервер
объявляет `gated`, — latency `ping`/`pong`, лобби, ally-пинги), общий рендер-кит
(камера/holo/территории), view-models HUD и тема-токены, плюс Vite-shell с живой картой и
`?join=`-диплинком. Полноценный in-match HUD ещё впереди — играбельный клиент для игроков
пока `prototype/` (см. [`docs/cross-platform-roadmap.md`](./docs/cross-platform-roadmap.md)).

## Разработка

Требуется Node ≥ 20 и pnpm 10 (`corepack enable`).

```bash
pnpm install            # установка (по зафиксированному pnpm-lock.yaml)

pnpm test               # тесты (Vitest)
pnpm run lint           # ESLint (включая правила детерминизма для ядра)
pnpm run typecheck      # tsc --noEmit по всем пакетам
pnpm run format         # Prettier --write
pnpm run prototype      # собрать играбельный prototype/dist/void-dominion.html

pnpm run check          # lint + typecheck + test + docs-check — гонять перед коммитом
```

`pnpm run check` — единый гейт (`docs-check` проверяет целостность ссылок в `docs/` и
словарь зон бэклога). На момент написания зелёные **~1720 тестов (159 файлов)**, ещё 53
осознанно пропущены.

**CI** зеркалит гейт на каждый пуш: [`ci.yml`](./.github/workflows/ci.yml) гоняет
`pnpm run check` против сервисного Postgres (так что durable-тесты тоже бегут) + SCA через
**OSV-Scanner** (`pnpm audit` снят — npm закрыл audit-эндпоинты, 2026-07). Рядом:
security-пайплайн ([`security.yml`](./.github/workflows/security.yml): Semgrep, CodeQL,
Trivy, OSV, Gitleaks, TruffleHog, zizmor, SBOM) и сборка APK
([`android.yml`](./.github/workflows/android.yml), rolling-релиз `alpha`). `main` защищён —
изменения едут через PR с зелёным CI (см. [`CONTRIBUTING.md`](./CONTRIBUTING.md)).

  data-driven очки, туман войны (`visibleState` + память + radar-сигнатуры).
- **Этап 2 (Слой действий)** — ✅ готово: envelope-валидация, zod-схемы payload'ов,
  сессии, идемпотентность, `clientSeq`; включается на сервере через `GATE=1`.
- **Этап 3 (Сервер)** — 🧪 крит-путь до онлайн-сессии закрыт: WS-мультиплеер + дельта-sync
  с туманом на рассылке + персистентность (Postgres) + офлайн-шедулер + мульти-матч реестр
  с гибернацией + MatchKeeper + аккаунты/JWT + action-gate (всё opt-in окружением, боевой

- **Этап 4 (Клиент)** — 🧪 играбельный клиент живёт в `prototype/` (браузер + Capacitor →
