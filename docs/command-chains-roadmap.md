# Цепочки команд (offline-приказы) — технический roadmap

> Игрок составляет **цепочку приказов**, а **сервер исполняет её реальными часами,
> пока игрок офлайн** — «выставил и ушёл спать». Прямая реализация открытого вопроса
> «□ Логика оффлайн-игрока» (`architecture.md:556`) и UX-принципа GDD §9 (автоматизация
> + защита оффлайн-игроков). Ложится на уже готовый субстрат: оффлайн-будилку
> (`persistence-accounts-roadmap.md` PA-4.1), `schedule`/`advanceTo`, per-player очередь
> действий (`backlog.md` F5), action-layer (Stage 2). Парные доки: `architecture.md`,
> `modulesystem.md`, `gdd.md` (§1 async, §7.5 принцип «готовые блоки», §9 оффлайн), combat
> (server-resolved §7.1 — прецедент «set-and-forget»). Формат — кирпичики: `id · зона · статус · deps
> · «Готово, когда»`. Зоны: `[core]` · `[srv]` · `[cli]` · `[data]`. Статусы: ⏳ · 🔒(dep).

> **⚠️ СТАТУС (сверено с кодом, 2026-07-18): реализация вырезана в REL-1.**
> Прототип цепочек приказов — `orderQueueModule` + `subscriptionModule` (вырезаны оба
> модуля цепочек), `state.orders`/`state.subscribers`, все действия
> `order.enqueue/clear/pop/remove/block/retry/hold`, `QStep`/`popChainStep` и клиентский
> конструктор цепочек — **удалён из игры ради простоты релиза** (см. `backlog.md`, REL-1).
> Этих символов больше нет в исходниках; `order.*`-конверты цепочек отклоняются гейтом
> (`prototype/src/gateparity.test.ts`). Отметки ✅/⏳ ниже относятся к УДАЛЁННОЙ
> реализации и сохранены как дизайн-запись для возможного возврата фичи — это НЕ отражение
> текущего кода. В игре остались только **стоячие приказы** (`standingOrdersModule`:
> `order.auto`/`order.scramble`/`patrol.stamp`) — отдельная ветка, не цепочки.

## 0. Зачем, где живёт, чем ограничено (read first)

- **Async-игра наказывает за сон** без оффлайн-автоматики. Цепочка = «выставил приказы →
  сервер исполняет часами → проснулся с результатом».
- **Игрок ОФФЛАЙН → исполнение обязано быть серверным/в-ядре.** Клиент-макрос не годится
  (некому слать приказы — игрок спит). Ложится на оффлайн-будилку (PA-4.1) + `schedule`/`advanceTo`.
- **Детерминизм-инвариант (PA-4.2):** результат за оффлайн = непрерывная игра. Цепочки
  исполняются **детерминированно** — триггеры суть scheduled/доменные события в порядке `(at, seq)`.
- **Над СВОИМИ активами, human-in-the-loop при выставлении.** Контраст с Q11
  (`open-questions-visuals-content-pings.md`): авто-приказ ЧУЖИМ флотам = грифинг-вектор и
  запрещён. Цепочка командует только своим — это безопасно.
- **Философия — ограниченные готовые блоки, НЕ Тьюринг-полный скрипт** (принцип GDD §7.5
  «готовые блоки, не конструктор»). Дёшево, безопасно (нет валидации произвольных «программ»), без бот-гонки и без
  «игра играет сама».

## 1. Строительные блоки

- **Шаг** = отложенный интент — те же действия, что вживую (`fleet.move`, постройка,
  `fleet.orbit`/`assault`/`bombard`, `launch`/`patrol`, …).
- **Последовательность:** шаг N+1 стартует по **завершению** шага N (доменное событие:
  `build.completed` / `fleet.arrived` / `battle.resolved`).
- **Триггеры** (курируемый набор, не произвольные): по прибытии · по достройке · по итогу
  боя · по таймеру/задержке · по порогу ресурса · при атаке.
- **Гварды** (ограниченные предикаты): `if HP < X` · `if враг в радиусе` · `if ресурс ≥ Y`
  — готовый набор, не произвольная булева алгебра.
- **Стоячие приказы** (повтор): патруль, авто-ребилд разрушенного, hold / retreat-if-attacked.

## Фаза 0 · Модель

### CC-0.1 · Схема цепочки + каталог блоков `[core][data]` ✅ (частично, прототип) — M
**Сделано (prototype/src/game.ts):** цепочка = `QStep[]` на флот в `state.orders`
(JSONB-сериализуемо, шаги `move|orbit|assault|load|unload|bombard|wait|repeat` + служебные
штампы `until`/`blocked`); валидация — ручной type guard `isQStep` (fail-secure) + срез
клиентских штампов в `order.enqueue`; лимиты `MAX_CHAIN_STEPS`=32, `wait`≤`MAX_WAIT_HOURS`.
**Остаток до полного ✅:** zod-схема вместо type guard; каталог триггеров/гвардов как
данные (`data/*.json`) — делать вместе с CC-2.1.

### CC-0.2 · РЕШЕНИЕ: где исполняется `[docs][core]` ✅ — S
**Зафиксировано (де-факто в коде):** **гибрид** — *хранение и валидация in-core*
(`orderQueueModule`: `order.enqueue/clear/pop/remove/hold/block/retry`, очередь живёт в
`GameState.orders`, реплей/детерминизм сохранены), *решение «что выдать» — чистая функция*
(`serverQueueActions`), *выдача — host-драйвер* (`netserver.runServerQueues`, зеркально
`runServerAI`; в одиночке — `main.driveQueues`). Обоснование: шаги переиспользуют живые
действия и их ownership-гварды без дублирования в ядре; `order.hold` кладёт `order.wake`
в `schedule`, так что offline=continuous не зависит от подключённых клиентов. Полный
in-core перенос драйвера остаётся опцией при портировании цепочек в `shared-core`.

## Фаза 1 · Последовательное исполнение `[core]`

### CC-1.1 · Завершение шага → следующий шаг `[core]` ✅ (прототип) — L
**Сделано:** флот, ставший idle, исполняет голову цепочки (драйверы `driveQueues` /
`serverQueueActions`+`runServerQueues`); каждый шаг проходит валидацию хендлером живого
действия; **fail-secure превзойдён:** невалидный шаг не скипается молча, а **блокирует
цепочку с причиной** (`step.blocked`, `order.block`, событие `order.blocked` владельцу;
`order.retry`/✕/живой приказ снимают блок) — мир не виснет, план не деградирует втихую.
Тесты: `prototype/src/orderqueue.test.ts`.

## Фаза 2 · Триггеры `[core]`

### CC-2.1 · Курируемые триггеры `[core][data]` ⏳ (сделаны: on-arrival, at-time, 🔁) — M
**Сделано:** on-arrival (idle-драйверы), at-time/delay (`wait`+`order.hold`→`order.wake` в
`schedule` — оффлайн-будилка просыпается точно к концу выдержки), повтор 🔁 (маркер `repeat`,
`popChainStep` ротирует шаг в хвост — патруль, живущий оффлайн); реактивный
«дежурный вылет»/авто-штурм — авторитетно в NET (см. CC-3.1: `standingOrdersModule`).
**Остаток:** on-build-done / on-battle-resolved / on-threshold / on-attacked; каталог
триггеров как данные; перевод драйверов с idle-поллинга на события шины.

## Фаза 3 · Гварды и стоячие приказы `[core]`

### CC-3.1 · Ограниченные предикаты + повтор `[core][data]` ⏳ (повтор + стоячие приказы ✅) — M
**Сделано:** повтор до отмены (🔁 выше) — авторитетно, оффлайн; **стоячие приказы промоучены
в авторитетное состояние** (`standingOrdersModule`): `order.auto {fleetId,on}` →
`state.autoAssault`, `order.scramble {fleetId,on}` → `state.patrols` (сервер сам считает
центр/радиус/запас вылетов — клиент ничего про патруль не диктует), `patrol.stamp` — рантайм-
штамп драйвера (топливо/перезарядка/каденс, bounds-check по спеке крыла); чистые серверные
драйверы `serverAutoAssaultActions` (зеркало autoEngage) и `serverPatrolActions` (зеркало
drivePatrols; видимость цели — `identifiedNodes` ВЛАДЕЛЬЦА: сквозь чужой туман патруль не
видит; война обязательна — авто-войны нет; перезарядка по абсолютным меткам `rearmAt`,
работает при сколь угодно редких пробуждениях оффлайн-комнаты); хост-цикл
`netserver.runServerStanding` рядом с runServerQueues; `autoAssault`/`patrols` фильтруются
в fog как `orders` (только свои флоты); клиент в NET шлёт эти экшены и читает состояние —
кнопки «⚔ авто-штурм»/«🛩 дежурный вылет» работают по сети (сингл остался на локальных
драйверах). E2E: order.auto по WS → флаг в авторитетном broadcast.
**Остаток:** гварды (HP / враг-в-радиусе / ресурс), авто-retreat/авто-ребилд/hold.
**Готово, когда:** «отступить если HP<X» и «ребилд если разрушено» работают оффлайн; тест.

## Фаза 5 · Лимиты, анти-абуз, UX `[srv][cli]`

### CC-5.1 · Лимиты + конфликт-резолюция `[core][srv]` ✅ (базово) — M
**Сделано:** макс длина цепочки (`MAX_CHAIN_STEPS`=32 → `E_QUEUE_FULL`), `wait` конечен и
ограничен (≤720ч; `Infinity`/`NaN` отвергаются — JSONB не портится), клиентские штампы
(`until`/`blocked`) срезаются на входе; **живой приказ перебивает цепочку** (`dropChains`
на Move/Stop/штурм/обстрел/отступление/слияние, с тостом); чистка цепочек мёртвых флотов
(sweep на `time.advanced`). **Остаток:** rate-limit на создание; лимит числа активных
цепочек; формальная стыковка с F5-очередью.

### CC-6 · Лимит по ПРИКАЗАМ игрока + подписка `[core][srv][cli]` ✅ — M
**Сделано:** один enqueue = один ПРИКАЗ = 1..`MAX_ORDER_STEPS`(4) скомпилированных шагов
с общим `group`-штампом (сервер ставит сам, клиентский `group` срезается) — автопаттерн
«⚔ захват» (тап по чужому миру → `[move, assault]`, владелец через fog) стоит ОДИН слот.
Лимит `CHAIN_ORDERS_BASE`=3 через хук `order.chainLimit`; `subscriptionModule` поднимает
подписчику (`state.subscribers`, стампится мета-слоем; внутри матча не выдаётся) до
`CHAIN_ORDERS_PREMIUM`=5 — монетизация только-удобство (`main-menu.md` §5). 🔁 вне лимита
(и не больше одного, `E_LIMIT`); `order.remove` сносит приказ ЦЕЛИКОМ по индексу любого
его шага. Панель: строки по приказам, счётчик `N/лимит`, кнопки гаснут на лимите, апселл
«🔒 … до 5» (скрыт у подписчика), цель приказа — ссылка (панорама + кольцо); пипы карты
нумеруются по приказам. Соло зеркалит правило (`enqueueOrder`); плейтест-стаб подписки —
`localStorage['vd.premium']='1'`. **Остаток:** реальная выдача подписки мета-слоем.

### CC-5.2 · Конструктор цепочек (клиент) ⏳ (список-версия ✅) — L
**Сделано (список-версия):** сборка тапами по карте (режим «➕ строить» без ловушек:
канвас-баннер, тосты «шаг N → мир · весь план ≈Xч», сброс по Back/пустому тапу); **превью
и ETA** (`drawAimPreview` от хвоста плана, `chainEta` — накопительное ETA шагов + итог);
**план виден на карте** (`drawChainPlans`: пунктир по `planRoute`, нумерованные пипы,
глифы ⚔/▲/▼/☄/⏸/🔁, бейдж `▸N`/`⚠` на маркере флота); **редактирование:** ✕ у каждого
шага (`order.remove`), «↩ последний», «✕ очистить», «▶ пробовать снова»; статус исполнения
(«▶ шаг исполняется / ⏸ бой / ⏸ выжидаем ~Xч»); маршрут через мир при мире получает бейдж
«⚔ нужна война» в момент постройки; группа флотов строит теми же кнопками.
**Остаток:** drag-reorder/вставка в середину, флоучарт-вид, шаблоны цепочек.

## Открытые вопросы

```
✔ Где живёт — РЕШЕНО (CC-0.2): хранение in-core + host-драйвер (гибрид).
✔ Политика отказа шага — РЕШЕНО: пауза-и-уведомить (blocked + order.blocked), не скип.
✔ Конфликты цепочка vs живой приказ — РЕШЕНО: живой приказ снимает план (dropChains).
✔ Видимость плана врагу — РЕШЕНО: state.orders фильтруется в fog (только свои флоты);
  перехват разведкой — будущая ОСОЗНАННАЯ механика шпионажа (SPY-*), не утечка.
□ Выразительность MVP: какой набор триггеров/гвардов шипим (ограниченный, не скрипт).
□ Единый каталог стоячих приказов (патруль эскадрилий и т.п.) — общая система?
□ Не превращается ли в «игра играет сама» — баланс автоматики vs вовлечённости (плейтест).
```

## Инварианты (нарушение = баг)

- **Детерминизм:** триггеры = scheduled/доменные события в `(at, seq)`; **offline = непрерывная
  игра** (PA-4.2); никаких `Date.now`/`Math.random` в ядре.
- **Оффлайн-first:** исполняется будилкой 24/7 (PA-4.1) **без клиента**.
- **Server-authority + per-step валидация** (Stage 2 / ownership-гварды хендлеров); **fail-secure**
  — невалидный шаг → dead-letter, не краш.
- **Bounded** (GDD §7.5): готовые блоки, не бот-язык; **над своими активами** (не Q11-грифинг).
- **Только модуль + данные; кернел не трогаем.**

## Последовательность

Порядок отражает УДАЛЁННЫЙ прототип (см. баннер выше): `CC-0.1` → `CC-0.2` → `CC-1.1` →
`CC-2.1` → `CC-3.1` → `CC-4.1` → `CC-5.1` / `CC-5.2`. Флотская цепочка была играбельна в
прототипе (сборка на карте, ETA, правка, 🔁, честный отказ), но **вырезана в REL-1 — в
текущей игре её нет**. При возврате фичи следующий рубеж — планетарные шаги
(«построй→построй→пошли флот», CC-2.1 on-build-done) и гварды CC-3.1.
