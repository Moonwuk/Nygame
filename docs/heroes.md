# Система героев (data-driven, корабль + способности + дерево навыков)

> Дизайн-док. Реализация дробится на кирпичики — см. **Блок HERO** в `backlog.md`.
> Минимальный скелет уже в ветке `claude/hero-abilities` (PR #31): герой-позиция +
> три хардкод-способности (`hero.move`, `hero.path.create`, `planet.annihilate`).
> Этот документ описывает, во что он вырастает, и **почему именно так**.

## Видение (со слов заказчика)

- Герой — **отдельный корабль** на карте: со статами, его можно **уничтожить**.
- Способности **и** пассивки — через «модули». Дерево навыков — это **бонусы к
  способностям**; есть пассивный **баф усиления флота** и возможность **настроить сам
  корабль** компонентами.
- Дерево навыков делится на две ветки: **трансгуманизм** (любители имплантов) и
  **псионика** (условно — те, кто имплантов чурается).
- Героев будет **много разных**; игрок может выставить **до трёх** одновременно.
- Гибель героя → **кулдаун перед респауном**. Спавн — только на **планетах**, но у
  кого-то будет способность спауниться **на флоте / у союзника**.

## Ключевое решение: всё — данные, движок — один

Слово «модуль» в проекте уже занято: **модуль = плагин ядра** (`GameModule`, код,
исполняется через шину; порядок версионируется в манифесте). Если читать «способность =
отдельный код-модуль ядра» буквально, то `3 героя × N способностей × дерево × фитинги`
расплодит **сотни плагинов** — ровно против тезиса движка («добавлять механики **через
данные**, а не переписывая логику», CLAUDE.md / `modulesystem.md`).

Поэтому:

> **Герои, их способности, пассивки, фитинги и деревья навыков — это ДАННЫЕ
> (`data/*.json`), которые интерпретирует ОДИН `heroModule`-движок.** Для эффектов,
> которые данными не выразить, есть **код-сем**: реестр `capability`
> `hero.effect.<type>` — экзотика плагинится точечным кодом, не размывая ядро.

Это в точности три механизма шины (`modulesystem.md`), применённые по назначению:

| Механизм шины | Для чего в героях |
|---|---|
| **события** (emit/on) | реакция движка на мир: `fleet.destroyed` → смерть героя → респаун-кулдаун |
| **хуки** (пайплайн значения) | пассивки: баф флоту = вклад в `fleet.speed` / `combat.damage` из данных |
| **capability** (опц. связь + дефолт) | код-сем для уникальных эффектов: `hero.effect.<type>`, иначе движок обрабатывает тип сам |

Большинство способностей выражаются **данными** (тип-эффект + параметры), и движок
исполняет их обобщённо; «придумать можно много» — добавлением строк в JSON, не кода.

## Модель состояния (`GameState`)

Герой **переиспользует флот** для позиции/движения/боя/гибели (решение: не дублировать
`movement`/`combat`). Запись `Hero` хранит **стойкую личность** героя (она переживает
смерть корабля как «можно зареспаунить»), а текущий корабль — это обычный `Fleet`,
на который `Hero` ссылается.

```ts
interface Hero {
  id: string;                 // инстанс героя (не архетип)
  owner: PlayerId;
  archetype: string;          // -> data/heroes.json
  branch?: 'transhuman' | 'psionic';
  fleetId?: FleetId;          // корабль, которым герой управляет сейчас; undefined = мёртв/не заспавнен
  fittings: string[];         // -> data/heroFittings.json (слоты корабля)
  skills: string[];           // разблокированные узлы дерева
  cooldowns: Record<string, number>; // per-ability readyAt (ms), как уже сделано
  respawnAt?: number;         // если мёртв: время, когда можно зареспаунить
}
```

`GameState.heroes: Record<string, Hero>` (ключ — инстанс-id, не `PlayerId`, как в скелете —
**это миграция**: до трёх героев на игрока, фильтр по `owner`). Кэп «3 активных» проверяется
при спавне. `tempLanes`/`topology`/`heroSeq` остаются как есть. Всё JSON-сериализуемо,
детерминировано; времена — через `schedule`, кулдауны/респаун — `readyAt`-таймстемпы.

Связь героя с кораблём: корабль — `Fleet`, помеченный `traits: ['hero']` (или несущий
особый юнит-архетип). Гибель: `combat` уже эмитит `fleet.destroyed {fleetId, owner}` и
удаляет флот (`combat.ts`) — `heroModule` слушает это событие, находит героя по `fleetId`,
зануляет `fleetId` и ставит `respawnAt = after(респаун-кулдаун)`.

## Данные (новые `data/*.json` + Zod-схемы)

В духе существующих (`UnitDef`/`PlanetTypeDef`…), двухарг `z.record(k, v)`, дефолты на всё:

- **`data/heroes.json`** — архетипы: `{ name, branch?, ship{ unit | stats }, slots, startAbilities[], startPassives[] }`.
- **`data/heroAbilities.json`** — способности: `{ type, cooldownHours, range, cost?, params{} }`.
  Действие `hero.ability {heroId, abilityId, target}`; движок **обобщённо** валидирует
  владение/живость/дальность/кулдаун/стоимость и диспатчит по `type`.
- **`data/heroPassives.json`** — пассивки: `{ hook, scope, params }` (напр.
  `hook:'fleet.speed', scope:'ownFleetsNear', params:{bonus:0.1}}` — баф усиления флота).
- **`data/heroFittings.json`** — компоненты корабля: `{ slot, statMods{}, grants?[] }`
  (настройка самого корабля: статы / выдаёт способность).
- **`data/heroSkillTrees.json`** — две ветки (`transhuman` / `psionic`): узлы
  `{ requires[], grants: { ability? | passive? | statBonus? | abilityParam? } }` —
  «дерево = бонусы к способностям».

Всё проходит `parseGameData` до ядра (A05/A08), как остальной контент; загрузчик
(`scenario.ts loadShippedData`) дополняется новыми фрагментами.

## Движок `heroModule` (что регистрирует)

- **`hero.spawn {heroId, at}`** — поднять корабль героя. Правила: цель — **свой мир**;
  герой не жив (`!fleetId`); `respawnAt` прошёл; активных у игрока < **3**. Создаёт `Fleet`
  (статы из `archetype + fittings`), проставляет `fleetId`. Коды: `E_NO_HERO,
  E_FORBIDDEN, E_HERO_ALIVE, E_RESPAWN_COOLDOWN, E_HERO_CAP, E_BAD_SPAWN`.
  Способность спавна **на флоте / у союзника** = тип-эффект, ослабляющий проверку `at`.
- **`hero.ability {heroId, abilityId, target}`** — обобщённый диспетчер: проверки →
  `capability('hero.effect.' + type)` если есть, иначе встроенный обработчик типа →
  ставит кулдаун, списывает стоимость, эмитит `hero.ability.used`. Существующие
  `path.create`/`annihilate` переезжают сюда как **типы-эффекты в данных**.
- **`hero.move`** — переезжает на обычный `fleet.move` для корабля героя (переиспользуем
  маршрутизацию). Отдельное `hero.move` из скелета удаляется в пользу флотового.
- **`hero.fit {heroId, slot, fitting}` / `hero.skill.unlock {heroId, node}`** — настройка
  корабля и прокачка дерева (валидация слотов/`requires`/ветки; обычно вне боя).
- **Пассивки → хуки**: при сборке движок навешивает из данных вклад в `fleet.speed` /
  `combat.damage` / `economy.production` по `scope` (свой флот рядом, гарнизон мира и т.п.).
- **`on('fleet.destroyed')`** → смерть героя + респаун-кулдаун (см. выше),
  эмит `hero.died {heroId, owner}`.

Без `heroModule` ничего из этого нет — мир работает как раньше (мягкая деградация).

## Жизненный цикл

```
ростер (пред-матч, выбор до 3 архетипов — как фракции/технологии, см. backlog C3/B3)
  → hero.spawn (на своём мире)
    → бой/движение как у флота; способности из данных; пассивки через хуки
      → fleet.destroyed → hero.died → respawnAt = now + respawnCooldown
        → hero.spawn снова (когда кулдаун прошёл и есть слот)
```

## Две ветки дерева

`branch: 'transhuman' | 'psionic'` на архетипе/узлах. Дерево — `data/heroSkillTrees.json`:
узлы дают бонусы к способностям/статам/параметрам. Балансные числа — **данные**, не код;
ветка влияет на доступные узлы и, возможно, фитинги (импланты vs пси-усиления). Конкретный
контент (что именно даёт каждая ветка) — отдельный наполняющий кирпич.

## Что переезжает из скелета (PR #31)

| Скелет (сейчас) | Целевая система |
|---|---|
| `Hero {owner, location, cooldowns}`, ключ = `PlayerId` | `Hero {id, owner, archetype, fleetId, fittings, skills, cooldowns, respawnAt}`, ключ = инстанс; до 3 на игрока |
| `location: PlanetId` (телепорт-сущность) | корабль-`Fleet` (`fleetId`), движение/гибель через `movement`/`combat` |
| хардкод `hero.path.create` / `planet.annihilate` / `hero.move` | типы-эффекты в `data/heroAbilities.json`, диспетчер `hero.ability` |
| фиксированные константы в `hero.ts` | поля данных (cooldown/range/params) |

`tempLanes` / `topology` / `dead_world` / приватность героя в `visibleState` —
**остаются** и переиспользуются.

## Инварианты (как не сломать)

- **Детерминизм / сериализуемость**: всё состояние — JSON; никаких `Date.now`/`Math.random`;
  времена через `schedule`, кулдауны/респаун — таймстемпы.
- **Мягкая деградация**: нет модуля/данных → нет героев, не падает.
- **Fail-secure**: каждая проверка → `reject(code)` со стабильным кодом, без утечки деталей.
- **Туман**: герой приватен (`visibleState` уже режет чужих); корабль героя виден как
  обычный флот через существующую видимость/сигнатуры; `tempLanes` публичны.
- **Сервер-авторитет**: клиент шлёт `hero.*`-интенты, всё считает ядро.

## Открытые вопросы

1. **Ростер** — ✅ решён (HERO-9): **пред-матч** выбор до 3 **разных** архетипов через
   `SlotAssignment.heroes` (снапшот при сборке, как C3/совет учёных; дубли — fail-secure).
   Найм по ходу матча — возможная будущая надстройка, не базовая модель.
2. **Очки**: даёт ли герой-корабль счёт врагу при убийстве? Сейчас армия очков **не даёт**
   (счёт только территориальный, GDD §8.1) — супер-юнитов в ядре больше нет. Если решим, что
   герой стоит очков, делается через хук `victory.score` данными, а не отдельным полем юнита.
3. **Уникальность инстансов** — ✅ решён (HERO-9): ростер — три **разных** архетипа
   (`E_DUPLICATE_HERO`), в духе совета учёных.
4. **Стоимость способностей**: ресурсы казны, отдельная «энергия героя» или только кулдаун?
5. **Бой**: герой-корабль воюет в общем стеке флота или всегда соло? Влияет на эскорт/защиту.

Эти вопросы блокируют только наполнение/баланс, не каркас — каркас (движок + схемы +
жизненный цикл) можно строить, пока они открыты.
