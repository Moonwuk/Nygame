# Технический роадмап

> Порядок сборки проекта (инженерная карта, не календарь).
> Спутник документа `architecture.md`. Рабочий чек-лист для разработки — переносится в репозиторий.

---

## Главный принцип

**Ядро (`shared-core`) строится первым и в изоляции — без сервера, без БД, без сети.**

Это позволяет покрыть его тестами на 100% и быть уверенным в фундаменте до того, как на нём что-то ставится. Прямая профилактика «хрупкого ядра» Bytro: проверенный фундамент → на нём всё остальное строится с уверенностью.

```
0 · Каркас проекта
        ↓
1 · Ядро shared-core ★      ← строится и тестируется в изоляции
        ↓
2 · Слой действий ★         ← здесь живёт безопасность и античит
        ↓
   ┌────┴────┐
3 · Сервер   4 · Клиент      ← параллельно после стабилизации ядра
   └────┬────┘
        ↓
5 · Игровой цикл             ← ПЕРВАЯ ИГРАБЕЛЬНАЯ ВЕРСИЯ
        ↓
6 · Контент                  ← фракции, события через данные
        ↓
7 · Закалка                  ← аудит OWASP, сеть, нагрузка
```

★ — ключевые узлы. От ядра и слоя действий зависит всё остальное; здесь сосредоточена вся гибкость и безопасность.

### Что можно делать параллельно

После стабилизации слоя действий (этап 2) сервер и клиент опираются на готовое ядро и развиваются параллельно. В одиночку — последовательно; но архитектура позволяет подключить второго человека без конфликтов.

### Безопасность — не отдельный этап

Server-authority, fail-secure и валидация закладываются с **этапа 2** (слой действий). Этап 7 — это финальный аудит и сетевая инфраструктура, а не «прикручивание безопасности задним числом».

## Этап 0 · Каркас проекта

**Цель:** настроить монорепу и инструменты так, чтобы дальше можно было только писать логику.

Deliverables:

- Монорепа (pnpm workspaces или Turborepo): пакеты `shared-core`, `server`, `client`
- TypeScript-конфиг, ESLint, Prettier — единые на всю репу
- Тестовый раннер (Vitest) — настроен на `shared-core`
- CI-скелет: прогон тестов + `npm audit` (OWASP A03)
- Зафиксированный lock-файл в репозитории (A03)
- Схемы игровых данных: первичные структуры `units.json`, `factions.json`, `buildings.json`, `events.json`, `resources`
- zod-схемы для валидации этих данных

Definition of done: `npm test` и `npm run lint` проходят на пустом каркасе; CI зелёный.

---

## Этап 1 · Ядро shared-core ★

**Цель:** детерминированный игровой движок как чистая функция. Сердце проекта.

Deliverables:

- Структура `GameState` (планеты, флоты, игроки, ресурсы — как описано в архитектуре)
- Seeded RNG (свой генератор со своим зерном; никакого `Math.random()`)
- Чистый редьюсер: `applyAction(state, action, context) → newState`
- Модель времени: `advanceTo(state, timestamp) → newState` (разрешение событий)
- Базовые системы: движение флотов, расчёт боя, экономика
- Движок трейтов/эффектов: универсальная система триггер→эффект
- Накопление ресурсов за период оффлайна (вычисление по формуле, не тик)
- Полное покрытие тестами в изоляции

Критические правила (из архитектуры):

- Детерминизм: одинаковый вход → всегда одинаковый выход
- Никакого `Date.now()` внутри логики — время передаётся параметром
- Ядро не знает о конкретных юнитах/фракциях — только выполняет правила над данными

Definition of done: ядро запускается и тестируется **без** сервера и БД; набор тестов «состояние А + действие Б → ожидаем В» зелёный; бой воспроизводим (одинаковый seed → одинаковый результат).

Зависит от: этап 0.

---

## Этап 2 · Слой действий ★

**Цель:** единый узел, где сходятся безопасность, античит и игровая логика.

Deliverables:

- Типы действий и их zod-схемы (валидация входа, A05)
- Конвейер валидации: проверка прав на каждое действие (A01)
- Fail-secure: любая ошибка проверки → отклонение, не пропуск (A10)
- Идемпотентность: ID действия `session:player:sequence` + дедупликация
- Контракт «клиент шлёт намерение, не состояние»

Definition of done: невалидное/несанкционированное действие отклоняется с кодом ошибки (без утечки деталей); повтор действия по тому же ID не выполняется дважды; ошибка в проверке ведёт к безопасному отказу.

Зависит от: этап 1.

---

## Этап 3 · Сервер

**Цель:** авторитетный сервер — источник правды.

Deliverables:

- Fastify-скелет
- PostgreSQL + JSONB: персистентность `GameState`
- Redis + BullMQ: планировщик событий (отложенные задачи «флот прибудет через N часов»)
- WebSocket-слой (ws): пуш diff'ов (in-memory slice уже есть: `MatchRoom` + per-player дельта-рассылка с туманом войны)
- Очередь per-player: последовательная обработка действий одного игрока (закрывает race conditions, A10)
- Фильтрация по видимости (туман войны) **перед** отправкой клиенту
- Аутентификация: JWT в WebSocket handshake (A07)
- Retry + dead-letter для упавших событий; переживание рестарта

Definition of done: матч идёт на сервере; рестарт сервера не теряет состояние (PostgreSQL) и события (Redis); клиент получает только видимые данные; два одновременных действия не вызывают двойную трату.

Зависит от: этап 2. Параллелен этапу 4.

---

## Этап 4 · Клиент

**Цель:** мобильный клиент с картой и синхронизацией.

> **Направление пересмотрено:** клиент строится как **PWA-first веб-приложение**
> (одна TS-кодовая база на десктоп-браузер, мобильный браузер и store-обёртки),
> переиспользующее ядро на клиенте; React Native понижен до опционального будущего.
> Детальный roadmap с задачами/подзадачами —
> [`docs/cross-platform-roadmap.md`](cross-platform-roadmap.md). Список ниже — исходный
> (RN) вариант, оставлен для контекста.

Deliverables:

- React Native — оболочка приложения
- WebSocket-подключение + graceful reconnect (с тем же JWT)
- Skia-рендер звёздной карты: зум, скролл, culling (только видимая область)
- Применение diff'ов от сервера к локальному состоянию
- Предпросмотр через `shared-core` («если атакую — что будет?»)
- Оптимизация: обновление UI только при изменении данных; подгрузка секторов по мере скролла

Definition of done: карта рендерится плавно на телефоне; переподключение после потери сети восстанавливает состояние без дублей; предпросмотр боя совпадает с серверным результатом (детерминизм).

Зависит от: этап 2. Параллелен этапу 3.

---

## Этап 5 · Игровой цикл — ПЕРВАЯ ИГРАБЕЛЬНАЯ ВЕРСИЯ

**Цель:** замкнуть полный цикл матча. Здесь продукт становится играбельным.

Deliverables:

- Лобби / матчмейкинг (5–10 игроков)
- Полный жизненный цикл матча: создание → игра → завершение
- Условия победы (контроль 60% секторов / уничтожение соперников)
- Push-уведомления (флот прибыл, на тебя напали)
- Одна играбельная фракция, базовые механики

Definition of done: можно реально сыграть матч от начала до конца с одной фракцией; геймплей и баланс тестируемы вживую.

Зависит от: этапы 3 и 4.

---

## Этап 6 · Контент

**Цель:** наполнить игру через данные, не трогая ядро.

Deliverables:

- Остальные фракции — описаны в JSON (Некромант, Рой, Древние, Симбионты и т.д.)
- Способности и трейты фракций — через движок эффектов
- Тёмные события (космические аномалии) — раз в 6–12 часов
- Балансные константы в конфиге

Проверка гибкости: новый юнит = новая запись в JSON за минуты; новая фракция = описание в данных, без изменения логики. Если это не так — вернуться к ядру.

Definition of done: ≥3–5 фракций с уникальными механиками играбельны; добавление контента не требует изменения кода ядра.

Зависит от: этап 5.

---

## Этап 7 · Закалка

**Цель:** аудит безопасности, сетевая инфраструктура, устойчивость к нагрузке.

Deliverables:

- Полный проход по чек-листу OWASP Top 10 2025 (см. `architecture.md`)
- Сетевая инфраструктура: Cloudflare (DDoS + скрытие IP), TLS 1.3, приватная сеть для БД и Redis
- Security misconfiguration (A02): закрытые порты, нет дефолтов, debug выключен
- Логирование и алерты (A09): всплеск отклонённых действий → сигнал о чите/атаке
- Нагрузочное тестирование (несколько параллельных матчей)
- Тест отказоустойчивости: убить сервер посреди матча → проверить восстановление

Definition of done: чек-лист OWASP закрыт; БД и Redis недоступны из интернета; сервер за Cloudflare; падение и рестарт сервера не теряют прогресс; нагрузка из нескольких матчей держится стабильно.

Зависит от: этап 6.

## Первый шаг разработки

Старт этапа 1 — структура `GameState` и сигнатура чистого редьюсера:

```
applyAction(state: GameState, action: Action, context: Context): GameState
```

С неё естественно вырастает всё ядро. Дальше — модель времени `advanceTo`, затем системы боя/экономики/движения, затем движок трейтов.

## Статус реализации

> Обновляется по мере разработки. Отражает, что уже лежит в репозитории.
> Подробный снимок текущего состояния (что готово и как работает, для
> восстановления контекста) — [`docs/state.md`](state.md).

- **Этап 0 · Каркас** — ✅ готово. Монорепа (pnpm workspaces: `shared-core`, `action-layer`, `server`, `client`), TypeScript (strict), ESLint (flat config + правила детерминизма), Prettier, Vitest, CI (lint + typecheck + test + `pnpm audit`), зафиксированный `pnpm-lock.yaml`. Игровые данные `data/*.json` + zod-схемы в `shared-core`.
- **Этап 1 · Ядро** — ✅ собрано (без движка эффектов — вынесен кирпичом, см. конец пункта). Фундамент: `GameState` (+ расписание событий), seeded RNG (sfc32, детерминизм + golden-тест), микроядро (`createKernel`/`applyAction`) с шиной событий, хук-конвейерами, реестром возможностей и манифестом модулей; fail-secure в редьюсере. **Модель времени `advanceTo`** (real-time): запланированные события исполняются по непрерывному времени в порядке (`at`, `seq`), континуальное накопление через событие `time.advanced`, dead-letter упавших событий (мир не зависает), детерминизм и guard от runaway-расписания. **Базовые модули-плагины**: **карта** (планеты-узлы + лейны/`links` + **типы секторов** через `sectorModule`: поле астероидов −25% скорости / +10% живучести, пустое пространство +15% скорости, +25% урона в своём секторе — всё data-driven через хуки) и движение флотов (`fleet.move` → маршрутизация Дейкстрой по лейнам, многохоповое движение, столкновение на узле → бой; хук `fleet.speed`), экономика (производство в **казну игрока** на `time.advanced`, хук `economy.production`, ежедневное **содержание** юнитов и гарнизонов) и **бой с захватом планет** (GDD §7: `battles` в состоянии, почасовые раунды `combat.tick`, линии перёд/середина/тыл + артиллерия, хук `combat.damage`, `unit.died` → `battle.resolved`; **двухфазный захват орбита→десант** §7.4 — десантные стеки vs гарнизон, смена владельца `planet.captured`, необоронённый мир занимается без боя) и **постройка зданий и юнитов за ресурсы казны** (`building.construct` / `building.upgrade` / `unit.build`: стоимость списывается из казны вперёд fail-secure, завершение отложено через расписание с учётом `timeScale`, готовые юниты пополняют гарнизон планеты, а вложение сгорает при потере планеты до завершения). **Здания** (`BuildingInstance` = тип + уровень + HP, data-driven `data/buildings.json`): уровни 1–3 с прокачкой (Форт→Крепость: HP и бонус обороны растут 35→65), каждое здание даёт гарнизону бонус к обороне через хук `combat.damage` (по умолчанию +1%), а наземный штурм изнашивает HP зданий и разрушает их (`building.destroyed`), снимая бонус. **Флот ⊕ наземная армия раздельны** (`armyModule`): у юнитов есть `domain` (`space`/`ground`), наземная армия (танки/десантная пехота/ополчение) живёт в гарнизоне и путешествует только грузом флота — `army.load`/`army.unload` перекладывают её между гарнизоном и трюмом флота в пределах `cargoCapacity` кораблей (`cargoSize` груза; отдельный десантный корабль везёт много); перевезённая армия и есть десант при штурме. **Типы планет** (`planetTypeModule`, data-driven `data/planetTypes.json`): тип мира (terran/barren/oceanic/volcanic/gas_giant) множит производство и даёт гарнизону владельца модификатор наземной обороны — через хуки `economy.production` и `combat.damage` (знак учитывается, складывается со зданиями), как секторы. **Сессионное дерево технологий** (`technologyModule`, `data/technologies.json`): `technology.research` списывает стоимость из казны, планирует `technology.complete` с учётом `timeScale`, хранит `Player.technologies`, проверяет prerequisites/one-active-research, открывает юниты/здания через хук `construction.requirement` и применяет бонусы через `economy.production`/`fleet.speed`/`combat.damage`; без модуля гейт мягко деградирует. **`timeScale`** (MatchConfig) масштабирует все длительности. **Победа и счёт** (`victoryModule`): `GameState.match`, scoreboard и `match.ended` по доминированию/уничтожению/лимиту счёта/тайм-ауту. **Туман войны** (`visibleState`, `state/visibility.ts`) — чистая проекция-граница безопасности: identify (1 прыжок) + radar-сигнатуры (`◆S/M/L` по `UnitDef.signature`/`radarRange`), прячет чужую казну/технологии, контент невидимых миров и всё расписание; покрыта тестами incl. anti-leak по JSON; память last-known (вариант B, `visibilityModule`), радарные бонусы тех/фракций (`radarRangeBonus` в effects/passives) и хелпер `isVisibleTo` — на месте. **Прототип** (`prototype/`, играбельный single-file HTML на реальном ядре) — карта, real-time, движение, бой, постройка, туман (память+сигнатуры, dev-превью). **Дипломатия в ядре** (D1–D4, `diplomacyModule`): стойки war/peace/pact/alliance в `state.diplomacy` (дефолт пары — war = FFA), эскалация односторонняя (`diplomacy.declare`), смягчение по взаимности через то же действие — consent-протокол D3 (первый дружественный `declare` пишет оффер в `state.diplomacyOffers`, встречный — коммитит пару; оффер виден только своим участникам), `E_BOT_ALLIANCE` (коалиции только между людьми), свип офферов на `player.eliminated`, capability `diplomacy` (stance→relation); `combat.isHostile` читает стойки напрямую. **D4:** прототип использует этот же ядровый модуль — одна реализация `diplomacy.declare` на весь репозиторий. **Система героев** (HERO-0..9, `heroModule` + 5 data-каталогов `heroes/heroAbilities/heroPassives/heroSkillTrees/heroFittings`): архетипы, способности, пассивки, дерево навыков и фитинги — данные, движок один; обобщённый `hero.ability` (диспетчер по `type` + capability `hero.effect.<type>`), `hero.spawn` (кэп 3 активных; свой мир / союзный мир / абордаж своего флота через маркер-способности), `hero.skill.unlock` (ветки transhuman/psionic, `requires`/цена), `hero.fit` (слот-бюджет архетипа, no-refit); позиция героя = нода его корабля, смерть по двум сигналам (`unit.died`/`fleet.destroyed`), респаун в столице; пред-матч ростер до 3 **разных** архетипов (`SlotAssignment.heroes`). **«Хранитель»** (Steward v1, `stewardModule`): `steward.delegate/recall` с тайм-боксом, серверный ИИ играет делегированное место позой `defend`, гейт техно `ai_stewardship` + учёный «Куратор» (roadmap v2 — `steward-roadmap.md`). **Совет учёных** (до 2, `Player.scientists`, гейты `has_scientist`, `research.slots`-бонус). **Шпионаж** (SPY-1/2: `espionage.spy` — краденые intel-окна + контрразведка с обнаружением) и **сессионная биржа** (`marketModule`: list/buy/cancel, эскроу, комиссия сжигается). **Предматчевые выборы** (C3): `SlotAssignment.technologies` выдаёт стартовые технологии как completed при сборке матча (fail-secure на неизвестный id). План эволюции ядра пройден: победа/счёт ✅ → туман войны ✅ → типы планет ✅ → фракции ✅ → древо технологий ✅ → дипломатия ✅ → герои ✅. Из deliverables этапа не собран только универсальный движок трейтов/эффектов: `EffectRuleSchema`/`data/events.json` валидируются, но интерпретатора-модуля нет (юнитовые `traits` модули читают точечно: artillery/immobile/hero) — вынесен отдельным кирпичом EFX-1 в backlog. Бывший боевой монолит разрезан по швам шины на `combat` (melee) + `orbital` (ПВО/бомбардировка) + `artillery` (standoff) + `intercept` (перехват на лейнах) — каждый модуль деградирует мягко. Дальше — UI-стадия (конструктор юнитов, панели, предматчевый экран) и серверные этапы.
- **Этап 2 · Слой действий** — ✅ готово и подключено. `@void/action-layer`: `ActionEnvelope` validation, authorization against server session, idempotency receipts, strict per-session `clientSeq` gate, per-type zod payload-схемы, стабильные коды отказа. Подключён к серверу единым гейтом перед редьюсером (SV-1.1/1.2), стора ограничены (FIFO/LRU).
- **Этап 3 · Сервер** — 🚧 крит-путь до онлайн-сессии закрыт. Авторитетный WS-сервер на Fastify (`/health`·`/ready`·`/metrics`·pino·drain), match-actor + `LazyRoomRegistry` (N матчей/процесс, гибернация/пробуждение, мир идёт 24/7), мульти-матч браузер с read-model `GET /matches` (+archive), строгий commit-before-broadcast с durable-персистентностью (Postgres/in-memory, матчи+места+квитанции переживают рестарт), туман-на-отправке. Безопасность: JWT-рукопожатие (пин алгоритма, привязка к матчу/сессии, Origin-allowlist), серверный `sessionId`, action-gate (validate→payload-schema→authorize→dedup→sequence) — под флагом `GATE=1`; клиент `action.v1` готов и самонастраивается по `gated`-рукопожатию. Аккаунты логин/пароль (scrypt) + session/join-JWT, MatchKeeper держит пул открытых матчей (`GET /matches/open`). CI гоняет gate против сервис-Postgres. Дальше: включить auth/gate на играбельном пути (прототип-хост), OIDC-идентичность, мульти-процессный масштаб.
