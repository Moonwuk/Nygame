# Режимы игры — roadmap

> Как в матче определяется **условие победы**. Сегодня движок уже поддерживает
> «обычный» режим (очки/доминирование/устранение/тайм-аут) через
> `victoryModule` + `config.victory`; этот док описывает, как это оформить в
> **выбираемые режимы** и как добавить новые (захват столицы, удержание точек и
> др.) **не трогая ядро** — новый режим = данные + опционально модуль.
>
> Парные доки: `gdd.md` (§3 завершение/очки, §8 очки), `modulesystem.md`
> (механика = модуль через шину), `matchmaking-roadmap.md` (`MM-1.1`/`MM-1.2` —
> где режим выбирается), `main-menu.md` (§4.1 список карт/режимов),
> `metagame.md` (AvA — командный режим меты). Формат — кирпичики:
> *зона · статус · зависимости · «Готово, когда».* Статусы: ✅ · ⏳ · 🔒(dep).

## Текущее состояние (факт)

`victoryModule` (`packages/shared-core/src/modules/victory.ts`) уже вычисляет из
авторитетного состояния **четыре** исхода и данные-конфиг их параметризует:

| Исход | Правило (сейчас) | Конфиг |
|---|---|---|
| **score** | набрать `scoreLimit` очков (база 500, GDD §3.2) | `config.victory.scoreLimit` |
| **domination** | держать долю захватываемых секторов | `config.victory.dominationPercent` (0.6) |
| **elimination** | остаться единственным «живым» (есть территория) | — |
| **timeout** | форс-финал по лимиту дней (скорость времени) | `config.victory.endsAt` / `timeScale` |

Очки — `CONTROL_BASE` + тип планеты + террейн + Σ построек, плюс seam
`hook('victory.score', …)` для модулей. То есть **новые режимы уже ложатся на
существующий data-driven каркас**: параметры через `config`, доп-правила через
модуль и хуки, kernel неизменен (инвариант `architecture.md`, `modulesystem.md`).

## Зависимости

Ядро строится в изоляции. **Выбор** режима в лобби — `matchmaking-roadmap.md`
(`MM-1.2`, сейчас написан про «фракцию» — расширить до «режим+фракция+профиль»).
Командные/межсессионные режимы (AvA) — `metagame.md` Контур 2 (сервер, Этап 3+).

## Фаза 0 · Каркас режимов (fmwk)

### GM-0.1 · `config.mode` + реестр режимов `[core][data]` ⏳ — M
**Цель:** режим — это **именованный пресет**: `victory`-конфиг + список активных
mode-модулей + требования к карте (какие теги узлов нужны). Один вход, из которого
собирается матч.
**Подзадачи:** zod-схема `GameModeDef` (`data/modes/*`): `id`, человекочитаемое имя
(i18n-ключ), пресет `victory`, `requiredMap` (напр. «нужны узлы-столицы»/«нужны
control-точки»), список модулей режима; `parseGameData` валидирует; матч-манифест
несёт `modeId`; консервация при старте (GDD §2 — режим фиксируется, смена
отклоняется сервером, fail-secure).
**Готово, когда:** матч стартует с `modeId`, конфиг победы берётся из пресета, смена
режима на лету отклоняется; тесты валидации.

### GM-0.2 · Мульти-исход и приоритет `[core]` ⏳ — S
**Цель:** несколько условий победы сосуществуют детерминированно (сейчас порядок
domination→score→timeout зашит). Режим объявляет, какие исходы включены и их
приоритет.
**Подзадачи:** упорядоченный список активных чек-условий в `config`; `victoryModule`
идёт по списку; дефолт = текущий порядок (обратная совместимость).
**Готово, когда:** режим включает/исключает/переупорядочивает исходы; golden-тест на
порядок при одновременном срабатывании.

### GM-0.3 · Выбор режима в лобби `[client][srv]` 🔒(GM-0.1) — M
**Цель:** игрок видит и выбирает режим при создании/входе в лобби.
**Подзадачи:** каталог режимов в read-model лобби; UI-карточка (имя/описание/иконка,
i18n); интент `createLobby(mapId, modeId)`; сервер валидирует «карта поддерживает
режим». Ложится на `MM-1.1`/`MM-1.2`, `main-menu.md` §4.1.
**Готово, когда:** режим выбирается в лобби и доезжает до манифеста матча.

## Фаза 1 · «Обычная» — очки (соло + коалиция) (в основном ✅)

### GM-1.1 · Пресет `standard` (score/domination/timeout) `[data]` ✅→⏳ — S
**Цель:** оформить уже работающее в явный режим-пресет.
**Подзадачи:** `data/modes/standard.json` = текущие дефолты (`scoreLimit` 500,
`dominationPercent` 0.6, timeout по скорости). Ничего в ядре не меняем.
**Готово, когда:** режим `standard` = сегодняшнее поведение, покрыт тестом-эквивалентом.

### GM-1.2 · Коалиционный порог очков `[core]` 🔒(диплом. D1/D2) — M
**Цель:** победа коалиции по суммарным очкам с порогом ниже линейного (GDD §3.3:
`500 × N × 0.7`) — явный стимул к союзам.
**Подзадачи:** суммирование очков по формальной коалиции (нужен статус альянса из
`diplomacyModule`, `core-roadmap.md` CR-3.2/3.3); порог из данных (тюн-баланс, не
код); совместная победа = место всем членам.
**Готово, когда:** коалиция побеждает по суммарным очкам при пороге `< N×соло`; тесты
на 2–4 игроков; «неформальный ненап» очки НЕ суммирует (нужен явный статус).
**Примечание:** без модуля дипломатии деградирует до чисто соло-очков.

## Фаза 2 · «Захват столицы» (HQ / regicide-lite)

### GM-2.1 · Сущность столицы `[core][data]` ⏳ — M
**Цель:** у каждого игрока есть **штаб/столица** — узел, потеря которого критична.
**Подзадачи:** флаг `isCapital`/`capitalOf` на планете в `GameState` (назначается при
сборке старта — Фаза фракций, `core-roadmap.md` CR-1.3); данные: где столица (домашний
узел стартового профиля); опц. усиленная оборона столицы по умолчанию (хук
`combat.defense`), чтобы захват требовал операции, а не рейда 1-го дня.
**Готово, когда:** старт помечает столицу игрока; сериализуется; тест.

### GM-2.2 · `capitalModule`: поражение по потере штаба `[core]` 🔒(GM-2.1) — M
**Цель:** игрок выбывает, когда его столица захвачена (а не когда потеряна ВСЯ
территория, как сейчас в `elimination`).
**Подзадачи:** на `planet.captured` — если это столица игрока P и захвачена врагом →
`player.eliminated{reason:'capital-lost'}` (тот же слив флотов/финализация, что в
`victoryModule`); последний выживший / держатель своей столицы → `elimination`.
Опции режима: «захват = мгновенно» vs «нужно удержать столицу T времени» (анти-
чиз против шального рейда). Деградация без модуля: обычные правила.
**Готово, когда:** захват столицы устраняет игрока; удержание своей = выживание;
командный вариант (пал последний штаб команды); тесты.

### GM-2.3 · UX столицы `[client]` 🔒(GM-2.1) — S
**Цель:** столица явно читается на карте/в панели (своя и чужая — как приоритетная
цель), таймер «под ударом», алерт при осаде штаба (async-лента, `main-menu.md` §5.1).
**Готово, когда:** столицы видны, атака штаба поднимает алерт.

## Фаза 3 · «Удержание точек» (control points / king-of-the-hill)

### GM-3.1 · Контрольные точки в данных `[core][data]` ⏳ — M
**Цель:** на карте есть особые **control-узлы**; победа — держать `K` из них
суммарно `T` игрового времени.
**Подзадачи:** тег `controlPoint` на секторах/планетах (данные карты); конфиг режима
`{ pointsToHold: K, holdDuration: T, contestReset: bool }`; `requiredMap` в
`GameModeDef` проверяет наличие ≥K точек.
**Готово, когда:** карта объявляет control-точки; конфиг режима валидируется.

### GM-3.2 · `holdPointsModule`: таймеры удержания `[core]` 🔒(GM-3.1) — L
**Цель:** детерминированно копить время удержания и объявлять победу.
**Подзадачи:** per-player (или per-coalition) прогресс в `GameState` (mode-состояние,
не в kernel); на `time.advanced` — если игрок держит ≥K точек, копим Δt; иначе
поведение по конфигу (пауза vs сброс `contestReset`); достиг `T` → `match.ended
{reason:'hold'}`. Смена владельца точки (`planet.captured`) обнуляет её вклад.
Оффлайн-безопасно: таймеры идут по серверному времени, мир 24/7 (`gdd.md` §9,
offline-планировщик сервера) — держатель побеждает, даже если соперники оффлайн.
**Готово, когда:** удержание K точек в течение T приводит к победе; перехват точки
корректно прерывает/сбрасывает; golden-тест на детерминизм таймеров.

### GM-3.3 · UX точек и прогресса `[client]` 🔒(GM-3.1) — M
**Цель:** видно, где точки, кто держит, сколько накоплено (прогресс-бары), кто
близок к победе.
**Готово, когда:** статус/прогресс каждой точки и лидера читается на карте/HUD.

## Фаза 4 · Мои предложения (кандидаты в режимы)

Приоритизированы по «дёшево на текущем каркасе» → «дороже». Все — данные+модуль,
kernel не трогаем.

### GM-4.1 · «Экономическая гонка» / ресурсный контроль `[data]` — S (дёшево)
Победа — первым накопить X ресурса **или** держать все `premium_shard`-планеты
(спорные точки, `gdd.md` §4.3) суммарно T. Почти чистый конфиг + маленький
чек-хук. Даёт невоенный путь к победе и делает горячие точки центром карты.

### GM-4.2 · «Регицид / охота на героя» `[core+data]` — M
У игрока есть ключевой **герой-флагман** (адмирал, `gdd.md` §6.2); его гибель =
поражение. Переиспользует `hero`-модуль и события боя; вариант «захвата столицы»,
но мобильная цель — рождает погони и засады.

### GM-4.3 · «Последний рубеж» / battle-royale `[core+data]` — M
Сжимающаяся безопасная зона: узлы на краю периодически становятся «мёртвыми»
(урон/потеря контроля), карта схлопывается к центру. Форсирует столкновения, лечит
«отсиделся в углу». Модуль-таймер + теги колец на карте.

### GM-4.4 · «Осада» (асимметричный attacker/defender) `[core+data]` — L
Одна сторона обороняет объект N времени, другая штурмует. Разные условия победы по
роли (расширяет мульти-исход GM-0.2). Хорошо ложится на командный формат и на
AvA-битву меты (`metagame.md`).

### GM-4.5 · Командные варианты (2v2v2 / фракционные блоки) `[core]` — M
Не отдельный режим, а **модификатор**: любой из режимов выше играется командами
(общий счёт/общая столица/общий пул точек). Опирается на коалиционный статус
(GM-1.2) и логику «пала последняя цель команды».

### GM-4.6 · Кооп против ИИ (PvE «волны») `[core+data]` — L
Игроки против общего ИИ-противника (Рой из `gdd.md` фракций); победа —
пережить/зачистить N волн. Требует ИИ-оркестратора; дальний кандидат, но сильный
онбординг/тренировка перед PvP.

### GM-4.7 · Связка с метой (AvA как режим) `[srv]` — 🔒 Контур 2
«Битва за сектор» из `metagame.md` — это командный режим (вероятно «Осада»/
«Удержание») с result-хуком в мета-казну. Отдельный крупный трек (`metagame.md`),
здесь только фиксируем стык: тот же движок режимов, командная конфигурация.

## Сквозные принципы

- **Data-driven, kernel неизменен.** Режим = `GameModeDef` (данные) + опц. модуль +
  хуки. Новое условие победы не редактирует ядро (`modulesystem.md`).
- **Детерминизм.** Любые таймеры/прогресс режима — в `GameState` и считаются от
  `ctx.now` (seeded, реплеируемо, golden-тесты). Никакого «живого» состояния вне
  симуляции.
- **Server-authority + fail-secure.** Выбор режима консервируется при старте (GDD
  §2); невалидный/несовместимый с картой режим — отклоняется сервером стабильным
  кодом (A10), не «падает».
- **Async / 24-7.** Условия (особенно удержание/тайм-ауты) работают по серверному
  времени и через offline-планировщик — победа не требует, чтобы кто-то был онлайн
  (`gdd.md` §9).
- **Коалиции — явный статус.** Любое «командное» правило (суммарный счёт, общая
  столица) считает только формальный альянс (`diplomacyModule`), не «мы не воюем».
- **Совместимость карт.** `requiredMap` в режиме + валидатор: режим со столицами/
  точками не запускается на карте без них (или карта помечает узлы под режим).

## Зависимости и последовательность

```
GM-0.1 config.mode + реестр        ← каркас, разблокирует всё
GM-0.2 мульти-исход/приоритет      ← нужен асимметричным режимам (осада)
GM-0.3 выбор в лобби               ← MM-1.1/1.2, main-menu §4.1
   │
   ├─ GM-1.* Обычная (score)        ← почти готово; коалиция 🔒 диплом. (CR-3.2/3.3)
   ├─ GM-2.* Захват столицы         ← 🔒 сущность старта/фракций (CR-1.3)
   ├─ GM-3.* Удержание точек        ← теги карты + mode-таймеры
   └─ GM-4.* предложения            ← поверх каркаса, по стоимости
```

**Порядок сборки (предложение):** GM-0.1/0.2 (каркас) → GM-1.1 (оформить обычный) →
GM-2 (захват столицы — самый близкий к существующему `elimination`) → GM-3
(удержание точек — новый mode-таймер) → выбор в лобби GM-0.3 → предложения GM-4 по
стоимости (сначала 4.1 экономическая, потом 4.2/4.3, дальше 4.4–4.7). Коалиционные
варианты подключаются, как только появится модуль дипломатии.

## Открытые вопросы

```
□ Захват столицы: мгновенно при захвате vs «удержать штаб T»; можно ли отбить свою
  столицу назад (реставрация) или выбывание необратимо; несколько столиц у команды.
□ Удержание точек: пауза vs полный сброс прогресса при перехвате; общий пул очков у
  коалиции vs индивидуальный; видно ли соперникам чужой прогресс (туман войны).
□ Мульти-исход: как ранжировать одновременные срабатывания разных условий; тай-брейк.
□ Совместимость режим↔карта: карты курируются под режим или узлы тегируются
  процедурно; кто отвечает за наличие ≥K точек/столиц.
□ Баланс порогов (score/hold-time/волны) — данные, тюнятся плейтестами, не код.
□ Награды по режиму: одинаковая ли мета-отдача за разные режимы (иначе фарм лёгкого).
```
