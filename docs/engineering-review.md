## 0. Что уже сделано хорошо (не трогаем, строим поверх)

| Решение | Почему это сильно |
|---|---|
| Микроядро + модули через шину | Новая механика = новый модуль, kernel не трогаем |
| Детерминизм (seeded sfc32, время — параметр) | Replay боёв, предпросмотр на клиенте, античит |
| Event-based время (`advanceTo`) | Сервер спит между событиями, масштабируется дёшево |
| Fail-secure (Rejection + dead-letter) | Ошибка → безопасный отказ, мир не зависает |
| Data-driven контент (JSON + zod) | Баланс правится без кода, новый юнит = строка в JSON |
| deepClone + чистый редьюсер | Immutability, undo/redo бесплатен, тесты тривиальны |
| Action-layer (начат) | Envelope, идемпотентность, sequence gate |

## 1. ОПТИМИЗАЦИЯ

### 1.1 Ядро (shared-core)

**Текущее состояние:** для 5–10 игроков и ~80 планет — более чем достаточно. Но есть точки,
которые станут горлышком при росте.

| Где | Проблема | Решение | Приоритет |
|---|---|---|---|
| `deepClone` на каждый `runStep` | JSON.parse/stringify на ~80 планет + флоты — O(N) на каждое действие | Structural sharing (Immer-style draft proxy) или copy-on-write по секциям. На старте OK, но при 500+ объектах станет заметно | Низкий (сейчас), Средний (сервер) |
| `isBombarded` | `Object.values(fleets)` на каждую планету, на каждый `time.advanced` в economy и construction | ✅ СДЕЛАНО: `bombardedPlanets(state)` считается O(fleets) один раз за тик, дальше O(1)/планета | ✅ |
| `earliestDue` | Линейный поиск по `scheduled[]` на каждый шаг `advanceTo` | ✅ СДЕЛАНО: `scheduled[]` держится отсортированным по (at,seq) → earliestDue O(1), вставка бинарным поиском | ✅ |
| `shortestPath` (Дейкстра) | Пересчитывается на каждый `fleet.move` | ✅ СДЕЛАНО: `RouteCache` в `movement.ts` мемоизирует пути (ключ включает версию топологии) | ✅ |
| `sideUnits` / `sideDamage` | Линейные проходы, но бой — редкое событие (раз в час), не hot path | Не оптимизировать, бой и так быстрый | — |

**Рекомендация на ближайшие PR:** не трогать deepClone — он работает. Кэш `isBombarded` — да (дёшево, выиграем на каждом `time.advanced`). Дейкстра — кэш матрицы при `createInitialState`.

### 1.2 Сервер (при реализации Этапа 3)

```
Правильная модель: один матч = один serialized actor.

Не делать:
  ✗ tick-loop по всем матчам каждую секунду
  ✗ полный GameState в каждый broadcast
  ✗ параллельная обработка действий одного игрока

Делать:
  ✓ wake-up по ближайшему scheduled event (BullMQ delayed job)
  ✓ advanceTo(now) → applyAction → persist → broadcast (после commit)
  ✓ optimistic locking (version = version + 1)
  ✓ diff-stream вместо full state (sync v2)
  ✓ per-match actor queue (сериализация внутри матча)
```

### 1.3 Клиент (при реализации Этапа 4)

```
Render pipeline:
  server diff → normalized store → visible render list → renderer

Критичные оптимизации:
  ✓ Viewport culling (рисовать только видимое + padding)
  ✓ Static/dynamic layers (звёзды/grid — перерисовка только при zoom/pan)
  ✓ LOD: на дальнем zoom — cluster markers, без подписей
  ✓ Simulation preview — off UI thread (web worker)
  ✓ Throttle panel updates (не чаще 3–5 сек, кроме боя)
  ✓ DPR-aware canvas (retina-чёткость без 4x overhead)
```

## 2. БЕЗОПАСНОСТЬ

### 2.1 Что уже есть (solid foundation)

- Fail-secure в ядре (Rejection + E_INTERNAL catch-all)
- Stable error codes (нет утечки внутренних данных)
- Server-authority как принцип (клиент → intent, сервер → truth)
- Action-layer: envelope validation, authorization, idempotency, sequence gate
- ESLint guards (no Math.random / Date.now в shared-core)
- `pnpm audit` в CI

### 2.2 Критические незакрытые зоны

| Зона | Риск | Что делать | Приоритет |
|---|---|---|---|
| **Туман войны** | Сервер отправит full state → читер видит всю карту | `visibleState(state, playerId)` — фильтрация ПЕРЕД broadcast. Тесты: `JSON.stringify(visible).not.toContain('hidden-fleet-id')` | 🔴 Критический |
| **Race conditions** | Double-spend (два building.construct за 100 metal при балансе 100) | Per-player action queue + DB optimistic lock. В ядре уже serialized, но при нескольких server instances — DB level | 🔴 Критический |
| **Rate limiting** | Спам действий / DDoS на WebSocket | Per-account rate limit (действия/мин), payload size limit, backpressure policy | 🔴 Высокий |
| **JWT auth** | WebSocket без аутентификации | JWT в handshake (проверка ДО апгрейда), refresh rotation | 🔴 Высокий |
| **Broadcast-before-commit** | Сервер отправил update → DB commit упал → клиенты видят призрачное состояние | Outbox pattern: broadcast ТОЛЬКО после commit | 🟠 Средний |
| **Action replay** | Клиент переотправляет чужие действия | Action-layer уже проверяет playerId vs session, но нужен signed envelope | 🟠 Средний |
| **Data integrity** | Подменённый units.json при загрузке | Hash/signature manifest при старте матча, проверка при каждом рестарте | 🟡 Позже |

### 2.3 Рекомендуемый порядок

```
1. visibleState (туман войны) — единственный слой между ядром и клиентом
2. Rate limiting + payload size — дёшево, закрывает целый класс атак
3. JWT в WebSocket handshake — без этого нет авторизации
4. Outbox pattern — integrity broadcasts
5. Per-match action serialization через DB lock — production readiness
```

Фракции, способности героев, тёмные события — всё зависит от этого | `traitModule`: подписан на события (unit.died, planet.captured, schedule), проверяет совпадение trigger, бросает `rng.chance(p)`, применяет effect. ## 5. ВОВЛЕЧЕНИЕ И ЗАТЯГИВАНИЕ

### 5.1 Модель удержания (retention framework)

Void Dominion — асинхронная стратегия на 20–100 дней. Главная проблема жанра — отток
после первых 3 дней. Решение — **четыре петли**, каждая с разным таймфреймом:

```
Петля 1 — Минуты (тактическое удовольствие)
  Зашёл → увидел, что изменилось → отдал приказы → увидел результат
  Ключ: мгновенная обратная связь + предпросмотр ("если атакую — что будет?")
  Реализация: shared-core на клиенте для preview, анимированные результаты

Петля 2 — Часы (операционное планирование)
  Флот летит 6 часов → стройка 12 часов → бой разрешается за 3 раунда
  Ключ: push-уведомления в нужный момент ("флот прибыл, нужно решение")
  Реализация: BullMQ → mobile push через Firebase/APNs

Петля 3 — Дни (стратегическая дуга)
  Экспансия → укрепление → конфликт → кризис → победа/поражение
  Ключ: нарастающая ставка (потерянные часы вложений → мотивация защищать)
  Реализация: scoreboard + угрозы (тёмные события, агрессивные соседи)

Петля 4 — Между сессиями (мета-прогрессия)
  Выиграл/проиграл → новые герои/модули/тактики → следующая сессия сильнее
  Ключ: поражение тоже даёт прогресс (мотивация не бросать)
  Реализация: горизонтальная прогрессия (новые опции, не +5% к стату)
```

### 5.2 Конкретные механизмы вовлечения

| Механизм | Как работает | Влияние на retention |
|---|---|---|
| **Push-уведомления (контекстные)** | "Ваш флот прибыл к Kepler-7. Противник строит верфь на соседнем секторе." Не спам, а decision points | 🔴 Критическое — без них асинхронная игра мертва |
| **Предпросмотр боя** | Перед атакой: "Ваш флот (4 крейсера) vs гарнизон (2 militia) — вероятная победа за 2 раунда, потери: 1 крейсер" | 🔴 Высокое — снижает фрустрацию, повышает agency |
| **Daily briefing** | При заходе: сводка за время оффлайн (что произошло, что требует внимания, что угрожает) | 🔴 Высокое — re-engagement |
| **Социальный клей (альянсы)** | Чат, совместные операции, shared map pins, координированные атаки. "Твой союзник атакует через 2 часа — подготовь флот" | 🔴 Высокое — social obligation loop |
| **Шпионаж / паранойя** | Перехват пингов, фальшивые метки, контрразведка. Рождает истории и обсуждения | 🟠 Среднее (позже) |
| **Тёмные события** | Раз в 6–12 часов: аномалия, астероидный шторм, чума — требует реакции. Разбавляет рутину, создаёт неожиданность | 🟠 Среднее |
| **Достижения / milestones** | "Первый захват", "10 боёв выиграно", "Полная карта разведана". Дают мета-ресурсы | 🟡 Дополнительное |
| **Сезонные ивенты** | Тематические модификаторы, уникальные карты, лимитированные награды | 🟡 Позже |

### 5.3 Anti-churn механизмы

```
Проблема: игрок отстал → чувствует безнадёжность → бросает

Решение 1 — Catch-up механика:
  Проигрывающие получают бонус к производству (через хук economy.production,
  масштаб по разнице в score). Не P2W, а встроенный баланс.

Решение 2 — Переговоры:
  Слабый игрок ценен как союзник (коалиционный порог ниже, GDD §3.3).
  Дипломатия даёт agency даже проигрывающему.

Решение 3 — Межсессионная прогрессия:
  Даже поражение даёт XP и ресурсы (GDD §3.4). Следующая сессия начнётся
  с новыми возможностями.

Решение 4 — Короткие сессии (timeScale ×4):
  30 дней вместо 100. Быстрее к результату, меньше выгорания.
```

## 6. ПЛАН РЕАЛИЗАЦИИ (порядок слоёв)

Каждый слой закрывает отдельный класс проблем и оставляет проект в рабочем состоянии.

### Слой A — Движок трейтов + фракции (гибкость + вовлечение)

```
Новые модули:
  traitModule     — исполнение trigger→effect из events.json
  factionModule   — стартовые лоадауты, пассивные бонусы, уникальные юниты

Новые данные:
  factions.json   — расширить: startUnits, startBuildings, passiveHooks
  events.json     — расширить: больше trigger→effect правил

Тесты: фракция Swarm захватывает планету → infect_planet срабатывает.
```

Почему первым: открывает весь уникальный контент. Без трейтов фракции — просто скин.

### Слой B — Туман войны (безопасность + вовлечение)

```
visibilityModule  — правила видимости (радиус, разведка, память)
visibleState()    — проекция: фильтрует GameState для конкретного игрока

Тесты: hidden fleet отсутствует в JSON visible state;
       разведанная планета помнится после ухода
```

Почему вторым: без fog-of-war мультиплеер нечестен. Это security boundary.

### Слой C — Дипломатия (гибкость + вовлечение)

```
diplomacyModule   — provideCapability('diplomacy', impl)
                  — действия: propose, accept, declare_war
                  — состояние: relations в GameState
                  — буферы по времени (нельзя предать мгновенно)

Тесты: союзные флоты не вступают в бой; предательство требует таймера
```

### Слой D — Генерация карты (красота + гибкость)

```
mapGenerator(seed, config) → planets + links
  — процедурная, детерминизм через seeded RNG
  — параметры: размер, плотность, типы секторов/планет
  — гарантии: связность, баланс стартовых позиций

Тесты: одинаковый seed → одинаковая карта; все планеты достижимы
```

### Слой E — Сервер (Этап 3 — оптимизация + безопасность)

```
Fastify skeleton → PostgreSQL persistence → BullMQ scheduler →
WebSocket sync (v2: diff-stream) → JWT auth → rate limiting →
fog-of-war filtering before broadcast → outbox pattern

Порядок PR внутри слоя (из deep-technical-roadmap):
  E1: Persisted MatchStore (interface + InMemory + PostgreSQL)
  E2: WebSocket reconnect + seq protocol
  E3: Scheduler wake-up skeleton
  E4: Rate limiting + payload size
  E5: JWT + session management
```

### Слой F — Клиент (Этап 4 — красота + вовлечение)

```
React Native + Skia → WebSocket client → shared-core preview →
push notifications → tactical map → planet cards → fleet controls →
daily briefing → alliance chat
```

### Слой G — Мета-игра и закалка

```
Межсессионная прогрессия → корпорации → AvA-битвы → OWASP аудит →
нагрузочное тестирование → production deploy
```
