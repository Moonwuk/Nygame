# Интерфейс управления корпорацией (межсессионный альянс) — дизайн

> Дизайн-документ **экрана управления корпорацией** — «штаба» межсессионного
> альянса внутри мета-шелла. Уточняет `main-menu.md` §4.6 (там корпорация —
> только *витрина/лидерборд*): здесь описан **рабочий кабинет** участника и
> офицеров. Парные доки: `metagame.md` (два контура, что такое корпорация и очки
> влияния/снабжения), `main-menu.md` (мета-шелл, куда встраивается экран),
> `accounts-roadmap.md` (Account/identity), `matchmaking-roadmap.md` (лобби/AvA),
> `gdd.md` (§4 экономика/монетизация, §5 двойная прогрессия, §9 координация
> альянсов). **Этап 4 (клиент) + сервер (мета, Этап 3+).**
>
> ⚠️ Это спецификация направления, **не «строим прямо сейчас»**. Экран жёстко
> зависит от аккаунтов (`AC-0.1`) и мета-слоя сервера (Контур 2 из `metagame.md`),
> которых ещё нет. Строится **последним** в мета-шелле (`main-menu.md` §6).

---

## 1. Что это и где живёт

«Корпорация» = межсессионный альянс из `metagame.md`: постоянное сообщество с
общим чатом, накопленными **очками влияния** и **корп-снабжения**, владением
секторами мета-галактики и общими AvA-битвами. Игрок принадлежит **одной**
корпорации между сессиями; внутри обычной сессии членство ни к чему не обязывает
(члены одной корпорации могут даже воевать между собой — `metagame.md` Контур 1).

Экран управления корпорацией — это **под-раздел мета-шелла**, а не отдельное
приложение. В карте экранов `main-menu.md` §3 корпорация присутствовала лишь как
таб в «Рейтингах» (витрина чужих). Этот документ добавляет **вход в свою
корпорацию** (из шапки/«Требует внимания» или из карточки в рейтинге) →
многотабовый кабинет.

| Слой | Роль |
|---|---|
| `shared-core` (Этап 1) | **никак.** Ядро детерминированно и не знает ни аккаунтов, ни корпораций, ни меню. Членство/влияние — состояние **между** матчами (`metagame.md`). |
| `server` (Этап 3+) | авторитетный источник всех проекций корпорации (профиль, ростер, казна влияния/снабжения, владения, лог) и приёмник интентов (вступить, назначить роль, объявить AvA, потратить снабжение). Server-authority + fail-secure. |
| `client` (Этап 4+) | рендер read-model'ей + отправка интентов; один TS-движок адаптивен под браузер и мобайл. |

## 2. Инвариант: кабинет — вне детерминированного ядра

По аналогии с инвариантом мета-шелла (`main-menu.md` §2) и метрик
(`metrics-roadmap.md`): **ни одно состояние корпорации не живёт в `GameState`.**
Членство, роли, казна влияния, владение секторами, расписание AvA — состояние
между матчами, оно на сервере. Ядро получает нужный срез **на вход** как часть
`GameData`/конфигурации матча (напр. пассивный бонус за владение сектором) и
отдаёт результат **на выход** (очки влияния, начисленные по итогу сессии). Экран:

- **читает** серверные проекции (профиль корпорации, ростер, казна, владения, лог);
- **шлёт интенты** (вступить/выйти, подать заявку, назначить роль, объявить/принять
  AvA, потратить корп-снабжение) — клиент **не авторитетен**;
- любой интент **fail-secure**: отказ `{ok:false, code}` со стабильным кодом (A10),
  без утечки причин наружу (тот же принцип, что «нет fog-лика»).

## 3. Карта экранов (IA)

Вход: шапка мета-шелла (значок своей корпорации) **или** «Требует внимания»
(«AvA через N часов», «новая заявка на вступление», «повышена роль») **или**
карточка в «Рейтинги → Корпорации». Кабинет — многотабовый экран:

```
┌──────────────────────────────────────────────────────────────────┐
│  ШАПКА КОРПОРАЦИИ: [эмблема] Название [TAG] · девиз                 │
│  влияние ⟡ N · снабжение ◈ M · секторов ▦ K · участников ♟ P/cap    │
│  моя роль: <Офицер> · [Пригласить] [Настройки] (по правам)          │
├────────┬──────────┬──────────┬───────────┬──────────┬─────────────┤
│ ОБЗОР   │ УЧАСТНИКИ │  КАЗНА    │  ВЛАДЕНИЯ  │  ВОЙНЫ    │   ЧАТ /     │
│ (сводка,│ (ростер,  │ (влияние/ │ (сектора   │ (AvA:     │   ЛОГ       │
│  лента, │  роли,    │ снабжение,│ мета-карты,│ вызовы,   │ (постоянный │
│  бонусы)│  заявки)  │ история)  │ бонусы)    │ расписа-  │  чат + audit│
│         │           │           │            │  ние,     │  log)       │
│         │           │           │            │  состав)  │             │
└────────┴──────────┴──────────┴───────────┴──────────┴─────────────┘
```

**Нет своей корпорации** → вместо кабинета экран-заглушка: *Создать
корпорацию* (стоимость/условия — открытый вопрос §7) и *Найти/подать заявку*
(поиск по названию/TAG, список открытых наборов). Приглашения приходят в
«Требует внимания».

## 4. По секциям

Формат (как в `main-menu.md` §4): *Назначение · Что показывает · Источник
(сервер) · Интенты (клиент→сервер) · Права · Инварианты/риски.*

### 4.1 Обзор — «пульс корпорации»

- **Назначение.** С одного взгляда: кто мы, как идём, что требует внимания.
- **Что показывает.** Эмблема/название/TAG/девиз; сводные метрики (влияние ⟡,
  снабжение ◈, секторов, участников P/cap, место в рейтинге корпораций);
  **активные пассивные бонусы** за владение секторами (`metagame.md`: напр.
  +5% к добыче до следующего передела); лента событий (новые участники, начатые/
  завершённые AvA, крупные траты снабжения); ближайшая назначенная AvA с таймером.
- **Источник.** Мета-слой (`metagame.md` Контур 2): членство, казна, владения,
  расписание/итоги AvA; рейтинг корпораций (`main-menu.md` §4.6).
- **Интенты.** В основном read-only.
- **Права.** Видят все участники; гости/претенденты — усечённый публичный срез.
- **Инвариант.** Активные бонусы — **витрина** серверного состояния; фактически
  они применяются снапшотом при старте матча (`gdd.md` §5.2), не «на лету».

### 4.2 Участники — ростер, роли, заявки

- **Назначение.** Управление людьми: кто в корпорации, кто чем занимается, приём/
  исключение.
- **Что показывает.** Ростер с presence (онлайн / в матче / оффлайн — как в
  `main-menu.md` §4.3), роль каждого, вклад (внесённые очки влияния — из
  `main-menu.md` §4.2 профиля), дата вступления, последняя активность; входящие
  **заявки на вступление**; исходящие **приглашения**.
- **Источник.** Мета-сервис членства + соц-presence (переиспользует presence из
  `main-menu.md` §4.3).
- **Интенты.** `applyToCorp(corpId)`, `inviteToCorp(userId)`, `accept/declineJoin(reqId)`,
  `assignRole(userId, role)`, `kickMember(userId)`, `leaveCorp()`, `transferOwnership(userId)`.
- **Права.** См. §5.1 (матрица ролей). Исключение/назначение — только офицеры+;
  передача владения — только глава.
- **Инварианты/риски.** Rate-limit на приглашения/заявки (анти-спам, как
  `actionRateMax`); нельзя исключить главу; понижение самого себя как единственного
  главы запрещено (иначе «безголовая» корпорация — fail-secure отказ); presence
  уважает приватность (`SE-10`, «невидимка»).

### 4.3 Казна — очки влияния и снабжения

- **Назначение.** Экономика альянса: сколько накоплено, куда тратим, кто внёс.
- **Что показывает.** Балансы **влияния** ⟡ (валюта объявления AvA — `metagame.md`
  §Контур 2) и **корп-снабжения** ◈ (аренда флагманов/зданий/модулей на AvA);
  история начислений (по итогам сессий участников) и трат (объявления AvA, аренда);
  топ-вкладчики.
- **Источник.** Мета-экономика (`metagame.md` §75–81): начисления — result-хук
  сессии, который сервер пишет в казну; траты — серверные интенты.
- **Интенты.** `spendInfluence(purpose, amount)` (напр. объявить AvA — см. §4.5),
  `allocateSupply(itemId, targetMember)` (выдать корп-снаряжение на битву).
- **Права.** Просмотр — все; траты влияния (объявление AvA) — глава; распределение
  снабжения — офицеры+.
- **Инварианты/риски.** Все балансы **авторитетны на сервере**, клиент только
  отображает; трата — атомарная серверная транзакция (нельзя потратить дважды,
  ср. `E_ALREADY_QUEUED` в прототипе стройки); экономика влияния/снабжения (курсы,
  источники, инфляция) — **открытый вопрос** `metagame.md` §96, здесь не решается.

### 4.4 Владения — сектора мета-галактики

- **Назначение.** «Что мы держим» на большой карте и какие бонусы это даёт.
- **Что показывает.** Список/мини-карта подконтрольных секторов мета-галактики,
  по каждому — пассивный бонус, дата захвата, угрозы (кто может бросить вызов),
  история переделов.
- **Источник.** Мета-галактика (`metagame.md` Контур 2): сектора, владельцы,
  бонусы. Полная мета-карта — отдельный экран (вне этого дока).
- **Интенты.** Переход к «объявить AvA за соседний сектор» (§4.5); просмотр read-only.
- **Права.** Просмотр — все; инициировать войну за сектор — глава.
- **Инварианты/риски.** Мета-карта создаётся **в момент объявления войны**
  (`metagame.md` §37) — до этого «владения» это persistent-состояние сервера, не
  живая сессия; пока Контур 2 не построен — таб деградирует в заглушку
  (развилка `main-menu.md` §4.6).

### 4.5 Войны (AvA) — вызовы, расписание, состав

- **Назначение.** Организация «битвы за сектор»: объявить/принять вызов, собрать
  состав, подготовить снаряжение.
- **Что показывает.** Активные/предстоящие AvA (соперник, оспариваемый сектор,
  дата/таймер), собранный состав участников (кто заявился, роли — адмиралы/
  губернаторы/генералы, `gdd.md` §6.2), доступное к аренде корп-снаряжение
  (флагманы/здания/модули за снабжение), входящие вызовы от других корпораций.
- **Источник.** Оркестрация AvA на сервере (`metagame.md` §36–47,
  `matchmaking-roadmap.md`): AvA — та же детерминированная сессия с командной
  конфигурацией и result-хуком.
- **Интенты.** `declareAvA(targetSectorId)` (тратит влияние, назначает дату,
  создаёт мета-карту), `acceptAvA(challengeId)`, `signUpForAvA(avaId)`,
  `setAvARoster(...)`, `rentCorpGear(itemId)` (тратит снабжение).
- **Права.** Объявление/принятие — глава; сбор состава/аренда — офицеры+; заявка
  на участие — любой участник.
- **Инварианты/риски.** Расписание и честность AvA — server-authority (инвариант
  ядра A01/A08, `game-integrity-roadmap.md`); формат строго командный, внутри-AvA
  дипломатия упрощена (все свои — союзники, `metagame.md` §39); баланс аренды vs
  личного арсенала — **открытый вопрос** `metagame.md` §98.

### 4.6 Чат / Лог — постоянная связь и прозрачность

- **Назначение.** Социальный клей (`metagame.md` §87) + подотчётность офицеров.
- **Что показывает.** Постоянный **корп-чат** (переживает сессии, в отличие от
  сессионного чата коалиции в прототипе); **audit-log** значимых действий (смена
  ролей, приём/исключение, траты казны, объявления AvA) — для доверия и разбора.
- **Источник.** Мета-сервис сообщений + журнал действий сервера.
- **Интенты.** `postCorpMessage(text)`, `pinMessage`, (модерация: `deleteMessage`,
  `muteMember` — офицеры+).
- **Права.** Чтение/письмо — все участники; модерация и просмотр полного audit-log —
  офицеры+.
- **Инварианты/риски.** Rate-limit на сообщения (анти-спам); хранение сообщений —
  минимизация PII (`SE-10`); чат авторитетен на сервере; корп-чат Контура 1 (внутри
  сессии) — **отдельная** сущность (не путать: сессионный чат сбрасывается).

## 5. Роли и права

### 5.1 Матрица ролей (черновик)

Иерархия по возрастанию прав. Точный набор ролей — открытый вопрос (§7).

| Действие | Глава | Офицер | Участник | Претендент/Гость |
|---|:---:|:---:|:---:|:---:|
| Читать обзор/владения/чат | ✓ | ✓ | ✓ | публичный срез |
| Писать в корп-чат | ✓ | ✓ | ✓ | — |
| Заявиться на AvA | ✓ | ✓ | ✓ | — |
| Приглашать / принимать заявки | ✓ | ✓ | — | — |
| Назначать роли (ниже своей) | ✓ | ✓* | — | — |
| Исключать участников | ✓ | ✓* | — | — |
| Распределять снабжение / собирать состав AvA | ✓ | ✓ | — | — |
| Тратить влияние / объявлять AvA | ✓ | — | — | — |
| Настройки корпорации, передача владения, роспуск | ✓ | — | — | — |

`*` офицер не может действовать против главы и других офицеров (только участников) —
защита от «переворота»; конкретика — открытый вопрос.

### 5.2 Fail-secure и авторитет

Каждый интент проверяется **на сервере** против роли отправителя; клиент прячет
недоступные кнопки, но это лишь UX — реальная проверка на сервере (клиент не
доверенный, `architecture.md` §5). Отказ — стабильный код без деталей (A10).

## 6. Сквозные принципы

- **Async-first.** Всё, что требует реакции (заявки, приглашения, «AvA через N
  часов», повышение), поднимается в «Требует внимания» мета-шелла (`main-menu.md`
  §5.1) и в push (§4.4 настроек меню). Никто не обязан быть онлайн в момент AvA.
- **Приватность/безопасность.** Минимизация PII, presence с приватностью,
  rate-limit соц-действий, отсутствие user-enumeration в поиске корпораций/людей
  (`main-menu.md` §5.2, `SE-10`).
- **Честность монетизации.** Кабинет корпорации **не** содержит «денежных» точек,
  дающих боевую мощь: снабжение/влияние зарабатываются игрой, донат из магазина
  (`gdd.md` §4.4) не покупает влияние/снабжение/состав. Иначе богатый альянс
  «покупает» AvA — прямое нарушение инварианта честности.
- **Кросс-платформа/локализация.** Один TS-движок → один адаптивный экран; тексты
  UI через i18n, имена контента — data-driven (`main-menu.md` §5.4).
- **Витрина ≠ источник.** Экран только отражает серверные проекции; бонусы за
  сектора применяются снапшотом при старте матча (`gdd.md` §5.2), а не мгновенно.

## 7. Зависимости, последовательность, открытые вопросы

```
accounts AC-0.1 (сущность Account)      ← ЖЁСТКИЙ блокер (как весь мета-шелл)
server Этап 3 (персистентность/authority)← ЖЁСТКИЙ блокер
metagame Контур 2 (мета-галактика/AvA)   ← нужен для «Владения»/«Войны»
        │
        ├─ Обзор/Участники/Чат  ← минимальный «клуб»: членство + роли + чат (можно раньше AvA)
        ├─ Казна                ← мета-экономика влияния/снабжения (metagame §96)
        ├─ Владения             ← мета-галактика (Контур 2)
        └─ Войны (AvA)          ← оркестрация AvA (последним, нужна мета-карта)
```

**Порядок сборки (предложение).** Экран корпорации строится **после** каркаса
мета-шелла и «Рейтинга корпораций» (`main-menu.md` §6). Внутри самого экрана —
деградация по готовности мета-слоя: сперва «социальный клуб» (Обзор без бонусов +
Участники + Чат) поверх членства и ролей; затем Казна (когда есть экономика
влияния/снабжения); Владения и Войны — последними, вместе с Контуром 2. Пока
подсистема не готова — таб показывает заглушку, а не ломается (принцип
`main-menu.md` §4.6).

**Открытые вопросы (решаем, когда дойдём):**

```
□ Модель ролей: сколько уровней (глава/офицер/участник?), кастомные роли/права,
  защита от «переворота» офицеров, что при уходе/неактивности главы (наследование).
□ Создание корпорации: цена (влияние? премиум-косметика?), лимит участников (cap),
  уникальность названия/TAG, модерация эмблем.
□ Вступление: заявка+подтверждение vs открытый набор vs только по инвайту; можно ли
  состоять в нескольких (по metagame — в одной); кулдаун на смену корпорации.
□ Экономика казны: источники/курсы влияния и снабжения, инфляция (наследует
  metagame §96); можно ли «жертвовать» личные ресурсы в казну и как это чтить.
□ AvA-организация: как собирается состав (лимит слотов, обязательность), правила
  аренды снабжения vs личного арсенала (metagame §98), кто платит цену поражения.
□ Чат/лог: срок хранения сообщений, объём audit-log, модерация и апелляции.
```
