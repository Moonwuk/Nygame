# Contributing — как мы работаем

Короткий рабочий регламент команды. Архитектура и инварианты — в `CLAUDE.md`;
живой снимок состояния — `docs/state.md`; план — `docs/roadmap.md` и
`docs/deep-technical-roadmap.md`.

## TL;DR

1. Ветку — **от `main`**, маленькую, под одну задачу.
2. `pnpm install` → пиши код → **`pnpm run check`** зелёный.
3. PR **в `main`**. Зелёный CI + ≥1 ревью → мердж → ветку удалить.
4. **`main` — единственная точка интеграции.** Не мердж в чужие ветки.

## Окружение

- Node ≥ 20 (CI гоняет на 22). pnpm зафиксирован полем `packageManager` —
  включи `corepack enable`, дальше `pnpm install`.
- **Гейт перед коммитом:** `pnpm run check` = `lint` + `typecheck` + `test`.
- Ещё: `pnpm test`, `pnpm run lint`, `pnpm run typecheck`, `pnpm run format`,
  `pnpm run prototype` (собрать играбельный `prototype/dist/void-dominion.html`).

## Работа через Claude Code (веб / CLI)

Правила те же, что для человека:

- **Base-ветка сессии = `main`.** Одна задача → одна сессия → одна ветка → один PR.
- Веб сам генерит имя ветки (`claude/...`) — это нормально; важно, чтобы PR шёл
  **в `main`**, а не в чужую ветку.
- Не продолжай чужую длинную ветку — начинай от свежего `main`.
- Подключение нового человека: его GitHub-аккаунт получает **Write**-доступ к репо
  (см. ниже) и ставит **Claude GitHub App** на свой GitHub. Среды у всех
  изолированы — общий только этот репозиторий.

## Спец-агенты по требованию

Под узкую тяжёлую задачу (security-ревью ветки, анализ баланса контента,
профилирование рендера, аудит детерминизма) запускай **сфокусированного
суб-агента** встроенным Agent tool Claude Code — с чётким ТЗ и проверяемой целью;
результат сверяй с гейтом и кодом, а не принимай на веру.

Почему так, а не плагин-каталог: мы намеренно **не берём фреймворки вроде
`template-bridge` целиком**. Они ставят «агентов» через неприпиннутый
`npx …@latest` — ровно тот supply-chain-вектор (OWASP A08), от которого защищает
наша SDLC-линия (SHA-пины actions, аудит зависимостей). Идею «специалист по
требованию» берём, но **без сторонних зависимостей и без неприпиннутого
внешнего кода**.

## Git-workflow

- **`main` защищён:** прямой push запрещён; только PR с зелёным CI и ревью.
- Ветки короткоживущие. Растёт задача — дроби на несколько PR.
- Часто подтягивай `main` (`git pull --rebase origin main`), чтобы не расходиться.
- **Не мердж feature-ветку в feature-ветку.** Всё сходится в `main`.
- Конфликт двух PR на один файл: первый мерджится, второй ребейзится на `main`.

## Как делить работу (чтобы не конфликтовать)

Архитектура изолирует — держись границ:

- **Пакеты** (`packages/`): `shared-core` (детерминированное ядро-симуляция),
  `action-layer` (Этап 2 — валидация/идемпотентность), `server` (Этап 3),
  `client` (Этап 4). Бери пакет — минимум пересечений.
- **Новая механика = новый модуль** в `packages/shared-core/src/modules/` +
  данные в `data/*.json`. Модули **не импортируют друг друга**, общаются только
  через шину (события / хуки / реестр возможностей). Двое пишут разные модули, не
  трогая общих файлов.
- **Ядро (`kernel/`) заморожено** — его не редактируют под фичу.
- **Контент** (`data/*.json`) — отдельная зона почти без конфликтов с кодом.

Перед стартом застолби зону в issue/таске, чтобы не пересечься с напарником.

## Инварианты ядра (нарушение = баг, не стиль)

Полностью — в `CLAUDE.md`. Кратко:

1. **Детерминизм:** ни `Math.random`, ни `Date.now` в ядре; seeded RNG; время —
   параметр `ctx.now`.
2. **Чистота/иммутабельность:** `applyAction` не мутирует вход; `GameState` —
   JSON-сериализуем (без классов/Map/Date).
3. **Только через шину:** модули не импортируют друг друга; точки расширения
   деградируют мягко (нет модуля → базовый дефолт).
4. **Fail-secure:** любая ошибка → `{ ok: false, code }`; путь — `h.reject(code)`;
   без утечки деталей наружу.
5. **Server-authority:** клиент шлёт намерение, не состояние.

## Тесты и документация

- Тесты лежат рядом с кодом (`*.test.ts`). Новая логика → тесты (обычно
  «состояние А + действие Б → ожидаем В»). Красный гейт не мерджим.
- **Цикл TDD:** RED (падающий тест на поведение) → GREEN (минимум кода до зелёного
  гейта) → REFACTOR (почистить, гейт держать зелёным) → COMMIT (маленький коммит).
  Гейт — `pnpm run check`. Для тривиальных правок — по здравому смыслу.
- После изменения поведения обнови **`docs/state.md`** (живой снимок). Документацию
  **сверяй с реальностью** (читай исходник, гоняй гейт), не пиши по памяти.

## Pull Request

- Маленький и сфокусированный; в описании — что и зачем. Чек-лист — в шаблоне PR.
- Ревьюера назначает `CODEOWNERS` (по зонам). Не мерджи красный CI.

## Настройка репозитория (владелец, один раз)

1. **Доступ.** Settings → Collaborators → добавить человека с правами **Write**
   (или через организацию/команду). Для Claude он ставит Claude GitHub App у себя.
2. **Branch protection** на `main` (Settings → Branches → Add rule, паттерн `main`):
   - Require a pull request before merging → Require approvals: **1**.
   - Require status checks to pass → отметить `build` и `audit`.
   - Require branches to be up to date before merging.
   - Запретить обход правил и прямой push в `main`.
3. **`CODEOWNERS`.** Заполни GitHub-ник'ами по зонам (см. `.github/CODEOWNERS`),
   когда распределите ответственность.
