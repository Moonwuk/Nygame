# Цепочки команд (offline-приказы) — технический roadmap

> Игрок составляет **цепочку приказов**, а **сервер исполняет её реальными часами,
> пока игрок офлайн** — «выставил и ушёл спать». Прямая реализация открытого вопроса
> «□ Логика оффлайн-игрока» (`architecture.md:547`) и UX-принципа GDD §9 (автоматизация
> + защита оффлайн-игроков). Ложится на уже готовый субстрат: оффлайн-будилку
> (`persistence-accounts-roadmap.md` PA-4.1), `schedule`/`advanceTo`, per-player очередь
> действий (`backlog.md` F5), action-layer (Stage 2). Парные доки: `architecture.md`,
> `modulesystem.md`, `gdd.md` (§1 async, §7.5 принцип «готовые блоки», §9 оффлайн), combat
> (server-resolved §7.1 — прецедент «set-and-forget»). Формат — кирпичики: `id · зона · статус · deps
> · «Готово, когда»`. Зоны: `[core]` · `[srv]` · `[cli]` · `[data]`. Статусы: ⏳ · 🔒(dep).

---

## 0. Зачем, где живёт, чем ограничено (read first)

- **Async-игра наказывает за сон** без оффлайн-автоматики. Цепочка = «выставил приказы →
  сервер исполняет часами → проснулся с результатом».
- **Игрок ОФФЛАЙН → исполнение обязано быть серверным/в-ядре.** Клиент-макрос не годится
  (некому слать приказы — игрок спит). Ложится на оффлайн-будилку (PA-4.1) + `schedule`/`advanceTo`.
- **Детерминизм-инвариант (PA-4.2):** результат за оффлайн = непрерывная игра. Цепочки
  исполняются **детерминированно** — триггеры суть scheduled/доменные события в порядке `(at, seq)`.
- **Над СВОИМИ активами, human-in-the-loop при выставлении.** Контраст с Q11
  (`open-questions-visuals-content-pings.md`): авто-приказ ЧУЖИМ флотам = грифинг-вектор и
  запрещён. Цепочка командует только своим — это безопасно.
- **Философия — ограниченные готовые блоки, НЕ Тьюринг-полный скрипт** (принцип GDD §7.5
  «готовые блоки, не конструктор»). Дёшево, безопасно (нет валидации произвольных «программ»), без бот-гонки и без
  «игра играет сама».

## 1. Строительные блоки

- **Шаг** = отложенный интент — те же действия, что вживую (`fleet.move`, постройка,
  `fleet.orbit`/`assault`/`bombard`, `launch`/`patrol`, …).
- **Последовательность:** шаг N+1 стартует по **завершению** шага N (доменное событие:
  `build.completed` / `fleet.arrived` / `battle.resolved`).
- **Триггеры** (курируемый набор, не произвольные): по прибытии · по достройке · по итогу
  боя · по таймеру/задержке · по порогу ресурса · при атаке.
- **Гварды** (ограниченные предикаты): `if HP < X` · `if враг в радиусе` · `if ресурс ≥ Y`
  — готовый набор, не произвольная булева алгебра.
- **Стоячие приказы** (повтор): патруль, авто-ребилд разрушенного, hold / retreat-if-attacked.

## Фаза 0 · Модель

### CC-0.1 · Схема цепочки + каталог блоков `[core][data]` 🔒(—) — M
**Подзадачи:** цепочка = список `{интент, триггер, guard?}` + **курсор** (текущий шаг) в
`GameState` (JSONB-сериализуемо); zod-схема; **каталог триггеров/гвардов — данные** (как
«готовые блоки»); привязка к владельцу.
**Готово, когда:** цепочка добавляется/парсится; курсор персистится; кривая отвергается.

### CC-0.2 · РЕШЕНИЕ: где исполняется `[docs][core]` 🔒(CC-0.1) — S
**Вопрос:** **in-core модуль** `command-chain` (детерминизм/реплей, reuse `schedule` + шины
+ ownership-гвардов хендлеров) **vs** server-layer оркестратор (ядро чистое, но риск для
инварианта offline=continuous). *Рекомендую in-core* — консистентно с тем, как
construction-очередь/движение/бой уже планируют продолжения в ядре.
**Готово, когда:** зафиксировано с обоснованием; влияет на CC-1.*.

## Фаза 1 · Последовательное исполнение `[core]`

### CC-1.1 · Завершение шага → следующий шаг `[core]` 🔒(CC-0.2) — L
**Подзадачи:** на завершающем доменном событии шага → эмит следующего интента → **валидация
хендлером** (ownership/state, как у живого действия); **fail-secure:** невалидный шаг (флот
уничтожен, цель исчезла) → скип/аборт + запись в `failures` (как dead-letter scheduled-события,
мир не виснет).
**Готово, когда:** двух-/трёхшаговая цепочка проходит сама; невалидный шаг не роняет матч; тест.

## Фаза 2 · Триггеры `[core]`

### CC-2.1 · Курируемые триггеры `[core][data]` 🔒(CC-1.1) — M
**Подзадачи:** on-arrival / on-build-done / on-battle-resolved / at-time(delay) / on-threshold /
on-attacked — через шину/`schedule`, детерминированно. Новый триггер добавляется данными.
**Готово, когда:** каждый триггер запускает свой шаг; воспроизводимо; offline=continuous-тест.

## Фаза 3 · Гварды и стоячие приказы `[core]`

### CC-3.1 · Ограниченные предикаты + повтор `[core][data]` 🔒(CC-2.1) — M
**Подзадачи:** готовые гварды (HP / враг-в-радиусе / ресурс); стоячие приказы (авто-retreat,
hold, авто-ребилд, патруль); повтор до отмены. Без произвольных условий.
**Готово, когда:** «отступить если HP<X» и «ребилд если разрушено» работают оффлайн; тест.

## Фаза 4 · Уведомления (проснулся → что случилось) `[srv][cli]`

### CC-4.1 · Итог/инцидент цепочки → лента + push `[srv][cli]` 🔒(CC-1.1, CP7) — M
**Подзадачи:** цепочка завершена / прервана / заблокирована → лента «требует внимания»
(`main-menu.md` §3/§5.1) + push (`cross-platform-roadmap.md` CP7). Защита оффлайн-игрока —
узнать об инциденте, а не проснуться у разбитого корыта молча.
**Готово, когда:** прерывание цепочки порождает уведомление; видно в ленте при заходе.

## Фаза 5 · Лимиты, анти-абуз, UX `[srv][cli]`

### CC-5.1 · Лимиты + конфликт-резолюция `[core][srv]` 🔒(CC-1.1) — M
**Подзадачи:** макс длина / число активных цепочек; rate-limit на создание; **bounded блоки**
(не Тьюринг); **приоритеты:** живой приказ игрока перебивает цепочку; событие врага → через
guard; одно действие на актив за тик (стыкуется с F5-очередью).
**Готово, когда:** лимиты работают; живой приказ переопределяет цепочку; абуз-паттерны отбиты.

### CC-5.2 · Конструктор цепочек (клиент) `[cli]` 🔒(CC-2.1, CP-рендер) — L
**Подзадачи:** собрать цепочку из блоков (список/флоучарт), превью «что произойдёт», статус
исполнения; редактирование/отмена онлайн.
**Готово, когда:** цепочку можно собрать мышью и выставить перед выходом.

## Открытые вопросы

```
□ Где живёт: in-core (рекоменд.) vs server-layer — фиксируется в CC-0.2.
□ Выразительность MVP: какой набор триггеров/гвардов шипим (ограниченный, не скрипт).
□ Политика отказа шага: скип vs аборт всей цепочки vs пауза-и-уведомить — настройка цепочки?
□ Конфликты: цепочка vs живой приказ vs действие врага — точные приоритеты.
□ Единый каталог стоячих приказов (патруль эскадрилий и т.п.) — общая система?
□ Шпионаж: перехватываются ли цепочки разведкой (как пинги/приказы, metagame §шпионаж)?
□ Не превращается ли в «игра играет сама» — баланс автоматики vs вовлечённости (плейтест).
```

## Инварианты (нарушение = баг)

- **Детерминизм:** триггеры = scheduled/доменные события в `(at, seq)`; **offline = непрерывная
  игра** (PA-4.2); никаких `Date.now`/`Math.random` в ядре.
- **Оффлайн-first:** исполняется будилкой 24/7 (PA-4.1) **без клиента**.
- **Server-authority + per-step валидация** (Stage 2 / ownership-гварды хендлеров); **fail-secure**
  — невалидный шаг → dead-letter, не краш.
- **Bounded** (GDD §7.5): готовые блоки, не бот-язык; **над своими активами** (не Q11-грифинг).
- **Только модуль + данные; кернел не трогаем.**

## Последовательность

`CC-0.1` → **`CC-0.2` (решение in-core, до Фазы 1)** → `CC-1.1` → `CC-2.1` → `CC-3.1` →
`CC-4.1` → `CC-5.*`. Минимально играбельно уже на CC-1.1 + пара триггеров (CC-2.1):
«построй→построй→пошли флот» оффлайн.
