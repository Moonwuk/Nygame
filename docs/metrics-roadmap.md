# Metrics roadmap — что и зачем измеряем

Метрики Void Dominion: что снимаем на плейтестах, как ловим десинки/баги,
производительность, баланс и (позже) прод-нагрузку. Брики в формате репозитория
(`фаза · id · зона · «Готово, когда»`). Парные доки: `multiplayer.md` (как тестим),
`game-integrity-roadmap.md` (честность/анти-чит), `operations-roadmap.md` (прод-ops),
`deep-technical-roadmap.md` (ядро), `engineering-review.md` (инженерия).

Зоны: `[core]` shared-core · `[server]` сервер/комната · `[cli]` клиент/рендер ·
`[tools]` харнесы/скрипты · `[docs]` процесс.

## Инвариант: метрики — ТОЛЬКО наблюдение

Сбор метрик не имеет права влиять на симуляцию. Источник данных — уже эмитируемые
доменные события (шина), серверные `ActionReceipt`/`StateDelta` и чистые хелперы
(`hashState`, `diffState`) **вне редьюсера**. Никаких `Date.now()`/счётчиков внутри
`applyAction`/`advanceTo` — иначе ломается детерминизм и реплеи (см. `CLAUDE.md`).
Поэтому все метрики либо считаются на сервере/клиенте поверх событий, либо
постфактум по логу.

## Категории и KPI

### A · Плейтест и связь — приоритет сейчас (ручные тесты 2-х) `[server][cli]`

| Метрика | Что показывает | Как снять | Цель |
|---|---|---|---|
| Connect success rate | надёжность входа | сервер: upgrade-попытки vs выданные `welcome` | > 95% |
| Time-to-connect | задержка входа | клиент: Connect → первый кадр мира | < 3 c |
| Lobby wait | сколько ждали второго | сервер: первый join → старт часов (есть лобби-гейт) | информативно |
| Disconnects / session | стабильность сокета | сервер: `close`-события на матч | < 1 |
| Reconnect success | восстановление | сервер: повторный `welcome` после `close` | > 90% |
| Session length | вовлечённость | сервер: join → последнее действие | информативно |
| RTT (latency) | «отзывчивость» | `ping`/`pong` (есть в протоколе; клиент считает RTT в `MultiplayerClient.onPong`, оверлей показывает FPS · RTT · sync) | p95 < 150 мс |
| Errors / crashes | баги | `pageerror` (клиент) + пойманные throw (сервер) → лог | 0 |

### B · Детерминизм и корректность — инвариант ядра `[core][server]`

| Метрика | Что показывает | Как снять | Цель |
|---|---|---|---|
| **Desync rate** | расхождение клиент↔сервер | `hashState`: сервер кладёт хеш в `delta`, клиент сверяет со своим | **0** |
| Action reject rate + коды | сколько и почему отклонено | сервер: `ActionReceipt {ok, code}` агрегатом | < 5%, без `E_INTERNAL` |
| Replay fidelity | стабильность RNG/реплеев | golden-тесты (`rng.test`) + хеш прогона | green |
| Dedup / idempotency | повтор не применяется дважды | receipts по `action.id` (уже есть) | 100% |
| Fog leak = 0 | сервер не шлёт скрытое | проверка: в `delta` игрока нет чужих скрытых сущностей | 0 утечек |

### C · Производительность — клиент `[cli][tools]`

| Метрика | Что показывает | Как снять | Цель |
|---|---|---|---|

| Память | утечки за матч | `performance.memory` сэмплом | стабильна |

### D · Производительность — сервер `[server][tools]`

| Метрика | Что показывает | Как снять | Цель |

## Роадмап

### M0 · Снятие данных с ручных тестов `[server][cli][docs]` — ✅ реализовано
**Цель:** после каждого живого теста 2-х есть машинные + ручные данные.
**Подзадачи:** (1) ✅ комната принимает `observe`-колбэк (наблюдение, не влияет на
симуляцию) — `RoomObservation` для join/leave/lobby/action/end; прото-сервер
(`prototype/netserver.ts`) пишет per-match **JSONL** в `playtest-logs/` и печатает
сводку по `SIGINT`. (2) ✅ опция `emitStateHash` кладёт `hashState(view)` в каждый
снапшот; клиент (`MultiplayerClient.onPong` + сверка хеша) считает RTT и desync,
прото-оверлей показывает **FPS · RTT · sync ✓ / desync ✗**. (3) ✅
`docs/playtest-template.md` — чеклист после теста (связь, баги, что путало, фан 1–5).
**Готово, когда:** один тест → JSONL + заполненный чеклист. *(механика готова и
покрыта тестами `matchRoom.test.ts`; ждёт первого живого прогона на двоих.)*

### M1 · Инструментирование ядра (детерминизм-safe) `[core][server]` 🔒(M0)
**Цель:** измеримые корректность и нагрузка без влияния на симуляцию.
**Подзадачи:** (1) read-only **metrics-наблюдатель** поверх шины — счётчики действий
по типам, реджектов, боёв, захватов; (2) **desync-детектор** через `hashState`
(сервер → хеш в `delta`, клиент сверяет, mismatch → лог + ресинк); (3) тайминги
`submitAction`/`broadcast` + размер `delta` в структурный лог.
**Готово, когда:** desync измеряется (цель 0), реджекты и тайминги в логах.

### M2 · Клиентская перф-телеметрия + перф-гейт `[cli][tools]` 🔒(M0)
**Цель:** регресс FPS виден до мерджа.
**Подзадачи:** формализовать headless FPS-харнес в проверку (idle/pan/zoom ≥ цель,
нон-блок отчёт в CI); клиент шлёт лёгкий перф-сэмпл (fps/rtt/mem) в лог во время
сетевой игры.
**Готово, когда:** падение FPS на прототипе ловится автоматически.

### M3 · Сборщик телеметрии + отчёт по матчу `[server][tools]` 🔒(M1)
**Цель:** из сырья — читаемый отчёт.
**Подзадачи:** коллектор событий → стор (старт: append-only JSONL/SQLite; позже
Postgres со Stage 3, см. `persistence-roadmap.md`); скрипт `metrics`: лог матча →
отчёт (длина, микс действий, реджекты, контроль во времени, desync, латентности).
**Готово, когда:** один матч → один отчёт одной командой.

### M4 · Балансная аналитика + self-play `[tools][core]` 🔒(M3)
**Цель:** баланс мерить без людей и быстро.
**Подзадачи:** headless **self-play харнес** (AI vs AI, seeded, детерминированно,
много матчей); агрегаты — win rate по фракции/слоту, длина, snowball index, usage.
**Готово, когда:** 1000 сид-матчей → таблица баланса за минуты.

### M5 · Ops-дашборды и алерты `[server]` 🔒(Stage 3)
**Цель:** прод под наблюдением.
**Подзадачи:** реал-тайм метрики (матчи/игроки, msg/s, p95 latency, error rate,
CPU/mem, cost) + алерты на **desync rate**, error rate, recovery time. Стыкуется с
`operations-roadmap.md`.
**Готово, когда:** дашборд живой + алерт срабатывает на искусственный десинк.

## Критический путь

`M0 → M1` — measureable correctness + базовая нагрузка (этого хватает, чтобы тесты
2-х были не «вслепую»). Дальше параллельно: `M2` (перф-гейт), `M3` (отчёты),
`M4` (баланс через self-play). `M5` подключается со Stage-3 сервером.

> Старт: **M0 сделан** — observe-JSONL, хеш состояния в снапшоте, клиентские RTT/desync
> и чеклист поверх кирпичей (`hashState`, `diffState`, `ActionReceipt`, `ping/pong`,
> `fpsEma`, соак/рестарт-харнесы). Следующий шаг — **M1**: metrics-наблюдатель,
> desync-детектор, тайминги/размер `delta` в структурный лог.
