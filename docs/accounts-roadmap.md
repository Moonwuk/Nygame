## Текущее состояние (факт)

Первый слой аккаунтов **реализован** (SE-1.x, `packages/server`): таблица `users`
(id/login/pass_hash, логин уникален без учёта регистра; Memory+Postgres), пароли —
scrypt из `node:crypto` (параметры вшиты в хеш → повышаемы без ломки записей),
`POST /auth/register` / `POST /auth/login` (uniform-отказ 401 без утечки существования
аккаунта + decoy-hash от timing-раскрытия, per-IP rate-limit), сессионный JWT
(`typ session+jwt`, отдельная audience — невзаимозаменяем с join-токеном), и
create/join `/matches` требуют `Authorization: Bearer` — ник места = логин сессии,
`accountId` штампуется в join-токен. `Player` в `GameState` по-прежнему игрок
**внутри матча**; связь с учёткой — через место (seat) и `accountId` в токене.

## Решение (decision record) — ОБНОВЛЕНО

**Исходное решение:** внешний OIDC-провайдер, свою auth для прода не пишем; свой
логин/пароль — «только если позже припрёт». **Обновление (владелец, при переходе к
самостоятельному деплою):** припёрло — свой логин/пароль реализован как первый слой
(см. «Текущее состояние»): scrypt вместо Argon2 (OWASP-приемлемый memory-hard KDF из
`node:crypto` — ноль новых зависимостей и нативных сборок в distroless-образе; параметры
вшиты в запись, так что миграция на Argon2/выше по стоимости не ломает существующие хеши).
**OIDC остаётся целевым для публичного прода** (AC-1.1: соц-вход, меньше поверхности) —
password-слой тогда станет одним из провайдеров, а не заменой. Refresh/ревокация сессий —
AC-0.2 (сейчас: долгоживущая сессия — `SESSION_TTL_SEC` деф. 7 дней — без refresh/ревокации).

## Фаза 0 · Модель аккаунта и хранение

### AC-0.1 · Сущность Account + связь с Player `[srv][data]` 🔒(F2) — M
**Цель:** отделить учётку от внутриматчевого игрока.
**Подзадачи:** таблица аккаунтов (`id`, внешние identity-ссылки, профиль, таймстемпы, статус); связь `account ↔ player(match)`; минимизация PII (см. SE-10).
**Готово, когда:** аккаунт создаётся/читается; матч-игрок ссылается на аккаунт.

### AC-0.2 · Сессии и refresh `[srv]` 🔒(AC-0.1) — M
**Цель:** управляемые сессии под WS/HTTP.
**Подзадачи:** выпуск сессии после входа; короткий access + refresh; ревокация/logout-all; привязка устройства (опц.); хранение сессий (Redis).
**Готово, когда:** сессия выдаётся/обновляется/отзывается; истёкшая отвергается.

### AC-0.3 · Уровень/опыт аккаунта `[srv][data]` 🔒(AC-0.1, AC-1.3) — M
**Цель:** мета-прогрессия аккаунта (доступ/престиж, **не сила**); гейт создания корпорации.
**Подзадачи:** накопление XP на финализации сессии (`RoomObservation 'end'` → коллектор `M3`, observation-only); уровень = производная проекция `level = max{L: threshold[L] ≤ totalXp}`, пересчёт на чтении; пороги + per-level карта открытий — **версионируемый server/meta конфиг** (не `data/*.json` ядра), схема fail-closed на `vertical`-теги; гейт `createCorp` берёт личность из JWT-subject (AC-1.3), не из payload; `accountLevel`/`accountXp` в `GameData`/снапшот матча **не попадают** (код-страж); GDPR export/delete (AC-2.3), PII-min (целое XP + производный уровень).
**Готово, когда:** XP копится на сервере; уровень пересчитывается из XP; `createCorp` гейтится fail-secure (`E_LEVEL_TOO_LOW`); страж подтверждает отсутствие `level`/`xp` в ядре. Дизайн — `account-level.md`.

## Фаза 1 · Провайдер входа

### AC-1.1 · OAuth/OIDC-вход `[srv][sec]` 🔒(AC-0.1) — L
**Цель:** вход без хранения паролей у нас.
**Подзадачи:** OIDC-flow (Google/Apple); валидация id-token (issuer/aud/exp, пиннинг алгоритма); создание/линковка аккаунта по subject; CSRF/state в flow.
**Готово, когда:** пользователь входит через провайдера; аккаунт создаётся/линкуется безопасно.

### AC-1.2 · Email magic-link (без пароля) `[srv][sec]` 🔒(AC-0.1) — M
**Цель:** запасной вход без пароля.
**Подзадачи:** одноразовая ссылка с TTL/nonce; транзакционная почта (см. operations); анти-перебор/rate-limit.
**Готово, когда:** вход по ссылке работает; ссылка одноразова и истекает.

### AC-1.3 · JWT для WS-рукопожатия `[srv][sec]` 🔒(AC-0.2) — M
**Цель:** связать сессию с авторитетным WS-входом. **(= SE-0.1 / бирка F7.)**
**Подзадачи:** выпуск JWT из сессии; валидация **до** upgrade (issuer/aud/exp/alg-pin); привязка к матчу/игроку; ротация ключей (SE-2).
**Готово, когда:** WS принимает только валидный JWT, привязанный к аккаунту/матчу.

## Фаза 2 · Профиль, привязки, безопасность учётки

### AC-2.1 · Привязка нескольких identity к аккаунту `[srv]` 🔒(AC-1.1) — M
**Подзадачи:** линковка Google+Apple+email к одной учётке; разлинковка; защита от захвата через линковку.
**Готово, когда:** один аккаунт держит несколько способов входа безопасно.

### AC-2.2 · Восстановление доступа и защита от захвата `[srv][sec]` 🔒(AC-1.2) — M
**Подзадачи:** безопасное восстановление (без user-enumeration, тайминг-безопасно); уведомление о входе/смене; опц. 2FA (ASVS L3).
**Готово, когда:** восстановление безопасно; подозрительный вход уведомляется.

### AC-2.3 · Приватность аккаунта (GDPR-база) `[srv][docs]` 🔒(AC-0.1) — M
**Подзадачи:** экспорт/удаление данных по запросу; политика хранения; согласия. **(= SE-10.1.)**
**Готово, когда:** есть механика удаления/экспорта по запросу.

## Последовательность

- **Критический путь:** AC-0.1 (модель) → AC-0.2 (сессии) → AC-1.1 (OIDC) → AC-1.3 (JWT/F7) —
  это разблокирует публичный WS-вход и кросс-платформу.
- **Magic-link/восстановление** — после транзакционной почты (operations).
- **Приватность** — до публичного запуска.
