# Архитектурный документ проекта

> Космическая grand strategy в сеттинге тёмного космоса.
> Рабочее название: **Void Dominion** _(можно заменить)_.
> Документ-опора для разработки. Живой — дополняется по мере работы.

---

## 1. Обзор проекта

Мобильная **real-time** grand strategy с массовым мультиплеером в духе игр Bytro Labs (Iron Order, Call of War, Supremacy 1914), но в сеттинге **тёмного космоса**: чужие цивилизации, выживание, атмосфера ужаса.

Real-time здесь — как у Bytro: мир идёт в **непрерывном реальном времени и круглосуточно**, даже когда игрок оффлайн. Игра **не пошаговая**. «Асинхронность» — это про паттерн участия (зашёл, отдал приказы на часы вперёд, вышел; флот летит реальные часы, стройка идёт реальные часы), а не про дискретные ходы.

Игры Bytro используются **только как вдохновение** — жанр, темп, структура геймплея. Весь код, арт, тексты и механики оригинальные. Жанр и игровые механики не защищены авторским правом; защищены лишь конкретные названия, логотипы, арт и дословный код. Подход «духовного наследника» — стандартная практика инди-разработки.

### Ключевые параметры

| Параметр                    | Значение                                                |
| --------------------------- | ------------------------------------------------------- |
| Игроков в сессии            | 5–10                                                    |
| Длительность матча          | 20–30 дней                                              |
| Провинций (секторов/планет) | ~60–80                                                  |
| Движение флотов             | 2–24 часа                                               |
| Строительство               | 6–48 часов                                              |
| Платформа                   | Mobile first (iOS + Android)                            |
| Темп                        | Асинхронный: зашёл, принял решения, вернулся через часы |
| Победа                      | Контроль 60% секторов или уничтожение всех соперников   |

### Главная цель архитектуры

**Гибкое, расширяемое ядро.** Ключевое наблюдение: у Bytro хрупкое ядро (технический долг с 2009–2012 гг.), поэтому новые механики не добавляются годами — только юниты с новыми цифрами. Наша цель — ядро, в которое легко добавлять механики, юниты, ресурсы и даже целые сеттинги без переписывания логики.

## 2. Философия ядра: Data-Driven

**Главный принцип: игровые объекты описываются данными, а не кодом.**

Ядро никогда не знает о конкретных юнитах, фракциях или ресурсах. Оно лишь выполняет правила. Всё конкретное живёт в данных (JSON).

```
┌─────────────────────────────────────┐
│         Game Engine (ядро)          │
│  движение · бой · экономика ·       │
│  события · дипломатия               │
└──────────────┬──────────────────────┘
               │ читает
┌──────────────▼──────────────────────┐
│         Game Data (данные)          │
│  units.json     factions.json       │
│  buildings.json events.json         │
│  abilities.json map_config.json     │
└─────────────────────────────────────┘
```

### Три уровня гибкости

**1. Юниты — полностью через данные.** Новый юнит = новая запись в JSON, без изменения кода.

```json
{
  "infected_cruiser": {
    "faction": "swarm",
    "stats": { "attack": 12, "defense": 4, "speed": 6 },
    "traits": ["infect_planet", "no_food_cost"],
    "abilities": ["spread_biomass"],
    "cost": { "biomass": 200, "dark_matter": 50 }
  }
}
```

**2. Механики — через систему трейтов и эффектов.** Вместо хардкода «фракция X делает особое действие» — универсальная система триггер→эффект. Любой юнит или здание можно наделить любым трейтом.

```
trait: "infect_planet"
  trigger: "planet_captured"
  effect:  "add_trait"
  params:  { trait: "infected" }
```

**3. Ресурсы — динамический список.** Не хардкодим «золото, железо». Ядро работает с абстрактным словарём ресурсов; вся экономика подхватывает новые автоматически.

```json
"resources": ["credits", "metal", "biomass", "dark_matter", "artifacts"]
```

### Что это даёт

- Новый юнит → минуты, не дни
- Новая фракция → описать в JSON, не переписывать логику
- Балансировка → правка цифр в файле, без изменения кода и деплоя
- Новый сеттинг → заменить JSON, ядро то же самое

## 3. Технологический стек

> **Актуальное решение по стеку — [`tech-stack.md`](tech-stack.md)** (decision record).
> Этот раздел — исходный набросок этапа проектирования; расхождения решаются в пользу
> `tech-stack.md`: планировщик — встроенная будилка → **pg-boss/Postgres** (без
> Redis/BullMQ), транспорт — сырой `ws`, HTTP-поверхность рядом — тонкий Fastify.

### Ключевое решение: TypeScript везде

Ядро-симуляция (бой, экономика, события) пишется **один раз** как отдельный пакет `shared-core` и используется и на сервере, и на клиенте:

- **Сервер** — авторитетная версия (правда хранится здесь)
- **Клиент** — та же логика для мгновенного предпросмотра («если атакую — что будет?»)

Это то, чего нет у Bytro: их логика заперта на сервере, клиент «тупой». У нас клиент показывает прогноз боя мгновенно, а балансить можно в одном месте. Для маленькой команды общее ядро — суперсила.

### Стек по слоям

| Слой                | Технология                      | Почему                                                                                               |
| ------------------- | ------------------------------- | ---------------------------------------------------------------------------------------------------- |
| Клиент              | PWA-first веб (TWA + Capacitor) | Один TS/web-артефакт на все платформы; карта Canvas2D→WebGL→WebGPU — см. `cross-platform-roadmap.md` |
| Ядро                | TypeScript-пакет `shared-core`  | Симуляция, общая для клиента и сервера                                                               |
| Сервер              | Node.js + TypeScript (Fastify)  | Быстрая разработка, тот же язык, хороший WebSocket                                                   |
| Реалтайм            | WebSocket (ws)                  | Сервер пушит только diff — что изменилось                                                            |
| Планировщик событий | BullMQ (на Redis)               | «Флот прибудет через 6 ч» = отложенная задача                                                        |
| База                | PostgreSQL + JSONB              | Состояние как JSONB — гибкая схема без миграций                                                      |
| Кэш + очередь       | Redis                           | Очередь событий + pub/sub для рассылки                                                               |
| Хостинг             | Fly.io или Railway              | Дёшево, managed PostgreSQL + Redis, приватная сеть из коробки                                        |

Всё open-source. Стоимость на старте — только хостинг (~$10–30/мес).

```
   (состояние)   (очередь + pub/sub)
```

## 4. Архитектура ядра

Три решения, которые определяют всё ядро и которые нужно зафиксировать **до** первой строки кода.

### 4.1 Модель времени — event-based

Игра **real-time** (континуальное реальное время, как у Bytro), не пошаговая. Event-based — это её **эффективная реализация**: состояние мира определено в любой момент времени, но сервер не тикает ежесекундно, а планирует события на будущее и вычисляет состояние по запросу через `advanceTo(state, now)`.

Симуляция строится не на тиках «каждую секунду», а на событиях во времени. Сервер **спит** между событиями — поэтому даже дешёвый VPS тянет много матчей.

```
Игрок отправил флот → летит 6 часов
  h.schedule(now + 6ч, 'fleet.arrival', payload)   // событие в state.scheduled
  Драйвер часов спит до ближайшего события (msUntilNextEvent → tick)
  Через 6 часов → advanceTo исполняет событие → обработали прибытие → разослали diff
```

(Истина расписания живёт в `state.scheduled`; будилка — in-process `clockDriver`,
durable-эволюция на мульти-процессе — pg-boss, см. `tech-stack.md`.)

Пограничные случаи (решены в ядре):

- Порядок при совпадении времени двух событий — детерминированный `(at, seq)`
- Производство ресурсов — не тики, а начисление по формуле за span через событие
  `time.advanced { from, to }`
- Накопление за часы оффлайна — `advanceTo` догоняет мир к `now` (ограниченными
  порциями, `partial`-продолжение)

### 4.2 Детерминизм

Ядро — **чистая функция**: одинаковый вход → всегда одинаковый выход. Это даёт replay боёв, точное предсказание на клиенте, лёгкое воспроизведение багов и античит-пересчёт.

```
❌ if (Math.random() > 0.5)   // невоспроизводимо
✅ if (rng.next() > 0.5)       // seeded RNG, воспроизводимо
```

Правила:

- Никакого `Math.random()` внутри ядра — только seeded random (генератор со своим зерном)
- Никакого `Date.now()` внутри логики — время передаётся как параметр

### 4.3 Структура состояния

`GameState` хранится в PostgreSQL как JSONB — гибкая схема без миграций при добавлении полей.

```json
{
  "planet_id": "kepler_7",
  "owner": "player_3",
  "resources": { "metal": 340, "biomass": 120 },
  "buildings": ["mine_t2", "shipyard"],
  "garrison": [{ "unit": "cruiser", "count": 4 }],
  "traits": ["infected", "rich_deposits"]
}
```

Добавил поле в игру — не нужна миграция БД. Просто кладёшь новое поле в JSON.

### 4.4 Версионирование данных

Версия правил привязана к матчу. Матч помнит, на какой версии данных он создан. Изменил баланс — идущие матчи доигрываются на старых правилах, новые подхватывают новые. Это часть и гибкости, и целостности (OWASP A08).

## 5. Безопасность: античит

### Золотое правило мультиплеера

> **Клиент НИКОГДА ничего не решает. Клиент только ПРОСИТ. Сервер решает.**

```
┌─────────────────────────────────────────────┐
│  Всё что приходит от клиента = НАМЕРЕНИЕ      │
│  Сервер ПРОВЕРЯЕТ и РЕШАЕТ                    │
│  Клиент получает только РЕЗУЛЬТАТ + видимое   │
└─────────────────────────────────────────────┘
```

Клиент может рисовать что угодно — реальное состояние только на сервере. Расчёт боя на клиенте — лишь предпросмотр; результат клиента сервером игнорируется полностью.

### Виды читерства и защита

| Угроза                             | Защита                                                                                |
| ---------------------------------- | ------------------------------------------------------------------------------------- |
| Подмена данных («999999 кредитов») | Сервер хранит ресурсы у себя; клиент шлёт только намерение, не «сколько у меня есть»  |
| Невозможные действия               | Сервер валидирует: флот его? цель в радиусе? время полёта верное? флот не занят?      |
| Спам запросов / боты               | Rate limiting на аккаунт                                                              |
| Чтение чужого (туман войны)        | Сервер фильтрует данные **перед** отправкой — на телефон не приходит то, что невидимо |
| Перемотка времени на телефоне      | Серверные часы; BullMQ-задачи срабатывают по времени сервера                          |

**Ключевой момент про туман войны:** сервер отправляет каждому игроку только то, что он должен видеть. Не «шлём всё и прячем на клиенте» (взлом клиента → видно всю карту), а «физически не отправляем невидимое» — взламывать нечего, данных просто нет на устройстве.

### Идемпотентность действий

Телефон — ненадёжная сеть. Нажал «атака», связь моргнула, повтор. Чтобы действие не выполнилось дважды:

- Каждое действие имеет ID: `session:player:sequence` (например `kepler:alice:47`)
- Уникальность нужна **локально** (сессия+игрок), не глобально
- Дедупликация нужна **коротко** (минуты), не вечно → хранение обработанных ID в Redis с TTL ~10 мин
- В любой момент в памяти только недавние действия по активным сессиям — тысячи записей, не миллионы

Реальная нагрузка при нашем темпе: ~20–50 действий на игрока в день, даже 500 активных сессий → ~3 действия/сек пиково. Миллионы действий за всю историю — это архив (PostgreSQL, партиции по сессии, удаление завершённых), а не рабочая нагрузка.

## 6. OWASP Top 10 2025

Полный разбор применительно к проекту. Актуальная редакция 2025 года; две новые категории — A03 (Supply Chain) и A10 (Exceptional Conditions).

### Приоритеты для нашей игры

| Категория                                     | Важность | Почему                                         |
| --------------------------------------------- | -------- | ---------------------------------------------- |
| **A01** Broken Access Control                 | 🔴🔴🔴   | Это и есть античит. Ядро всего                 |
| **A06** Insecure Design                       | 🔴🔴🔴   | Проектирование до кода — то, что делаем сейчас |
| **A02** Security Misconfiguration             | 🔴🔴     | Открытый Redis/PostgreSQL = мгновенный взлом   |
| **A10** Mishandling of Exceptional Conditions | 🔴🔴     | 30-дневные матчи + читеры                      |
| **A05** Injection                             | 🔴       | Валидация входа критична                       |
| **A03** Software Supply Chain                 | 🔴       | Много npm, маленькая команда                   |
| **A07** Authentication Failures               | 🔴       | Ценные аккаунты                                |
| **A04** Cryptographic Failures                | 🟠       | Важно, но стандартно                           |
| **A08** Integrity Failures                    | 🟠       | Версионирование уже частично заложено          |
| **A09** Logging & Alerting                    | 🟠       | Бонус: ловит читеров                           |

### A01 — Broken Access Control (для нас #1)

Остаётся на первом месте в OWASP — практически все приложения имеют ту или иную форму провала контроля доступа. Для нас это центральная угроза и одновременно античит.

- Проверка прав на **каждое** действие, на сервере, по принципу «запрещено по умолчанию»: флот его? планета его? видит ли цель?
- SSRF теперь влит в A01 — если появятся исходящие запросы (вебхуки, аватары по URL), только allowlist разрешённых адресов

### A02 — Security Misconfiguration (поднялась на #2)

Реальные взломы всё чаще из-за криво настроенной инфраструктуры, не из-за дыр в коде.

- Дефолтные пароли/ключи убраны (PostgreSQL, Redis)
- Redis и PostgreSQL **не** торчат в интернет (только приватная сеть)
- CORS строго (только домен клиента)
- Debug-режим выключен в проде
- Заголовки безопасности (HSTS и пр.); лишние порты закрыты
- Сообщения об ошибках не verbose (пересекается с A10)

### A03 — Software Supply Chain Failures (новая)

Много npm-пакетов (Fastify, BullMQ, ws…) = наш ландшафт риска.

- Lock-файлы зафиксированы в репозитории (`pnpm-lock.yaml` + `mobile/package-lock.json`)
- SCA через **OSV-Scanner** в CI (`npm audit`/`pnpm audit` сняты — npm закрыл audit-эндпоинты)
- Минимум зависимостей; проверка пакетов перед добавлением
- Dependabot/Renovate для обновлений безопасности
- Осторожно с typosquatting (fastfiy вместо fastify)

### A04 — Cryptographic Failures

- Пароли — только хеш (Argon2 или bcrypt), никогда открытым текстом
- WSS/HTTPS везде
- JWT подписаны надёжным секретом (не «secret123»)
- Секреты в переменных окружения, не в коде/репозитории
- Платёжные данные (при монетизации) — через провайдера, не храним сами

### A05 — Injection

- SQL — только параметризованные запросы (ORM/query builder, не конкатенация строк)
- Валидация **всего** входа от клиента через zod-схемы
- Имена игроков/чат — экранирование (защита от XSS)
- JSONB — валидировать структуру перед записью

### A06 — Insecure Design (этим занимаемся сейчас)

Сам факт проектирования безопасности до кода закрывает A06.

- Threat modeling — какие угрозы, как защищаемся
- Server-authority заложен в фундамент
- Rate limiting как часть дизайна
- Принцип наименьших привилегий

### A07 — Authentication Failures

- Надёжные требования к паролю
- Защита от брутфорса (лимит попыток входа)
- Безопасное восстановление пароля (непредсказуемые токены)
- Сессии с разумным сроком + refresh (JWT)
- 2FA опционально для ценных аккаунтов
- Одинаковый ответ на «нет юзера» и «неверный пароль» (защита от перебора аккаунтов)

### A08 — Software or Data Integrity Failures

- Обновления игры/данных из доверенного источника
- Проверка целостности конфигов (units.json не подменён)
- CI/CD pipeline защищён (кто может деплоить)
- Версионирование данных (уже заложено)
- Безопасная сериализация/десериализация

### A09 — Security Logging & Alerting Failures

Акцент сместился с пассивной записи на проактивное **оповещение**.

- Логировать события безопасности (вход, отклонённые действия, аномалии)
- **Не** логировать чувствительное (пароли, токены)
- Алерты на подозрительное: всплеск отклонённых действий = атака/чит
- Логи защищены от подделки, достаточно контекста для расследования
- Двойная польза: логи безопасности **и** обнаружение читеров

### A10 — Mishandling of Exceptional Conditions

Как приложение обрабатывает ошибки, исключения, таймауты и пограничные случаи. Три блока критичны для нас:

**1. Fail-secure, а не fail-open.** Любая ошибка в проверке = действие отклоняется, не пропускается. Сомнение — в пользу запрета.

```
❌ catch { выполнить(action) }   // КАТАСТРОФА
✅ catch { отклонить(action); залогировать(error) }
```

Если валидация боя выбросила исключение — бой не происходит, флот на месте, событие в лог.

**2. Не сливать данные через ошибки** (CWE-209). Клиент получает только код ошибки; детали — в серверные логи. Особенно важно: ошибка не должна раскрыть, что в невидимом секторе.

```
Клиенту: { "error": "ACTION_REJECTED", "code": "E_INVALID_MOVE" }
В лог (не клиенту): полный стек, контекст, ID игрока
```

**3. Race conditions в проверках безопасности.** Прямо про нашу игру — двойная трата:

```
Два действия одновременно, по 100 кредитов, у игрока ровно 100:
❌ оба проходят проверку → 2 корабля за 100 (чит)
✅ действия одного игрока обрабатываются ПОСЛЕДОВАТЕЛЬНО (очередь per-player)
```

BullMQ умеет concurrency per-key — это закрывает и race conditions, и идемпотентность.

## 7. Отказоустойчивость

Устойчивость 20–30 дневного матча (часть A10).

| Сценарий                  | Решение                                                                                                          |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| Сервер упал посреди матча | Состояние в PostgreSQL (источник правды); BullMQ-задачи в Redis переживают рестарт → ни одно событие не потеряно |
| Событие не обработалось   | BullMQ retry с backoff; после N попыток → dead-letter queue + алерт. «Флот завис» не происходит молча            |
| Таймаут внешней операции  | Запрос к БД завис → таймаут → fail-secure (отклонить, не зависнуть). Игрок видит «повторите», не вечную загрузку |
| Дубль при retry           | Каждое событие идемпотентно по ID → retry не дублирует списание/начисление                                       |

```
┌────────────────────────────────────────────────┐
│  Когда что-то идёт не так — система ЗАПРЕЩАЕТ,   │
│  МОЛЧИТ для клиента, ЛОГИРУЕТ для себя,          │
│  и НЕ ТЕРЯЕТ состояние.                          │
│  Сомнение = отказ. Ошибка = безопасный отказ.   │
└────────────────────────────────────────────────┘
```

## 8. Сетевая инфраструктура и безопасность

### Полный сетевой стек

```
┌─────────────────────────────────────────────────┐
│  Телефон (React Native)                          │
│  только WSS, только видимые данные               │
└───────────────────────┬─────────────────────────┘
                        │ WSS (TLS 1.3)
┌───────────────────────▼─────────────────────────┐
│  CDN / Edge (Cloudflare)                         │
│  DDoS-защита, WAF, TLS termination, скрытие IP   │
└───────────────────────┬─────────────────────────┘
                        │ только трафик с Cloudflare
┌───────────────────────▼─────────────────────────┐
│  Reverse Proxy (Caddy / Nginx)                   │
│  маршрутизация, TLS, health checks               │
└───────────────────────┬─────────────────────────┘
                        │ приватная сеть
┌───────────────────────▼─────────────────────────┐
│  Game Server (Node.js + Fastify)                 │
│  изолирован, БЕЗ публичного IP                   │
└───────┬───────────────────────────┬─────────────┘
        │ приватная сеть            │ приватная сеть
┌───────▼────────┐         ┌─────────▼────────┐
│  PostgreSQL    │         │  Redis           │
│  без публичного│         │  без публичного   │
│  доступа       │         │  доступа          │
└────────────────┘         └──────────────────┘
```

**Золотое правило:** база и Redis никогда не видны из интернета. Только game server обращается к ним по приватной сети. Закрывает огромный класс атак (A02).

### Стек по слоям

| Слой           | Технология               | Зачем                                                   |
| -------------- | ------------------------ | ------------------------------------------------------- |
| Edge / DDoS    | Cloudflare               | DDoS-защита, WAF, скрытие реального IP                  |
| TLS            | TLS 1.3                  | Шифрование, современный быстрый протокол                |
| Reverse proxy  | Caddy или Nginx          | TLS termination, маршрутизация, авто-сертификаты        |
| Транспорт игры | WSS                      | Реалтайм-пуш diff'ов, зашифрован                        |
| Приватная сеть | VPC / private networking | Изоляция БД и Redis (managed-платформы дают из коробки) |
| Секреты        | Doppler / env-переменные | Ключи не в коде                                         |

### Сетевые угрозы и защита

- **DDoS** — Cloudflare поглощает на edge; реальный IP скрыт. Критично: прямые подключения мимо прокси запрещены файрволом (иначе утечка IP → удар напрямую)
- **MITM** — TLS 1.3 везде; опционально certificate pinning на клиенте
- **Атака на WebSocket** — аутентификация в handshake (JWT), авторизация на каждое сообщение, валидация каждого сообщения (zod), rate limit, лимит размера payload
- **Исчерпание ресурсов** — лимит соединений на аккаунт, таймаут простоя, backpressure, heartbeat (ping/pong → закрытие мёртвых соединений)
- **Внутреннее перемещение** — сегментация подсетей, наименьшие привилегии, нет SSH с публичного интернета (только VPN/bastion)

### WebSocket-безопасность (главный транспорт)

- Аутентификация в handshake: JWT проверяется **до** апгрейда до WS
- Авторизация на каждое сообщение (не «залогинился = всё можно»)
- Валидация каждого сообщения (zod); кривой формат → закрыть соединение
- Rate limit и лимит размера сообщения
- Graceful reconnect: переподключение с тем же JWT; идемпотентность спасает от дублей
- Heartbeat: детект мёртвых соединений → освобождение ресурсов

### Прагматичный старт

```
Cloudflare (бесплатный тариф) — DDoS + скрытие IP + TLS
Fly.io / Railway — managed, приватная сеть из коробки
БД и Redis — managed, в приватной сети
Caddy — авто-TLS

→ закрывает ~90% сетевых угроз почти без усилий, ~$10–30/мес
```

### Модель синхронизации состояния (async grand-strategy)

Жанр определяет подход. У нас Bytro-style **async**: приказы исполняются реальными
часами, партии идут неделями, на карте — десятки-сотни флотов, двигаются медленно.
Поэтому тезис «сеть и данные важнее рендера» для нас **верный**, но техники быстрых
RTS берём выборочно — иначе недели уйдут на решение несуществующих проблем.

**Что делаем:**

- **Server-authority.** Клиент шлёт намерение (`Action`), сервер считает и рассылает результат. Инвариант ядра + Stage 2 `action-layer` (валидация/авторизация/идемпотентность).
- **Дельты.** Сервер шлёт только изменения — entity-level патч (изменённые/удалённые сущности по коллекциям + изменённые top-level поля). Полный снапшот — только при входе и ресинке. Кодек — в `shared-core` (`diffState`/`applyDelta`), `applyDelta(prev, diffState(prev,next))` ≡ `next`. **✅ сделано.** Юнит «в покое» = ноль байт.
- **AOI / фильтр видимости.** Дельта диффится против `visibleState(playerId)` — игрок получает только то, что видит. **⏳ ждёт тумана войны (backlog A1).**
- **Хранение.** Redis — горячее состояние (живая логика), Postgres — холодное (история/бэкап); сохранения **событийные/throttled** (по ключевым событиям вроде захвата провинции, не каждую секунду). **⏳ Stage 3 (F2/F3).**
- **Безопасность транспорта.** JWT в WS-handshake, валидация + rate-limit + лимит payload на каждое сообщение (см. выше; F7).

**Чего НЕ делаем — и почему** (фиксируем, чтобы никто не реализовывал технику не из нашего жанра):

| Техника                                               | Почему не для нас                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ----------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Client-prediction + rollback reconciliation           | Для мгновенного движения в шутере/RTS. Флот летит реальные часы — предсказывать нечего, лага движения нет.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Фиксированный тик 20–50 мс / lockstep                 | У нас событийная модель (`advanceTo` к `now` + `scheduled`), а не покадровая симуляция.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| Instanced WebGL / LOD / GPU-culling / atlas           | Для тысяч юнитов в кадре. У нас десятки блипов на canvas, offscreen уже отсекается, DOM кэшируется.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ECS + object pools + Float32Array + Web Workers (A\*) | Оптимизация под нагрузку, которой нет: маршрут — раз в час, Дейкстра по ~30 узлам, уже кэшируется (`RouteCache`).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Бинарный протокол (ArrayBuffer)                       | Экономит при тысячах юнитов в тике 20 Hz. У нас событий мало и редко — JSON (+gzip) хватит надолго. Поздняя микрооптимизация.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Шардирование мира                                     | Только при реальных 10k+ игроков; сейчас преждевременно.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| Integer fixed-point для детерминизма                  | Пока не нужен: детерминизм через seeded RNG (sfc32, golden) + время `ctx.now`, а float-расхождения **удержаны** тем, что ядро держится в бит-точном подмножестве IEEE-754 (`+ − × ÷ √ min max floor ceil` + целочисленные операции), запрещённом ESLint-ом от трансцендентных/`Math.hypot` в `shared-core/src/**`. Единственная иррациональная операция живого пути — дистанция (`state/route.ts`, `Math.sqrt`, бит-точна **по спецификации**: IEEE-754 предписывает корректное округление `√`, а ECMA-262 не относит `Math.sqrt` к implementation-approximated). Остаточный риск ловит desync-детектор `hashState` (опц. `emitStateHash` на сервере). То есть «не расхождений нет by design», а **риск удержан подмножеством**; fixed-point — резерв на случай, если понадобятся трансцендентные. |

## 9. Оптимизация: влияние безопасности

**Вывод: безопасность почти не влияет на производительность, а местами помогает.**

### Где стоимости практически нет

Access control, fail-secure, zod-валидация, параметризованные запросы — наносекунды/микросекунды на действие. При темпе ~3 действия/сек на пике это неизмеримо мало.

### Где стоимость есть, но управляемая

- Хеширование паролей (Argon2, ~100мс) — только при логине, не в игровом цикле
- TLS (~1–5% CPU) — на уровне прокси (TLS termination), ядра не касается
- Логирование — асинхронное, не блокирует обработку

### Где безопасность помогает оптимизации

- **Server-authority + туман войны** → клиент получает только diff и только видимое = **меньше** трафика и нагрузки на батарею
- **Rate limiting** → защита от спама = защита от перегрузки сервера
- **Fail-secure + таймауты** → зависшие операции отклоняются, ресурсы освобождаются

### Нагрузка на телефон

- Карта — рендерить только видимую область (culling), статичные слои кэшировать, анимации минимальны
- Обновлять UI только при реальном изменении данных, не чаще раза в 3–5 сек
- Подгружать секторы по мере скролла (как Google Maps), не всю карту сразу
- 2D flat вместо 3D/изометрии для мобайла

## Сводка ключевых принципов

1. **Data-driven ядро** — объекты в JSON, ядро не знает о конкретике. Гибкость превыше всего.
2. **TypeScript везде** — `shared-core` один раз, на клиенте и сервере.
3. **Server-authority** — клиент просит, сервер решает. Основа античита.
4. **Event-based время** — сервер спит между событиями (BullMQ). Дёшево масштабируется.
5. **Детерминизм** — ядро как чистая функция. Replay, предсказание, античит.
6. **Fail-secure** — ошибка = безопасный отказ, молчание для клиента, лог для себя.
7. **Изоляция инфраструктуры** — БД и Redis в приватной сети, сервер за Cloudflare, TLS везде.
8. **Безопасность ≠ тормоза** — наша архитектура снижает нагрузку на сеть и батарею.
