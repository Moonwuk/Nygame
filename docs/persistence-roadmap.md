# Персистентность и планировщик — технический roadmap

> **Блок 3/7** из `tech-research.md` (§3 персистентность, §4 планировщик). Хранение
> `GameState` (Postgres+JSONB), снапшот+event-log (реплей/аудит из детерминированного ядра),
> миграции сейв-формата, долгогоризонтный планировщик (Redis/BullMQ или pg-boss), бэкапы/восстановление.
> Хардненинг БД (роли/RLS/шифрование) — `secure-environment-roadmap.md` (SE-3/4). Формат —
> кирпичики. Расширяет **F2/F3** (`backlog.md`). Статусы: ✅ · ⏳ · 🔒(dep).

## Текущее состояние (факт)

Всё **in-memory**: `MatchRoom` хранит `GameState` и receipts в памяти; отложенные события
двигаются только на действие/sync (`scheduled` в состоянии, без внешней будилки). Рестарт теряет
матч (🔴-пробел). `GameState` — JSON-сериализуемый (готов к JSONB), версия данных пинится per-match.

## Зависимости

`server-roadmap.md` (SV-0.1 скелет, match-actor) · `core-roadmap.md` (CR-0.1 миграторы формата,

## Фаза 0 · Хранение состояния (Postgres + JSONB)

### PE-0.1 · Схема и load/save `GameState` `[srv]` 🔒(F1) — L
**Цель:** матч переживает рестарт. **Бирка F2.**
**Подзадачи:** таблица матчей (`id`, `data_version`, `seq`, `state JSONB`, таймстемпы); **JSONB-дисциплина** — запрашиваемые/индексируемые поля (статус, игроки, версия, время хода) в **нормализованные колонки**, JSONB только для opaque-состояния (не свалка — иначе full-scan убьёт перф); load/save через match-actor; оптимистическая блокировка по `seq` (анти-конкурентная запись); индексы по статусу/игрокам.
**Готово, когда:** матч сохраняется и точно восстанавливается после рестарта; конкурентная запись отбивается.

### PE-0.2 · Стор квитанций (идемпотентность) `[srv][act]` 🔒(PE-0.1) — M
**Цель:** receipts переживают рестарт. **Бирка E2.**
**Подзадачи:** интерфейс receipt-стора (вместо in-memory) → Redis (горячее, TTL ~10мин) или Postgres; дедуп по `actionId` across рестартов.
**Готово, когда:** повтор действия после рестарта не выполняется дважды.

### PE-0.3 · Hot/cold: Redis горячее, Postgres холодное `[srv]` 🔒(PE-0.1) — M
**Подзадачи:** активные матчи — горячее состояние в Redis; периодический/по-событию сброс снапшота в Postgres; eviction неактивных; согласовать с match-actor lifecycle.
**Готово, когда:** активный матч обслуживается из горячего слоя; холодный — из Postgres.

## Фаза 1 · Снапшот + event-log (реплей/аудит)

### PE-1.1 · Журнал действий + снапшоты `[srv]` 🔒(PE-0.1) — L
**Цель:** детерминированный реплей/аудит «даром» (ядро это позволяет).
**Подзадачи:** append-only лог действий на матч (упорядочен по `seq`); периодические снапшоты для быстрой загрузки (снапшот + хвост лога = текущее состояние); компактизация старого лога.
**Готово, когда:** состояние восстанавливается из (снапшот + лог); реплей (CR-0.2) читает лог.

### PE-1.2 · Миграция сейв-формата в хранилище `[srv][data]` 🔒(PE-0.1,CR-0.1) — M
**Цель:** апдейт правил не ломает хранимые матчи. **(Парно CR-0.1.)**
**Подзадачи:** при загрузке применять миграторы формата N→N+1 (логика в ядре, CR-0.1); хранить `data_version`; ленивая миграция/доигрывание на старой версии; тест round-trip.
**Готово, когда:** хранимый матч версии N грузится после апдейта без потери.

## Фаза 2 · Планировщик отложенных событий

### PE-2.1 · Будилка по `scheduled`-событиям `[srv]` 🔒(F1) — L
**Цель:** «флот прибудет через N ч» срабатывает даже когда все оффлайн. **Бирка F3.**
**Подзадачи:** **pg-boss по умолчанию** (очередь на самом Postgres — минус одна инфра-зависимость; BullMQ/Redis берём только если Redis уже нужен для горячего состояния/кэша) с задачами по серверному времени; пробуждение match-actor на due-событие; **idempotent** обработка; dead-letter упавших (мир не зависает). **(Решение зафиксировано — `sprint-1.md`.)**
**Готово, когда:** отложенное событие исполняется в срок без активности игроков; падение — в dead-letter.

### PE-2.2 · Корректный offline-catch-up `[srv][core]` 🔒(PE-2.1) — M
**Цель:** доход/события за оффлайн считаются **формулой за период**, не тиками (инвариант ядра).
**Подзадачи:** при пробуждении/действии `advanceTo` догоняет часы одним непрерывным `time.advanced`; сверка с реплеем (детерминизм); защита от runaway-расписания (есть guard).
**Готово, когда:** 8-часовой оффлайн даёт тот же результат, что непрерывная игра (тест).

## Фаза 3 · Бэкапы и восстановление

### PE-3.1 · Зашифрованные бэкапы + PITR + учение restore `[srv][sec]` 🔒(PE-0.1) — M
**Цель:** не потерять прогресс. **(Парно SE-3.4.)**
**Подзадачи:** автоматические шифрованные бэкапы Postgres; Point-In-Time-Recovery; **регулярно тестируемый** restore; RTO/RPO; бэкап/AOF для Redis-горячего (или допускаем потерю горячего → восстановление из Postgres-снапшота).
**Готово, когда:** restore проверен на учении; определены RTO/RPO.

### PE-3.2 · Переживание рестарта посреди матча `[srv]` 🔒(PE-0.1,PE-2.1) — M
**Подзадачи:** состояние в Postgres/Redis, события в планировщике; убийство сервера → восстановление часов и матча без потери (= SE-9.1).
**Готово, когда:** kill посреди матча → восстановление без потери прогресса (учение).
