# DevSecOps-пайплайн — зачем эти сканеры и как с ними жить

Учебно-боевой слой безопасности в CI (GitHub Actions, `.github/workflows/security.yml`;
мигрирован с GitLab — см. `audit-2026-06-27.md`). Цель — **shift-left**:
ловить уязвимости в пайплайне на каждый push/MR, а не в проде. Этот проект —
полигон: TS-монорепо сейчас, сервер/Docker/API (Этап 3) — впереди.

## Карта классов сканеров (что, зачем, OWASP)

| Класс                | Что анализирует             | Что ловит                                                                         | OWASP 2021 | Инструмент                        |
| -------------------- | --------------------------- | --------------------------------------------------------------------------------- | ---------- | --------------------------------- |
| **SAST**             | исходный код (статически)   | инъекции, eval/Function, prototype pollution, path traversal, ReDoS, XSS-паттерны | A03, A04   | Semgrep                           |
| **SCA**              | манифест/lock зависимостей  | известные CVE в чужом коде (supply chain)                                         | A06        | pnpm audit, osv-scanner           |
| **Secret detection** | код + git-история           | утёкшие токены/ключи/пароли                                                       | A02, A07   | Gitleaks                          |
| **IaC/misconfig**    | Dockerfile/compose/k8s/yaml | небезопасные дефолты, открытые порты, root                                        | A05        | Trivy config                      |
| **Container scan**   | образ (ОС + libs)           | CVE в базовом образе и слоях                                                      | A06        | Trivy image _(когда будет образ)_ |
| **DAST**             | ЖИВОЙ запущенный API        | сломанная authz, инъекции в рантайме, заголовки                                   | A01, A03   | OWASP ZAP _(Этап 3)_              |
| **SBOM**             | граф зависимостей           | «затронуты ли мы новым CVE?» + provenance                                         | A08        | Syft (CycloneDX)                  |

**Почему именно набор, а не один сканер:** у каждого класса своя слепая зона.
SAST видит твой код, но не чужие CVE. SCA видит CVE в депах, но не логические дыры.
Secret-скан не лезет в логику. DAST видит рантайм, но не покрывает мёртвые ветки.
**Defense-in-depth** = слои с осознанным перекрытием (поэтому Trivy fs дублирует
часть SCA/секретов другим движком — это норма, а не дубль ради дубля).

## Почему открытые инструменты, а не «магия GitLab»

Управляемые сканеры GitLab (`Security/*.gitlab-ci.yml`) удобны, но:

- полноценный UX (Security Dashboard, виджет в MR, vulnerability management) —
  **только Ultimate**; на Free джобы прогоняются, но без дашборда;
- это чёрный ящик. Для **развития навыков** полезнее гонять Semgrep/Gitleaks/Trivy/
  OSV напрямую: видно правила, форматы (SARIF), коды возврата, тюнинг.

В пайплайне оставлен `include:`-блок управляемых шаблонов как альтернатива.

## Gating: ratcheting, а не «сломать билд в день 1»

Включишь всё блокирующим — получишь красный пайплайн с сотней находок и команду,
которая отключит сканеры. Поэтому:

1. **Стоп-краны** (план): секреты (Gitleaks) и high/critical в зависимостях
   (pnpm audit). Их мало, они однозначны.
2. **Триаж → baseline → ужесточение.** Разобрал находки, ложные подавил (см. ниже),
   реальные починил — переводишь сканер в блокирующий. Планка едет вверх (ratchet),
   назад не падает.

**Фактическое состояние (SEC-1 выполнен, 2026-07-10):** оттриаженные сканеры —
**Semgrep, Gitleaks, OSV-Scanner, Trivy fs, Trivy image** — **блокирующие**: их
gate-шаг валит джобу при ЛЮБОМ ненулевом выходе сканера (находка ИЛИ сбой самого
скана — fail-secure: сканер, который не отработал, ничего не сертифицирует).
Артефакты/сентинелы выгружаются до gate, поэтому сводный отчёт собирается всегда.
CodeQL, TruffleHog, zizmor, Scorecard и джоба `check` — пока информационные
(`continue-on-error`); блокирующий `pnpm audit` бежит в основном гейте `ci.yml`.

**Запись триажа (baseline, 2026-07-10):** по коду и зависимостям находок — **ноль**.
Gitleaks v8.18.4 (версия из CI) локально по чистому дереву — 0 утечек; `pnpm audit` —
0 уязвимостей; CI-прогоны пиненных образов (Semgrep p/typescript+p/javascript+
p/security-audit, OSV по обоим lock-файлам, Trivy fs vuln+secret+misconfig) —
0 находок, сканы подтверждены сентинелами. Отдельная история — **Trivy image**: гейт
сразу вскрыл, что сборка образа молча падала с бампа node 22→26 (#106, corepack
выпилен из Node ≥25) и образ вообще не сканировался (fail-open). После починки
Dockerfile оживший скан нашёл 12 CVE базовой ОС рантайма (distroless
nodejs22-debian13: libc6 + zlib1g), все **без доступного фикса** и без runtime-пути
в Node-only сервере — приняты как документированный риск в `.trivyignore` (ревью
при бампе digest); прежние bookworm-записи (perl/ncurses) удалены как неприменимые
к distroless. `.semgrepignore` / `.gitleaks.toml` НЕ созданы намеренно: подавлять
нечего, а пустой файл подменил бы дефолтные списки игнора сканеров. Появится ложная
находка — создавай файл по правилам ниже.

## Триаж и подавление ложных (важный навык)

- **Severity-порог:** начинай с HIGH/CRITICAL, шум среднего/низкого — позже.
- **Semgrep:** инлайн `// nosemgrep: rule-id` с обоснованием, или `.semgrepignore`.
- **Gitleaks:** `.gitleaks.toml` (allowlist по regex/пути/коммиту) для тестовых
  фикстур и заведомо публичных значений.
- **Trivy:** `.trivyignore` (по CVE-id) с датой ревью и причиной.
- **osv-scanner:** конфиг с `[[IgnoredVulns]]` + срок.
- Правило: **каждое подавление — с комментарием «почему» и датой**. Подавление без
  обоснования = технический долг безопасности.
- SARIF-артефакты у всех джоб → можно открыть в любом SARIF-вьюере или залить в
  GitHub Code Scanning / DefectDojo для агрегации.

## Дорожная карта зрелости (бери как кирпичики)

1. **Сейчас:** SAST + SCA + секреты + SBOM + Trivy fs на каждый MR.
2. **Container scan:** появился `Dockerfile` сервера → `trivy image` на собранный образ.
3. **DAST:** поднялся API (Этап 3) → ZAP baseline против `packages/server`.
4. **Supply-chain integrity (A08):** подпись образов/артефактов `cosign`, provenance
   по **SLSA**, проверка подписи на деплое.
5. **Pinning:** образы сканеров и базовые образы — по `sha256`-дайджесту, не по тегу.
6. **Custom Semgrep-правила** под инварианты ядра (напр. запрет `Math.random`/
   `Date.now` в `shared-core` — детерминизм как security-граница против рассинхрона).

## Безопасность самого пайплайна (часто забывают)

CI — это привилегированная среда исполнения и **сама поверхность атаки**:

- **Least privilege:** GitLab CI токены/переменные — минимально нужные, маскированные
  (masked) и защищённые (protected, только защищённые ветки/теги).
- **OIDC вместо долгоживущих секретов** для доступа к облаку (короткие токены).
- **Protected branches:** в `main` только через MR + зелёный пайплайн + ревью
  (см. `CONTRIBUTING.md`).
- **Не эхай секреты** в логи; артефакты со сканов не должны утаскивать исходники.
- **Пинуй версии** сторонних `include:`/образов; чужой шаблон в пайплайне = чужой код
  с твоими правами.

## Запуск

Пайплайн стартует автоматически на push/MR в GitLab. Локально те же сканеры:

```bash
docker run --rm -v "$PWD:/src" semgrep/semgrep semgrep scan --config=p/security-audit /src
docker run --rm -v "$PWD:/src" zricethezav/gitleaks detect --source=/src --redact
docker run --rm -v "$PWD:/src" ghcr.io/google/osv-scanner --lockfile=/src/pnpm-lock.yaml
docker run --rm -v "$PWD:/src" aquasec/trivy fs --scanners vuln,secret,misconfig /src
docker run --rm -v "$PWD:/src" anchore/syft /src -o cyclonedx-json
```
