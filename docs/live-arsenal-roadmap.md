# Живой арсенал: мид-сессионный анлок модулей · технический роадмап

> Фича по запросу владельца: приобрёл/заточил модуль в мете (за валюту, в т.ч.
> донатную) → он **сразу открывает постройку НОВЫХ кораблей с этим модулем в уже
> идущей сессии** у себя в мире. **Существующие корабли не трогает, бой в момент
> покупки не меняет** — сила проходит через обычную петлю «построй → довези до фронта»
> (ресурсы + время). Слой — **server/meta + одна точечная правка авторизации
> постройки**; детерминированное ядро не меняется.
>
> **Важно про рамку:** это НЕ решение «продавать ли силу» — оно уже принято проектом
> (`economy-roadmap.md` дизайн-лок «P2W с гайками», `main-menu.md` §5.3). Здесь только
> **как** приобретённая сила доходит до живого матча, детерминизм-safe. Реально новое —
> ослабление **снапшота-на-старте** для билд-каталога (см. §1). Спутники:
> `economy-roadmap.md` (валюты Суверены/Варранты, 4 пути к силе, заточка EC-2), 
> `arsenal-roadmap.md` (ARS-2 стор владения, ARS-3 снапшот в матч), `gdd.md` §2
> (консервация) / §5.2 (снапшот-на-старте), `account-level.md` (гейты).

**Вывод:** цепочка «донат → Суверены → Варранты → приобрёл/заточил модуль → лёг в
`ArsenalStore`» — уже в дорожной карте (EC-* + ARS-2). Не хватает **одного звена**:
чтобы это владение читалось живьём при постройке в идущем матче, а не по стартовому
снапшоту.

## 1. Ключевое решение (какой инвариант ослабляем)

Сейчас (ARS-3 / GDD §5.2 / main-menu §5.2): **что можно построить, заморожено снапшотом
на старте матча** — «матч не меняется, если игрок качнулся в середине».

Эта фича **осознанно ослабляет снапшот-на-старте — ТОЛЬКО для билд-каталога**
(«что я МОГУ построить»). Всё остальное сохраняется:

- ✅ **Существующие корабли неизменны** — no-refit держится (GDD §2 консервация «деки
  построенного»): купленный модуль не меняет статы уже летящих кораблей.
- ✅ **Детерминизм / реплеи / гибернация / хеш-сверка целы** — потому что постройка
  остаётся обычным залогированным действием, статы берутся из общих данных, а проверка
  владения живёт на серверном гейте (вне ядра, как авторизация сессии).
- ✅ **P2W-политика не меняется** — она уже принята; эта фича лишь доводит её до живого
  матча по F2P-справедливым правилам (§6).

Ослабляется ровно: **«билд-каталог заморожен на старте» → «билд-каталог живой»** (растёт,
когда аккаунт приобретает модуль). Это единственная семантическая правка.

## 2. Цепочка «донат → сила → фронт» (end-to-end)

```
реал ─► Суверены ◆ ─► (обмен) Варранты ⌖ ─► аукцион/заточка модуля (EC-2/EC-3)
                                                     │  (или F2P: RNG-дроп по месту / крафт)
                                                     ▼
                                          модуль лёг в ArsenalStore (ARS-2)
                                                     ▼
                            LARS-1: unit.build{modules} авторизуется против ЖИВОГО арсенала
                                                     ▼
                     строишь НОВЫЙ корабль с модулем в идущей сессии (ресурсы + время)
                                                     ▼
                         везёшь на фронт (время в пути) ─► усиление в бою
```

Мгновенного «докупил → вывернул бой» нет by design: сила гейтится **постройкой +
логистикой**, а не падает в драку.

## LARS-0 · Решение: живой билд-каталог `[docs]` 🔒(ARS-0)

Зафиксировать ДО кода (резолюцией здесь же):

1. **Область ослабления.** Живым становится ТОЛЬКО билд-каталог; построенное — нет
   (no-refit). _Зафиксировано выше §1._
2. **Источник истины владения при постройке** — `ArsenalStore` (ARS-2), читается на
   **серверном гейте** при допуске действия, НЕ из состояния матча. _Рекомендация:
   принять._
3. **Паритет F2P (гайка).** Тот же живой-билд-путь работает для **заработанных** модулей
   (RNG-дроп/крафт), не только купленных — иначе это привилегия платящих, а не механика.
   _Рекомендация: обязательно._
4. **Тайминг применения** (баланс §6): только на своих мирах; опц. «лаг снабжения»
   (задержка доступности после покупки), чтобы фронт не усиливался мгновенно. _Решить в
   LARS-3._

**DoD.** Резолюции вписаны; LARS-1 разблокирован.

## LARS-1 · Живая авторизация постройки `[core/srv]` 🔒(ARS-2, ARS-3)

**Цель.** Приобретённый модуль сразу доступен к постройке в идущем матче; чужой/неимеемый
— отклоняется. Детерминизм не страдает.

**Опора.** `validateLoadout` (легальность, `loadout.ts`) — не трогаем; `unit.build{modules}`
(залогированное действие); `ArsenalStore.listOf(accountId)` (ARS-2); server-authority
(инвариант #5 — ядро доверяет допущенному действию).

**Реализация.** Перенести гейт **владения** модулем из внутристейтового стартового
снапшота (ARS-3) в **серверную живую проверку** при допуске `unit.build{modules}`:
модуль во владении СЕЙЧАС (`ArsenalStore`) → строить можно; иначе `E_NOT_OWNED`
(fail-secure). Ядро видит только: допущенное действие + `data.modules[id]` (статы) →
результат постройки детерминирован и воспроизводим из журнала. Реплей/пробуждение спящего
матча не перепроверяют владение (действие уже в журнале с квитанцией). Продажа модуля
ПОСЛЕ постройки корабль не «разстраивает» (как ARS-3: изменение арсенала не трогает уже
случившееся).

**Тесты.** Модуль, приобретённый в середине матча, строится; неимеемый → `E_NOT_OWNED`;
существующие корабли и их статы не меняются; повторный реплей журнала даёт тот же
результат (детерминизм); продажа-после-постройки не отменяет корабль; клиентская
хеш-сверка не расходится.

**DoD.** «Что я могу построить» читается из живого арсенала; ядро и реплеи целы.

## LARS-2 · Цепочка валют → живой фронт end-to-end `[srv]` 🔒(EC-1.2, EC-2/EC-3, LARS-1)

**Цель.** Собрать существующие экономические звенья с живым билдом в один проверяемый
путь: Суверены → Варранты → приобретение/заточка → `ArsenalStore` → доступно к постройке.

**Опора.** EC-1.2 (кошелёк Суверены/Варранты, атомарные операции), EC-2.1 (заточка) /
EC-3 (аукцион), ARS-2 (стор владения), LARS-1 (живой гейт). Жёсткий гейт экономики
(`economy-roadmap.md`: аккаунты/биллинг/аудит-леджер/per-player очередь/туман — фича не
стартует в коде до них).

**Реализация.** Интеграционная сборка (без новых механик): приобретение через
EC-2.1/EC-3 идемпотентно кладёт `ArsenalItem` в `ArsenalStore`; тот же itemId сразу виден
LARS-1. Всё server-authoritative, каждое движение валюты/предмета — в аудит-леджер
(EC-1.3/PE-1.1).

**Тесты.** Полный путь на фикс-сценарии: обмен Суверены→Варранты → покупка модуля →
предмет в арсенале → постройка в идущем матче проходит; недостаток валюты → отказ без
списания (атомарность); дубль приобретения не плодит предмет.

**DoD.** От доната до нового корабля на фронте — один воспроизводимый путь с аудитом.

## LARS-3 · Баланс и гайки честности `[srv/data]` 🔒(LARS-1)

**Цель.** Не дать фиче стать «мгновенным P2W-выигрышем» и удержать F2P-паритет.

**Опора.** Постройка + логистика уже гейтят силу во времени; `economy-roadmap.md` гайки
(pity, кэтчап, soulbound); данные-тюнинг.

**Реализация (ручки, данными где можно).**
- **Только свои миры** — строить с новым модулем можно на своих планетах (обычное правило
  постройки; подтвердить, что чужого фронта не касается).
- **Лаг снабжения (опц.)** — задержка доступности модуля к постройке после приобретения
  (данные: `supplyLagHours`), чтобы покупка не доезжала до фронта абсурдно быстро.
- **F2P-паритет** — заработанные модули (дроп/крафт) идут тем же живым-билд-путём;
  тест-инвариант «купленный и заработанный модуль одинаково строятся».
- **Прозрачность** — в UI видно источник (куплено/выпало/скрафчено) и что усиление
  придёт через постройку, не мгновенно.

**Тесты.** Лаг снабжения соблюдается; заработанный = купленный по строимости; на чужих
мирах не строится.

**DoD.** Сила приходит через петлю, не мгновенно; F2P не поражён в правах.

## LARS-4 · UI: живая Верфь в матче `[proto]` 🔒(LARS-1, ARS-5)

**Цель.** Игрок видит свежеприобретённые модули прямо в Верфи идущего матча и понимает,
что усиление — через постройку.

**Опора.** Верфь прототипа (SHIP-2), витрина арсенала (ARS-5), кодекс (`openCodex`).

**Реализация.** Верфь в матче показывает и только что приобретённые модули (фильтр по
живому арсеналу); «где взять» ведёт в магазин/аукцион; ясная метка «прибудет постройкой +
логистикой», не мгновенно; источник предмета на карточке.

**Тесты.** uitest-паттерн: свежий модуль появляется в Верфи матча; метка тайминга видна;
неимеемый не предлагается.

**DoD.** Путь «купил → строю в этом матче» очевиден и не обещает мгновенного эффекта.

## Что меняется в инвариантах (по-честному)

- **Ослабляется:** снапшот-на-старте (GDD §5.2 / main-menu §5.2 / ARS-3) — **только для
  билд-каталога** («что можно построить» становится живым). Это единственная
  семантическая правка; её надо отразить в ARS-3 и GDD §5.2 при реализации.
- **НЕ меняется:** консервация построенного / no-refit (GDD §2) — существующие корабли
  неизменны; детерминизм ядра, реплеи, гибернация, хеш-сверка, server-authority; принятая
  P2W-политика (эта фича её реализует, не вводит).

## Зависимости

Тяжело гейтится (ничего не стартует в коде раньше):

- **Экономика:** EC-1.1 (модель предмета), EC-1.2 (кошелёк Суверены/Варранты), EC-2.1
  (заточка) / EC-3 (аукцион) + жёсткий гейт `economy-roadmap.md` (аккаунты, биллинг,
  аудит-леджер PE-1.1, per-player очередь F5, туман F6, лутбокс-комплаенс EC-0.4).
- **Арсенал:** ARS-1 (схема предмета), ARS-2 (ArsenalStore — источник живого владения),
  ARS-3 (снапшот, который правим на живой гейт).
- **Аккаунты:** SE-1.x / AC-* (личность → владелец арсенала), гейт по уровню
  (`account-level.md`).

## Критический путь

`EC-1.* + ARS-2` (владение существует и durable) → **LARS-0** (решение) → **LARS-1**
(живой гейт постройки — сердце фичи) → **LARS-2** (цепочка валют end-to-end). Затем
параллельно **LARS-3** (баланс/гайки) и **LARS-4** (UI). Без EC/ARS-фундамента фича —
только дизайн (как и вся мета-экономика за жёстким гейтом).
